# Missing Conns — implementation spec for 0.1.0 symmetry

**Date:** 2026-04-27
**Source:** API-gap audit (agent run 2026-04-27, summarized in
[`note-2026-04-27-01.md`](note-2026-04-27-01.md)) plus a manual cross-
walk of `conn::int` / `conn::uint` against the symmetric mesh.
**Context:** Hand-off doc for an agent implementing the missing Conns
identified during publish-prep. Locations and test files in this doc
target the **post-T7 layout** specified in
[Plan 15](../plans/plan-2026-04-27-03.md). If T7 has not landed when
implementation starts, place the new Conns in the matching pre-T7
file (e.g. `src/conn/int.rs` / `src/conn/uint.rs`) and rebase
through T7.

All names follow the [§Conn-name format](../../CLAUDE.md) rule: 8
ASCII chars, two 4-char sides, smaller-resolution / coarser tier on
the right, zero-padded digits. Modules are chosen per the
[§Module organization](../../CLAUDE.md) precedence: **(1) specific >
general**, then **(2) right-side wins** as tie-breaker. All test
files live under `tests/` (per Plan 15 T8's
integration-tests-everywhere policy).

**Total to implement: 45 Conns.** All four "should ship" families
share a single design pattern (saturating cast in the narrowing /
non-widening direction, lossless widen in the inverse direction); a
single macro pair similar to the existing `uint_uint!` /
`int_uint!` should cover them.

## 1. Std signed-int narrowing (`I_M_N` for `M > N`)

**Shape:** `Conn<i_M, i_N>` (single-sided / `Conn::new_left`),
mirroring the existing `uint_uint!` pattern.

- `ceil`: `x as i_N` saturating to `[i_N::MIN, i_N::MAX]`.
- `inner`: `x as i_M` (lossless widen — guaranteed by `M > N`).

If the implementing agent prefers the **full-triple** shape
(`Conn<Extended<i_M>, Extended<i_N>>`) for symmetry with the
`ext_int!` widening family, that's also acceptable — pick whichever
best matches existing convention and document the choice in the
implementation commit.

## 3. Cross-sign U→I non-widening (`U_M I_N` for `bits(u_M) ≥ bits(i_N)`)

**Existing:** `U_M I_N` for `bits(u_M) < bits(i_N)` — strict widening,
so the entire `u_M` range fits in `i_N`'s positive half (defined via
`ext_int!` in `src/conn/int.rs`). The other direction is missing.

**Shape:** `Conn<u_M, i_N>` (single-sided).

- `ceil`: `x as i_N` saturating to `i_N::MAX` when `x > i_N::MAX`.
- `inner`: clamp negative `i_N` values to `0`, then `as u_M` with
  saturation at `u_M::MAX` if `bits(u_M) < bits(i_N)`.

## 5. Test battery for each new Conn

Apply uniformly to all 45 new Conns. Use the existing
`crate::property::laws::*` predicates and `crate::property::arb::*`
strategies (or extend `arb` if a generator doesn't yet exist for the
new type pair).

**Property tests** (one `proptest!` block per Conn — one-sided
families test L-side laws only):

- `conn_galois_l`
- `conn_closure_l`
- `conn_kernel_l`
- `conn_monotone_l`
- `conn_monotone_r`
- `conn_idempotent`
- `conn_floor_le_ceil` (skip if saturation forces `floor > ceil` at
  extremes — document in the proptest module if so)

**Spot-check unit tests** per Conn:

- Saturation arms (each MIN/MAX boundary forced).
- Round-trip for at least 5 representative finite values.
- Identity at zero.

The four families share a saturating-cast / lossless-widen shape,
so a single `narrow_props!` macro can drive the property battery for
all 45 — mirror `single_sided_props!` in `src/conn/uint.rs`.

## 6. Conns the audit flagged but that DO NOT need new code

Document these explicitly in the implementing PR's description so
reviewers see the rationale.

### 6a. Lossless float widening — already available via `Conn::inner`

The API-gap audit listed `F032F064`, `F016F032`, `F016F064`,
`B016F032`, `B016F064` as "missing widening Conns". Re-examination
shows the lossless widening direction is the **`inner` adjoint** of
the existing narrowing Conn:

```rust
// existing:
const F064F032: Conn<ExtendedFloat<f64>, ExtendedFloat<f32>>;

// the lossless f32 → f64 widen, no new Conn needed:
let widened: ExtendedFloat<f64> = F064F032.inner(some_extended_f32);
```

A separate `F032F064: Conn<ExtendedFloat<f32>, ExtendedFloat<f64>>`
would have `ceil = floor = inner = lossless_widen` (since widening
is a function, not a relation), making the triple degenerate and
redundant with `F064F032.inner`. **Skip.** If a downstream user
finds the inner-adjoint access ergonomically painful, revisit
post-1.0.

### 6b. Reverse decimal-tier Conns — same reason

`FD00FD12`, `FD03FD06`, etc. (coarser → finer) are accessible as
`FD12FD00.inner(coarse)`, `FD06FD03.inner(coarse)`, etc. Same
rationale. **Skip.**

### 6c. `Extended<T>` ↔ `T` lifting Conns — defer to 0.2

`Conn<T, Extended<T>>` (lift) and `Conn<Extended<T>, T>`
(extract-with-saturation) are trivial injection / projection. They
have no Galois-correspondence content beyond what
`Extended::Extend(x)` and `match` already give callers. **Defer.**

### 6d. `F032FD<N>` cross-tier — blocked by design

Documented in `src/conn/fixed/decimal.rs:274–283` (post-T7:
`src/conn/std/i64/decimal.rs`): an `inner` that narrows `i64 → f32`
collapses large runs of Rung values onto the same f32, forcing an
O(plateau) correction per ceil/floor call. **Skip.** Users widen at
the boundary: `F064FD<N>.ceil(ExtendedFloat::Extend(arg_f32 as f64))`.

For the agent picking this up:

- [ ] Read this doc + [Plan 15](../plans/plan-2026-04-27-03.md) +
      [`CLAUDE.md`](../../CLAUDE.md) §Module organization.
- [ ] Confirm Plan 15 T7 has landed (per-crate module reorg + file
      renames). If not, work in the pre-T7 layout
      (`src/conn/int.rs` / `src/conn/uint.rs`) and rebase through.
- [ ] Pick the macro shape: extend `uint_uint!` and `int_uint!`
      to also cover narrowing and U→I cases, OR write two new
      macros (`int_int_narrow!` and `uint_int!`). The existing
      single-sided pattern is the natural fit.
- [ ] Implement all 45 Conns in alphabetical-by-name order within
      each module. Each Conn is one `pub const` plus a one-line
      doctest at the definition site (per the "Doctest + README
      discipline" memory note: every new public Conn const gets a
      doctest).
- [ ] Add proptest blocks per §5. Each new Conn must drive the full
      law battery before its commit lands.
- [ ] `cargo test --workspace` green at every commit (per
      CLAUDE.md repository convention).
- [ ] `cargo clippy --all-targets -- -D warnings` clean.
- [ ] Update `src/lib.rs` rustdoc table to list the new Conn
      families (or, if the table is too noisy, add a one-paragraph
      summary pointing at the per-module rustdocs).
- [ ] No README mirror discipline on these — the headline doctest
      pattern is for new families, not for symmetric extensions of
      existing ones.
- [ ] Sequence as a follow-up MR after Plan 15's structural work
      lands; this doc is the spec for that follow-up.
