# Plan 29A — Phantom kind tag on the existing 3-fn `Conn`

## Goal

Land kind-discipline (`L` / `R` / `T`) on the existing 3-fn `Conn<A, B>`
struct via a phantom marker, with kind-bound property predicates and
kind-aware composition. Validates the kind-tag pattern in a small
diff that can ship as one commit; defers the structural 2-fn /
marker-type migration to the follow-up sprint **Plan 29B**
(`doc/plans/plan-2026-04-29-01.md`).

## Why split

The chosen design from `plan-2026-04-29-01.md` (strict 2-fn `Conn`
with marker-type triples and `ViewL` / `ViewR` projections) is the
right long-term shape, but the migration touches every const
declaration site (~52 consts), every macro generator (~10 macros),
and every test block in the crate — net ~150+ test callsites of
`.ceil()` / `.floor()` / `.inner()` need rewriting. That's one giant
atomic commit because intermediate states can't compile.

This sprint stages the work:

- **29A (this sprint):** add the kind tag, bound the property-fn
  layer by kind, add `Meet` for compose specialization. The 3-fn
  struct stays. ~10 const-site annotations change, no test rewrites.
  Self-contained and reviewable.
- **29B (follow-up):** structural migration to 2-fn `Conn` +
  `ViewL` / `ViewR` / `Triple` markers + `compose!` declaration-form
  macro for triple composition. The kind-discipline pattern from
  29A keeps working through the migration.

The user opted for staging with the intent to squash the two sprints'
commits into one before merging the MR — so the final history shows a
single atomic landing while the local development happens
incrementally.

## Tasks

### T1 — Sealed `Kind` trait + `L` / `R` / `T` markers + implication traits

Add `mod kind` to `src/conn.rs` with:

```rust
mod kind {
    mod sealed { pub trait Sealed {} }
    pub trait Kind: sealed::Sealed + Copy + 'static {}

    #[derive(Copy, Clone, Debug)] pub struct L;
    #[derive(Copy, Clone, Debug)] pub struct R;
    #[derive(Copy, Clone, Debug)] pub struct T;

    impl sealed::Sealed for L {} impl Kind for L {}
    impl sealed::Sealed for R {} impl Kind for R {}
    impl sealed::Sealed for T {} impl Kind for T {}

    pub trait ImpliesL: Kind {}
    impl ImpliesL for L {} impl ImpliesL for T {}

    pub trait ImpliesR: Kind {}
    impl ImpliesR for R {} impl ImpliesR for T {}
}

pub use kind::{Kind, L, R, T, ImpliesL, ImpliesR};
pub type ConnL<A, B> = Conn<A, B, L>;
pub type ConnR<A, B> = Conn<A, B, R>;
pub type ConnT<A, B> = Conn<A, B, T>;
```

### T2 — Re-parameterize `Conn` with default `K = T`

Add a `PhantomData<fn() -> K>` field to `Conn`. Default kind = `T`,
so existing `Conn::new(...)` consts annotated `Conn<A, B>` continue
to compile unchanged (they get `K = T` via the default).

```rust
pub struct Conn<A, B, K: Kind = T> {
    pub(crate) ceil: fn(A) -> B,
    pub(crate) inner: fn(B) -> A,
    pub(crate) floor: fn(A) -> B,
    _k: PhantomData<fn() -> K>,
}

impl<A, B, K: Kind> Copy for Conn<A, B, K> {}
impl<A, B, K: Kind> Clone for Conn<A, B, K> { fn clone(&self) -> Self { *self } }
```

The pre-existing 3-fn shape stays. Only the phantom marker is added.

### T3 — Per-kind constructor placement

```rust
impl<A, B> Conn<A, B, T> {
    pub const fn new(ceil: fn(A) -> B, inner: fn(B) -> A, floor: fn(A) -> B) -> Self { ... }
    pub const fn new_iso(forward: fn(A) -> B, back: fn(B) -> A) -> Self { ... }
}

impl<A, B> Conn<A, B, L> {
    pub const fn new_l(ceil: fn(A) -> B, inner: fn(B) -> A) -> Self { ... }  // formerly new_left
}

impl<A, B> Conn<A, B, R> {
    pub const fn new_r(inner: fn(B) -> A, floor: fn(A) -> B) -> Self { ... }  // formerly new_right
}

impl<TT> Conn<TT, TT, T> { pub const fn identity() -> Self { ... } }
```

`new_left` is renamed to `new_l` and `new_right` to `new_r` (per the
naming convention pinned in plan-2026-04-29-01.md). The existing
~92 callsites get a sed-style rename.

The `ceil`/`inner`/`floor` field-access methods (`Conn::ceil`,
`Conn::inner`, `Conn::floor`) stay where they are: in a kind-agnostic
`impl<A, B, K: Kind> Conn<A, B, K>` block. They read fields directly,
so they work on any kind.

### T4 — Kind-bound free fns

The free-fn surface in `src/conn.rs` (`ceiling`, `floor`, `upper`,
`lower`, their `1` / `2` lifters, plus `interval` / `midpoint` /
`round` / `truncate` / `median`) gains kind bounds:

```rust
pub fn ceiling<A, B, K: ImpliesL>(c: &Conn<A, B, K>, a: A) -> B { ... }
pub fn upper<A, B, K: ImpliesL>(c: &Conn<A, B, K>, b: B) -> A { ... }
pub fn ceiling1<A, B, K: ImpliesL, F>(c: &Conn<A, B, K>, f: F, b: B) -> B
where F: FnOnce(A) -> A { ... }
// ...etc, all L-side ops require ImpliesL.

pub fn floor<A, B, K: ImpliesR>(c: &Conn<A, B, K>, a: A) -> B { ... }
pub fn lower<A, B, K: ImpliesR>(c: &Conn<A, B, K>, b: B) -> A { ... }
// ...etc, all R-side ops require ImpliesR.

// Two-sided: needs both, effectively K = T.
pub fn interval<A, B, K>(c: &Conn<A, B, K>, x: A) -> Option<Ordering>
where K: ImpliesL + ImpliesR, ... { ... }
pub fn round<A, B, K>(c: &Conn<A, B, K>, x: A) -> B
where K: ImpliesL + ImpliesR, ... { ... }
// ...truncate, midpoint, median, all 1/2 lifters likewise.
```

After this lands, calling `floor(&l_kind_conn, x)` is a compile error
("`L: ImpliesR` is not satisfied"), exactly the discipline the kind
tag is supposed to provide.

### T6 — `Meet` trait + kind-specialized `compose!` macro

Type-level meet for kind composition (`T ∧ L = L`, `T ∧ R = R`, etc.):

```rust
pub trait Meet<Other: Kind>: Kind { type Output: Kind; }

impl Meet<L> for L { type Output = L; }
impl Meet<L> for T { type Output = L; }
impl Meet<T> for L { type Output = L; }
impl Meet<T> for T { type Output = T; }
impl Meet<R> for R { type Output = R; }
impl Meet<R> for T { type Output = R; }
impl Meet<T> for R { type Output = R; }
// no Meet<R> for L → mixed compose is a type error.
```

Update the `compose!` macro to thread the meet through. The
generated `Conn::new` call site picks up the meet kind from the
parents.

If the macro form makes the `Meet` plumbing awkward, the fallback is
a typed `pub const fn compose<A, B, C, K1: Meet<K2>, K2: Kind>(...)`
that takes two Conns by value and returns the meet-kinded result.
Either path is acceptable; pick whichever lands cleanest.

### T7 — Annotate existing one-sided consts with their kind

The 33 `new_left` consts (renamed `new_l`) need a kind annotation:
`Conn<A, B>` → `Conn<A, B, L>` or `ConnL<A, B>`. Same for the 15
`new_right` (now `new_r`) consts → `Conn<A, B, R>` or `ConnR<A, B>`.

The 28 `Conn::new` triple consts and 16 `new_iso` consts stay at
the default kind `T`, so their `Conn<A, B>` annotations continue
unchanged.

The macros that generate one-sided consts (`uint_uint!`,
`int_uint!`, `int_int_narrow!`, `uint_uint_narrow!`,
`int_uint_narrow!`, `nz_uint_ext!`, `uint_int_sat!`) need their
output-type signature updated to reflect the kind explicitly. Bulk
edit; mechanical.
