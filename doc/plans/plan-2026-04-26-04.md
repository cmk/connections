# Plan 05 — Property tidy-up: predicates as public API

## Goal

Expose every Galois-/lattice-law predicate as a `pub fn` callable
from downstream crates, behind a `testing` feature gate that flips
`proptest` from dev-dep to optional regular dep. Standardise naming
on the existing `<algebra>_<lawname>` convention. Wire the orphan
lattice predicates in `src/property.rs` into in-crate proptest blocks.

## Context

`src/property.rs` already has 28 lattice law predicates as `pub fn`s
(`heyting_adjunction`, `coheyting_excluded_middle`,
`boolean_double_neg_id`, `symmetric_involution`, etc., shipped in
commit `797462e`). They follow a clean naming convention —
`<algebra>_<lawname>` — and the right signature shape (take `&T`,
return `bool`, no proptest dependency). **But they're inaccessible
to downstream consumers**: `src/lib.rs:81` declares them with
`#[cfg(test)] pub(crate) mod property;`, so `proptest` doesn't leak
into the public build at the cost of making the predicates
test-only. They are also, currently, **defined but never called** —
no in-crate proptest block exercises them.

Conn-side predicates, by contrast, are split between two styles:

- Free-standing `bool`-returning helpers in `src/conn.rs` (identity
  Conn) and `src/conn/float.rs` (f64↔f32) — already the right shape,
  just the wrong scope.
- Inline-in-macro predicates in `src/conn/fixed.rs` and
  `src/conn/sample.rs` — four macros (`props_for_pair!`,
  `float_conn_props!`, `props_for_conn!`, `props_for_pico_conn!`)
  redefine the same Galois/closure/kernel laws over and over.

Strategies (`bounded_fine`, `bounded_coarse`, `safe_fine`,
`arb_extended_*`) live in the test modules of `conn/fixed.rs` and
`conn/sample.rs`. The fixed-tier ones are already `pub(crate)` post-
MR !9; the sample-tier ones are still private.

This sprint moves all of that into one canonical, feature-gated
`property` module, splits predicates from strategies cleanly, and
writes the proptest call sites that exercise the orphan lattice
predicates.

Build gate: `cargo build` (no features) compiles cleanly with
`property::laws` visible but `property::arb` gated out.

### T2 — Split `property/` into `laws.rs` + `arb.rs`

(`heyting_*`, `coheyting_*`, `biheyting_*`, `symmetric_*`,

Always public, no feature gate.

`src/property/arb.rs`: move the existing `arb_f64`, `arb_f32`,

Existing import sites — `src/conn.rs:137`, `src/lattice.rs:401`,

### T3 — Conn law predicates

Add to `src/property/laws.rs` following the existing
`<algebra>_<lawname>` convention. Single `Ple`-bound family.

| Predicate | Invariant |
|---|---|
| `conn_galois_l` | `ceil(a) ≤ b ⟺ a ≤ inner(b)` |
| `conn_galois_r` | `inner(b) ≤ a ⟺ b ≤ floor(a)` |
| `conn_closure_l` | `a ≤ inner(ceil(a))` |
| `conn_closure_r` | `inner(floor(a)) ≤ a` |
| `conn_kernel_l` | `ceil(inner(b)) ≤ b` |
| `conn_kernel_r` | `b ≤ floor(inner(b))` |
| `conn_monotone_l` | `a1 ≤ a2 ⟹ ceil(a1) ≤ ceil(a2) ∧ floor(a1) ≤ floor(a2)` |
| `conn_monotone_r` | `b1 ≤ b2 ⟹ inner(b1) ≤ inner(b2)` |
| `conn_idempotent` | `(inner ∘ ceil)² = inner ∘ ceil` |
| `conn_floor_le_ceil` | `floor(a) ≤ ceil(a)` |
| `conn_roundtrip_ceil` | `ceil(inner(b)) = b` (exact embedding) |
| `conn_roundtrip_floor` | `floor(inner(b)) = b` (exact embedding) |
| `conn_ulp_bound` | `ceil(a).0 - floor(a).0 ≤ 1` |

Plus preorder predicates (currently inline in `lattice.rs` test mod):

| Predicate | Invariant |
|---|---|
| `lattice_reflexive` | `x ≤ x` |
| `lattice_transitive` | `x ≤ y ∧ y ≤ z ⟹ x ≤ z` |
| `lattice_antisymmetric` | `x ≤ y ∧ y ≤ x ⟹ x ≡ y` (Ple equivalence) |
| `lattice_bot` | `Bot ≤ x` (where Bot is the type's bottom) |
| `lattice_top` | `x ≤ Top` |

Signatures follow the existing pattern — `&T: Ple` for lattice
predicates, `&Conn<A, B>` plus value inputs for Conn predicates.

### T5 — Rewrite the four property macros

Each macro becomes a thin `proptest!` wrapper that just feeds
strategy values to predicate calls:

```rust
macro_rules! props_for_pair {
    ($mod:ident, $conn:ident, $Fine:ident, $Coarse:ident, $prec:expr) => {
        mod $mod {
            use super::*;
            use $crate::property::{laws, arb};

            proptest! {
                #[test]
                fn galois_l(x in arb::fixed_fine($prec), c in arb::fixed_coarse($prec)) {
                    prop_assert!(laws::conn_galois_l(&$conn, $Fine(x), $Coarse(c)));
                }
                // ... etc
            }
        }
    };
}
```

Same shape applies to `float_conn_props!`, `props_for_conn!`,
`props_for_pico_conn!`. All 21 + 2 + 15 + 6 = 44 invocation sites
keep the same generated test names (so proptest-regressions seed
files keep working) but their bodies are now `prop_assert!(laws::*)`
calls.

## Review

### Implementation outcome

All planned tasks landed except T7 (in-crate proptest tests for the
orphan Heyting/Coheyting/Boolean/Symmetric predicates). T7 is
deferred because no concrete lattice instance exists in the crate
(no `impl Heyting for $T` anywhere in `src/`), so there's nothing
to drive the predicates against. They are exposed for downstream
use, satisfying the prime directive.

### Deviations from the plan

- **`property::arb` stays flat instead of subdivided.** Plan §
  Module layout proposed `arb/{float,fixed,rate,extended}.rs`. With
  ~15 strategy functions total, subdivision wasn't earning its
  keep — single `property/arb.rs` is more discoverable. The
  `<tier>_<role>` naming convention disambiguates without paths.

- **Predicate bound: `Ple` not `PartialOrd`.** Plan §
  Predicate signatures specified `Ple`-only. First-pass
  implementation used `PartialOrd` to cover `ExtendedFloat<f64>` /
  `Extended<T>` (which had no `Ple` impl). Switched back to `Ple`
  after the bare-`f64` identity tests broke on NaN — `<=` on `f64`
  returns `false` for `NaN <= NaN`, but `Ple::ple` patches it to
  reflexive per the N5 lattice. Added `Ple` impls for
  `ExtendedFloat<T>`, `Extended<T>`, and the `SampleRate` types so
  the single `Ple`-bound family covers every Conn-side type.

- **`conn_idempotent` and `conn_roundtrip_*` use Ple equivalence,
  not `==`.** Same NaN reflexivity issue: `f64::NaN == f64::NaN`
  is `false`. The predicates now compare via `x.ple(&y) && y.ple(&x)`,
  the N5 equivalence, which handles NaN correctly.

### Emergent additions

- **`Ple` impl for `ExtendedFloat<T>`, `Extended<T>`, and
  `SampleRate` types** (lattice.rs + sample.rs). Required to keep
  the predicate API uniform; trivially correct delegations to the
  existing `PartialOrd` impls (which already encode the N5 patches).

### Additional deviations (caught in Tier-1 review)

- **`peq` helper deleted, contradicting plan §T6.** Plan §T6 said
  "the `peq` helper (line 352) stays — it's a Ple-equivalence
  predicate, not a law, useful as a building block." The
  implementation deleted it along with the other `conn/float.rs`
  free-standing helpers. The Ple-equivalence idiom (`x.ple(&y) &&
  y.ple(&x)`) is now inlined into `conn_idempotent` and
  `conn_roundtrip_*`. Downstream crates wanting that primitive
  must inline the two-call pattern themselves rather than calling
  `crate::conn::float::peq`. Net loss: a small ergonomic helper.
  Re-add as `pub fn lattice_peq<T: Ple>(x: &T, y: &T) -> bool` in
  `property::laws` if a downstream consumer asks for it.

- **`float_conn_props!` idempotence scope reduced to one condition
  of four.** The original `prop_idempotent(a, b)` checked all four
  symmetric idempotences:
    1. `inner(ceil(inner(ceil(a)))) ~~ inner(ceil(a))` — closure on source
    2. `ceil(inner(ceil(inner(b)))) ~~ ceil(inner(b))` — counit via ceil
    3. `inner(floor(inner(floor(a)))) ~~ inner(floor(a))` — closure via floor
    4. `floor(inner(floor(inner(b)))) ~~ floor(inner(b))` — counit via floor
  The new `conn_idempotent(c, a)` predicate checks only (1).
  Conditions (2)–(4) cannot be violated by a Galois connection
  whose kernel laws hold (now tested separately as `conn_kernel_l`
  / `conn_kernel_r`), so the loss is theoretical for correct
  implementations — but it removes a defense-in-depth check that
  the lossy `f64 ↔ f32` connection's `ceil_f64_f32` /
  `floor_f64_f32` got right. Re-add as a `conn_counit_idempotent`
  predicate (or extend `conn_idempotent` to take an optional
  b-side) if a future bug surfaces in a lossy connection that the
  kernel laws miss.

- **`conn_ulp_bound` predicate** with caller-supplied rung
  extractor (`|b| b.0` for fixed-ladder, `|s: $Rate| s.0.to_bits()`
  for sample-rate types). The fixed-ladder `floor_le_ceil` test
  bundles the strict ULP check via this predicate; the
  sample-rate `floor_le_ceil` does the same with the
  rate-side extractor (replacing the previous separate `ulp_bound`
  test).

- **Two new doctests on `property::mod`** showing the consumer
  pattern: `rust,no_run` for direct predicate calls and `ignore`
  for the proptest-style usage.

### Test count change

519 → 491 (-28). Breakdown:
- `ceil_monotone` + `floor_monotone` collapsed into `monotone_l`
  across props_for_pair! (×21), props_for_conn! (×15),
  props_for_pico_conn! (×6) = -42
- `ulp_bound` separate test collapsed into `floor_le_ceil` on
  pico macro (×6) = -6
- `inner_then_ceil_integer_ratio` (one test checking both ceil
  and floor) split into `roundtrip_ceil_integer_ratio` +
  `roundtrip_floor_integer_ratio` (×15) = +15
- Float-conn tests added explicit `kernel_l`/`r`, `closure_l`/`r`
  splits = +4
- f64 identity tests added `monotone_l`/`r` etc. = +1

Net: -42 - 6 + 15 + 4 + 1 = -28. ✓

### Recommendations for future work

- **Wire the orphan lattice predicates** when a concrete Heyting
  instance lands. The `property::laws` module already has
  `heyting_*`/`coheyting_*`/`boolean_*` ready to drive.

- **Macro consolidation.** With predicates extracted, the four
  conn-property macros all do the same thing (drive `laws::*`
  against typed inputs). They could collapse into one generic
  invocation. Skip until the property surface settles.

- **Trait-based predicate dispatch** (e.g.
  `trait ConnLaws { fn check_galois(&self, …); }`) becomes
  attractive once enough downstream callers materialize.
