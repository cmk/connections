# Plan 27 — Convert `time/` Conns to one-sided so `floor_le_ceil` holds

## Goal

Convert the six `time/` full-triple Conns whose `inner` is non-order-
reflecting to one-sided `Conn::new_left(ceil, inner)`. After this lands,
every shipped `time/` Conn passes `conn_floor_le_ceil` over its full
source domain — by construction, since `new_left` sets `floor = ceil`
as a fn pointer.

Choosing `new_left` (not `new_right`) for two reasons: (1) the parent
Haskell library biases heavily toward left-Galois Conns (~100 `Cast 'L`
vs. ~2 `Cast 'R`), so this aligns with that convention; (2) `new_left`
preserves the existing `Conn<source, rung>` type shape — no signature
flip needed.

Affected:

| Conn       | File                  | Conn type                                   |
|------------|-----------------------|---------------------------------------------|
| `TIMENANO` | `time/clock.rs`       | `Conn<Extended<Time>, i64>`                 |
| `TIMESECS` | `time/clock.rs`       | `Conn<Extended<Time>, i64>`                 |
| `DATEJDAY` | `time/date.rs`        | `Conn<Extended<Date>, i32>`                 |
| `OFDTNANO` | `time/offset.rs`      | `Conn<Extended<OffsetDateTime>, i128>`      |
| `OFDTSECS` | `time/offset.rs`      | `Conn<Extended<OffsetDateTime>, i64>`       |
| `PDTMDATE` | `time/datetime.rs`    | `Conn<PrimitiveDateTime, Extended<Date>>`   |

## Motivation

Plan 24's reframe established that every shipped `Conn<A, B>` must
satisfy `conn_floor_le_ceil` — full triples by `inner` being order-
reflecting, one-sided Conns trivially via `floor = ceil` structurally.
Plan 24 flagged the five `Conn<Extended<T>, U_plain>` Conns above (and
later PDTMDATE with the dual `Conn<T, Extended<U>>` shape) as candidates
for the same audit.

Each is structurally non-order-reflecting at the source extremes:

- For `Conn<Extended<T>, U_plain>` Conns: `inner: U_plain → Extended<T>`
  saturates a whole range of out-of-range integers to `NegInf source` /
  `PosInf source`. Multiple distinct `U_plain` values map to the same
  synthetic source value → `inner` non-injective on the rung side. At
  source = `NegInf source`, Galois forces `ceil = U_plain::MIN`, `floor
  = (smallest in-range U_plain) - 1`; same dual at `PosInf source`. In
  both cases `floor > ceil`. The existing code is honest about this —
  `DATEJDAY`'s comments say "every b qualifies" for `ceil(NegInf)` and
  set `floor(NegInf) = MIN_JD - 1`, which by `i32` ordering is greater
  than `i32::MIN = ceil(NegInf)`.
- For `PDTMDATE` (`Conn<PrimitiveDateTime, Extended<Date>>`): `inner`
  saturates `NegInf rung` to `PrimitiveDateTime::MIN`, but
  `inner(Finite(Date::MIN)) = PrimitiveDateTime::new(Date::MIN,
  Time::MIDNIGHT) = PrimitiveDateTime::MIN` (per `time` crate). Same
  source value from two distinct rung values → non-injective at the
  bottom. The existing module docstring already acknowledges this
  ("`inner` is non-injective at the extremes ... so adjacent ceil/floor
  at those points disagree").

Per the rule (no third case), the fix is to convert each to
`Conn::new_left(ceil, inner)`. The existing `ceil`/`inner` from each
full-triple definition becomes the one-sided pair; the existing `floor`
is dropped (its synthetic-end disagreement was the bug). Since
`new_left` sets `floor = ceil` as a fn pointer, callers reading the
unused side get the ceil value.

### Sub-unit rounding loss for TIMESECS / OFDTSECS / PDTMDATE

Three of the six (`TIMESECS`, `OFDTSECS`, `PDTMDATE`) are sub-unit
projections where the existing full-triple `ceil` and `floor` genuinely
differ on Finite inputs (`ceil` rounds up to the next whole unit,
`floor` truncates). Converting to `new_left` keeps `ceil` and drops
`floor`'s truncating behavior: callers reading `<Conn>.floor(t with
subsec)` will get `ceil`'s answer (rounded up to next whole unit), not
the truncated answer they'd get under the existing full-triple.

This is a **behavioral change** documented in each Conn's rustdoc.
Users who need the dual rounding direction can:

1. Use the floor value and add 1 conditionally (when `t.subsec() > 0`).
2. Wait for a future paired-Conn proposal (e.g. `TIMESECSCEIL` and
   `TIMESECSFLOOR` shipping the two adjoint pairs as separate Conns
   — out of scope for Plan 27, deferred).

### Why not full-triple fixes?

For each of the six, `inner` cannot be made order-reflecting without
either restricting the rung type to a refinement (not feasible in Rust)
or re-shaping `inner` such that the violation just shifts elsewhere.

## Tasks

### T1 — Convert each Conn to `new_left`

Per file:

- `time/clock.rs`: `TIMENANO`, `TIMESECS` → `Conn::new_left(ceil, inner)`.
- `time/date.rs`: `DATEJDAY` → `Conn::new_left(ceil, inner)`.
- `time/datetime.rs`: `PDTMDATE` → `Conn::new_left(ceil, inner)`.
- `time/offset.rs`: `OFDTNANO`, `OFDTSECS` → `Conn::new_left(ceil, inner)`.

Drop the existing `floor` fn from each (its synthetic-end definition
was the bug; the constructor sets `floor = ceil` automatically).

For each, also update:
- The Conn's rustdoc to match the new shape (one-sided, Galois L only,
  `floor = ceil` structurally). For the rounding Conns
  (`TIMESECS`/`OFDTSECS`/`PDTMDATE`), explicitly note the sub-unit-floor
  semantic loss.
- The proptest battery: drop `galois_r`, `closure_r`, `kernel_r`,
  `monotone_r` (these don't hold for `new_left`); add
  `conn_floor_le_ceil` if not present.

## Deferred

- **`fixed/` macro-generated Conns** — `fixed/i8.rs`, `i16.rs`, `u8.rs`,
  `u16.rs` (and likely `u32.rs`, `i32.rs`, `u64.rs`, `i64.rs`,
  `u128.rs`, `i128.rs`) all have similar saturation-plateau
  `floor_le_ceil` violations. They're emitted by `fix_fix_i*!` /
  `fix_fix_u*!` / `int_int_narrow!` macros in `fixed.rs`. Fixing them
  requires modifying the macros plus the per-Conn test batteries. Out
  of scope for this sprint — the `time/` Conns are the immediate
  priority. The in-place "intentionally NOT asserted" comments were
  already updated to "tracked as known debt — Plan 27" by Plan 24's
  reframe; a follow-up plan picks them up.
- **Paired-Conn alternative for `TIMESECS` / `OFDTSECS` / `PDTMDATE`**
  — splitting each into `<Conn>CEIL` and `<Conn>FLOOR` so users can
  pick the rounding direction explicitly. Out of scope; revisit if the
  loss of the truncating direction turns out to be load-bearing for
  downstream callers.
- **Phantom-kind soundness for `Conn`** — the Rust port currently keeps
  `new_left` / `new_right` / `new` / `new_iso` constructors on a single
  `Conn<A, B>` type, so the `prop::conn::*_l` / `*_r` law families don't
  enforce kind-discipline at compile time (the Haskell parent does this
  via `Cast (k :: Side) a b` with `'L` / `'R` / `'N` and per-kind
  property fns). Plan 28 candidate: re-parameterize as `Conn<A, B, K =
  N>` with sealed `Kind` markers + per-impl constructors + property fns
  bounded by `K`. Worked example confirmed the soundness payload is
  recoverable in Rust without GADTs (the only thing lost is `forall k.`
  — which the Haskell port doesn't lean on at the property layer
  anyway). Deferred to keep this MR scoped.

## Review

### What landed

Six Conns flipped from full-triple `Conn::new(ceil, inner, floor)` to
one-sided `Conn::new_left(ceil, inner)`:

| Conn       | File                  | Behavioral delta on Finite inputs                |
|------------|-----------------------|--------------------------------------------------|
| `TIMENANO` | `time/clock.rs`       | None — `ceil = floor` on Finite (ns precision)   |
| `TIMESECS` | `time/clock.rs`       | `floor` now returns `ceil`'s round-up on subsec  |
| `DATEJDAY` | `time/date.rs`        | None — `ceil = floor` on Finite (no sub-day)     |
| `OFDTNANO` | `time/offset.rs`      | None — `ceil = floor` on Finite (ns precision)   |
| `OFDTSECS` | `time/offset.rs`      | `floor` now returns `ceil`'s round-up on subsec  |
| `PDTMDATE` | `time/datetime.rs`    | `floor` now rolls forward on sub-day input       |

Synthetic-end values (`floor(NegInf)` / `floor(PosInf)`) also change —
they now return `ceil`'s saturation values (`i64::MIN` / `MAX_RUNG +
1`) instead of the previous distinct full-triple values
(`MIN_RUNG − 1` / `i64::MAX`). Documented in each Conn's saturation
spot-check.

### Property battery delta

For each converted Conn the `_r` family is dropped (doesn't hold under
`new_left`): `galois_r`, `closure_r`, `kernel_r`, `monotone_r`, plus
`roundtrip_floor` (now redundant with `roundtrip_ceil`). The
`floor_le_ceil` probe added during T0 stays — it's structural under
`new_left` but worth keeping as a regression sentinel.

### Decision deviations from the plan

The plan's first draft proposed `Conn::new_right(inner, floor)`, which
preserves the truncating semantics on the 3 sub-unit Conns but lives
on the right-Galois half of the API. Switched to `new_left(ceil,
inner)` mid-sprint after a math review: (1) it preserves the existing
`Conn<source, rung>` type signature (no convention break vs. the
"rung on right" rule), (2) it aligns with the parent Haskell library's
heavy left-bias (~100 `Cast 'L` vs. ~2 `Cast 'R`), and (3) the
behavioral delta is the same magnitude either way — just opposite
rounding direction.
