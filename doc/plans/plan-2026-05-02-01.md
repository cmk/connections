# Plan 34 ‚Äî Catamorphism extractors for `Extended` and `ExtendedFloat`

## Context

`Extended<T>` and `ExtendedFloat<T>` are 3-variant enums (a synthetic
bottom, a wrapped `T`, a synthetic top) used as the targets of every
range-extending Conn in the crate. The Haskell `connections` library
ships catamorphism extractors per variant type ‚Äî `maybe`, `either` ‚Äî
that take one continuation per arm and fold the value to a single
result. The Rust port has no equivalent: every "produce a `U` from
this `Extended<T>`" call site has to write a `match` by hand.

This is a Rust idiom: `Option::map_or` / `map_or_else` and
`Result::map_or_else` are the same catamorphism, shipped as inherent
methods. Mirror that here.

## Goal

Add `fold` and `fold_with` inherent methods to `Extended<T>` and
`ExtendedFloat<T>`, generic over inner `T`, in the closure-last
order Rust uses elsewhere.

## Tasks

### T1 ‚Äî `Extended::fold` / `Extended::fold_with`

**File:** `src/extended.rs` ‚Äî new `impl<T> Extended<T> { ‚Ä¶ }` block
inserted before the `PartialOrd` impl (`extended.rs` had no inherent
block prior).

```rust
pub fn fold<U>(self, neg_inf: U, pos_inf: U, f: impl FnOnce(T) -> U) -> U;
pub fn fold_with<U>(
    self,
    neg_inf: impl FnOnce() -> U,
    pos_inf: impl FnOnce() -> U,
    f: impl FnOnce(T) -> U,
) -> U;
```

Closure-last argument order ‚Äî receiver, eager endpoints, closure ‚Äî
matches `Iterator::fold`, `Option::map_or`, `Result::map_or_else`.
The trade is variant-textual order (`NegInf, Finite, PosInf`) for
closure-last ergonomics; the latter wins on idiom + trailing-block
readability.

Doctest at the definition site of `fold` exercising all three arms.

### T2 ‚Äî `ExtendedFloat::fold` / `ExtendedFloat::fold_with`

**File:** `src/float.rs` ‚Äî new `impl<T> ExtendedFloat<T> { ‚Ä¶ }` block
inserted *outside* the `impl_float_arith!` macro (which is per-T) and
between `impl_float_ext!` invocations and `impl_float_arith!`.
Methods don't touch inner-T arithmetic, so they generalise over any
inner type. Same shape as T1 with `bot` / `top` parameter names.

Doctest at the definition site of `fold` mirroring T1.

### T3 ‚Äî Tests (per-module `#[cfg(test)]` blocks)

Per-variant unit tests, a `Cell`-flag check verifying `fold_with`
doesn't evaluate unselected arms, and a proptest asserting strict
and lazy folds agree:

```
fold_with (const gx) (const gy) f == fold gx gy f
```

driven by `arb_extended_i64()` for `Extended<i64>` and
`extended_float_f64()` for `ExtendedFloat<f64>`. Codomain on the
`ExtendedFloat` test is `i64` via `to_bits` so the `PartialEq` on
the result doesn't trip over NaN.

- Free-function aliases `extended(‚Ä¶)` / `extended_float(‚Ä¶)`. Method
  path syntax (`Extended::fold(x, ‚Ä¶)`) already gives the free-fn

- Reference-taking `fold_ref(&self, ‚Ä¶)` for non-`Copy` inner `T`.

  this crate has `T: Copy`. Add if a downstream caller hits the
  by-value-consume issue.
