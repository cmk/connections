# Plan 33 — Order-reflecting property + audit punch-list + README rigor

## Context

Plan 32 (MR !50, branch `sprint/floor-le-ceil`, **not yet merged**)
demoted ~140 non-triple Conns to `Conn::new_l`, added an inlined
`floor_le_ceil` check to `law_battery!`'s `full` subset, and shipped
a "Why one-sided?" section in the README with a sketch-level proof
of the equivalence

> `inner` order-reflecting ⟺ `floor(a) ≤ ceil(a)` ∀ a

and a one-element-source / three-element-target counterexample.

That sprint left three threads loose:

1. **The predicate that backs the proof isn't testable.** `floor_le_ceil`
   is the *corollary* of `inner` being order-reflecting; the
   *fundamental* property has no `prop::conn::*` predicate function and
   no `law_battery!` arm that exercises it directly. A future Conn whose
   `inner` is order-reflecting on the sampled domain but fails at an
   unsampled boundary (the `f09sys` shape) would slip past
   `floor_le_ceil` — only the boundary draw catches it. An
   `order_reflecting` predicate that quantifies over `b1, b2` directly
   gives proptest 2× more leverage to find that boundary.
2. **Audit items from `doc/chats/triple-adjunctions.md` are still open.**
   Plan 32 closed only the lawfulness bugs (T1–T5). The audit's
   "Suggested priority of fixes" table listed ~20 follow-up items
   (test-coverage gaps, doc rot, polish). A handful are 1-line fixes that
   shouldn't carry over to a third sprint.
3. **The README's proofs are sketches.** The user wants the chains
   written out, not paraphrased. The Plan 32 edit reads more like
   "trust me" than like a worked proof.

This plan addresses all three. **Depends on Plan 32 (MR !50)
merging first** — branch from `main` once `sprint/floor-le-ceil`
lands.

### T1 — Add `order_reflecting` predicate to `prop::conn`

**File:** `src/prop/conn.rs` — insert in the "Triple-bound predicates"
section, immediately after `floor_le_ceil` (line 167 in the Plan-32
worktree).

```rust
/// `inner(b1) ≤ inner(b2) ⟹ b1 ≤ b2` — `inner` is order-reflecting.
///
/// **The defining property of an adjoint triple.** Given the per-side
/// L/R Galois laws, this is equivalent to `floor(a) ≤ ceil(a)` for all
/// `a` (see the "Why one-sided?" section of the crate README for both
/// directions of the proof). `floor_le_ceil` is the corollary; this
/// predicate is the cause.
///
/// Quantifying over `(b1, b2) ∈ B²` gives a proptest twice the surface
/// area of `floor_le_ceil` (which only sees `a ∈ A`), so this catches
/// boundary violations the rounding-sandwich check would miss when the
/// source-side strategy under-samples extremes.
pub fn order_reflecting<T, A, B>(_t: &T, b1: B, b2: B) -> bool
where
    T: Triple<A, B>,
    A: Copy + PartialOrd,
    B: Copy + PartialOrd,
{
    let conn_l = <T as ViewL<A, B>>::L;
    let a1 = conn_l.upper(b1);
    let a2 = conn_l.upper(b2);
    !(a1 <= a2) || (b1 <= b2)
}
```

The `inner` is taken from the L view (`conn_l.upper`); for a true
triple, `ViewL::L.upper == ViewR::R.lower` by construction (same
function pointer), so the choice of view is arbitrary. We use L for
consistency with the existing `idempotent` pattern in the macro (`src/prop/conn.rs:55-57`).

### T5 — Expand README's "Why one-sided?" section with rigorous proofs

**File:** `README.md` lines 115-159 (Plan 32 worktree numbering — the
section between "How connections work" and "Quick tour" added by
commit `baf7e89`).

The Plan-32 edit reads as a sketch ("combine the two closure laws
and lift through the order-embedding `inner`"). The user wants the
chains written out so a reader can verify each step. Replace the two
proof paragraphs with explicit step-by-step derivations matching the
chat's final form:

```markdown
**Sufficiency** (`inner` order-reflecting ⟹ `floor(a) ≤ ceil(a)`).
Take any `a ∈ A`. The two closure laws give

    inner(floor(a)) ≤ a ≤ inner(ceil(a))

so by transitivity `inner(floor(a)) ≤ inner(ceil(a))`. Since `inner`
is order-reflecting, this lifts to `floor(a) ≤ ceil(a)`. ∎

**Necessity** (`floor(a) ≤ ceil(a)` everywhere ⟹ `inner` order-reflecting).
Take `x, y ∈ B` with `inner(x) ≤ inner(y)`. Chain:

    x ≤ floor(inner(x))         (kernel of inner ⊣ floor, with b = x)
      ≤ ceil(inner(x))          (assumption at a = inner(x))
      ≤ y                       (L-Galois ceil(a) ≤ b ⟺ a ≤ inner(b),
                                 with a = inner(x), b = y; the RHS
                                 inner(x) ≤ inner(y) is given)

So `x ≤ y`. ∎

(Categorically: in an adjoint triple `f ⊣ g ⊣ h` over posets, `g`
fully faithful ⟺ counit of `g ⊣ h` is iso ⟺ unit of `f ⊣ g` is iso
⟺ `h ≤ f`. The two displays above are that equivalence written for
posets, where "fully faithful" reduces to "order-reflecting" and
"iso" to "equality".)
```

Then sharpen the counterexample to make the L-Galois / R-Galois
forcing visible (the Plan-32 wording skips the derivation):

```markdown
**Counterexample.** Let `A = {a}` (one element) and `B = {b₁ < b₂ < b₃}`,
with `inner: B → A` the constant map (`inner(b) = a` for all `b` —
monotone but maximally non-injective).

L-Galois forces `ceil(a) ≤ b ⟺ a ≤ inner(b) = a`. The RHS is always
true, so `ceil(a) ≤ b` for all `b`. The smallest such `b` is `b₁`, so
`ceil(a) = b₁`.

R-Galois forces `inner(b) ≤ a ⟺ b ≤ floor(a)`. The LHS is always true,
so `b ≤ floor(a)` for all `b`, giving `floor(a) = b₃`.

Both per-side adjunctions hold, every monotonicity check passes — and
yet `floor(a) = b₃ > b₁ = ceil(a)`. The "triple" type-checks, the
per-side laws are satisfied, but `round(a)` (which compares
`inner(floor(a)) = a` and `inner(ceil(a)) = a` to pick the closer
endpoint, finds them equal, and falls through to `truncate`) returns
the wrong endpoint, with no in-band signal that anything is wrong.
```

Final paragraph (closing of section) gets a back-reference to the new
predicate:

```markdown
The `prop::conn::law_battery!` `full` subset enforces both
`order_reflecting` and its corollary `floor_le_ceil` so future triple
markers can't silently re-introduce the flaw — a violation aborts
`cargo test`, not a downstream caller.
```

**Placement decision.** Keep the section where Plan 32 placed it
(between "How connections work" and "Quick tour"). Rationale: the
section explains *why* `Conn` ships in three shapes (`ConnL`, `ConnR`,
triple marker), which is foundational structural context that comes
before the API tour. Examples 3 and 6 (the two examples that touch
triple markers) gain a one-line link back into this section in T6.
