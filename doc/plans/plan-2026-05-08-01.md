# Plan — Extended functor-lift over `Conn`

## Context

`doc/plans/plan-2026-05-07-06.md` (the user's draft) records the
`gmst_rad` bridge gap: composing `F064HDUR : Conn<F064, Extended<HD>, L>`
with the iso `EUTCHDUR : Conn<Epoch, HD, K>` to ship a single
`F64_SECS_EPOCH_VIA_UTC` const fails to type-check. `compose_l!` chains
`.ceil` calls pairwise — `EUTCHDUR.ceil` takes `Epoch`, but the
preceding step lands in `Extended<HD>`, not bare `HD`. The same gap
blocks every future compose that wants to tunnel a float-walking
Conn (`F064HDUR`, `F064ETAI`, `F064ETT`, `F064ETDB`, `F064ETDE`)
through any of the iso-shape `Epoch ↔ HD` Conns.

A **`Conn<HD, Extended<HD>>`** "deExtended" bridge is *not* the right
fix: it cannot satisfy the Galois law on the synthetic markers (any
choice for `inner(NegInf)` and `ceil(NegInf)` makes one of `closure_l`
or `kernel_l` fail at the `MIN`/`MAX` boundary; both orientations
collapse the same way). The user's fallback —  **lift both endpoints
of the bare-side Conn through the `Extended` functor** — is the
lawful move. Lifting `Conn<A, B, K>` to `Conn<Extended<A>, Extended<B>, K>`
is the categorical functor-lift; on the `Finite` arm it acts pointwise,
and the synthetic markers map identically (`NegInf ↦ NegInf`,
`PosInf ↦ PosInf`). Every law in `prop::conn::*` is preserved by
construction.

The Haskell port has the same operation as
`mapped :: Functor f => Cast k a b -> Cast k (f a) (f b)` in
`Data/Connection/Cast.hs`. Rust without HKT specialises this to the
single concrete functor that has crate-wide saturation semantics:
`Extended`.

## Goal

Ship `lift_l!` / `lift_r!` (expression-form) and `lift_k!`
(declaration-form, marker) macros in `connections::extended` that
emit a Conn whose endpoints are wrapped in `Extended<…>`. With these
in place, downstream crates (immediate consumer: `linklab-orbital::time`)
can declare their own `compose_l!`/`compose_k!` chains that bridge
bare-side and `Extended`-side endpoints — the connections crate ships
the algebra, not the named consts.

## Why a macro and not a function

`Conn<A, B, K>` stores two `fn(_) -> _` pointer fields, and Rust
fn-pointers cannot capture environment. A free fn
`lift_l(parent: Conn<A, B, L>) -> Conn<Extended<A>, Extended<B>, L>`
would have to synthesise an `fn(Extended<A>) -> Extended<B>` that
calls `parent.f` — which requires capturing `parent.f`, which
fn-pointers do not support. The macro form sidesteps this: each
expansion is a fresh non-capturing closure that references the
parent through a stable path (a `pub const` or marker), so the closure
coerces to a fn-pointer at compile time. This is the same pattern
`compose_l!` / `compose_k!` already use (`src/conn.rs:919`, `:989`).

## Tasks

### T1 — `lift_l!` / `lift_r!` expression macros

**Location:** new section at the bottom of `src/extended.rs`,
`#[macro_export]`ed so they reach the crate root next to
`compose_l!` / `compose_r!`.

**Shape** (sketch — final form must round-trip through `cargo expand`):

```rust
#[macro_export]
macro_rules! lift_l {
    ($parent:expr) => {
        $crate::conn::Conn::new_l(
            |a| match a {
                $crate::extended::Extended::NegInf   => $crate::extended::Extended::NegInf,
                $crate::extended::Extended::Finite(x) => {
                    $crate::extended::Extended::Finite(
                        $crate::conn::ConnL::ceil(&$parent, x)
                    )
                }
                $crate::extended::Extended::PosInf   => $crate::extended::Extended::PosInf,
            },
            |b| match b {
                $crate::extended::Extended::NegInf   => $crate::extended::Extended::NegInf,
                $crate::extended::Extended::Finite(y) => {
                    $crate::extended::Extended::Finite(
                        $crate::conn::ConnL::upper(&$parent, y)
                    )
                }
                $crate::extended::Extended::PosInf   => $crate::extended::Extended::PosInf,
            },
        )
    };
}
```

`lift_r!` mirrors with `ConnR::floor` / `ConnR::lower`. Using the
fully-qualified trait call (`<_ as ConnL<_,_>>::ceil`) lets `$parent`
be either a `Conn<A, B, L>` value or any `ConnL`-impling marker
(`iso!` / `conn_l!` / `compose_k!` outputs all qualify).

**Const-context support.** The `Conn::new_l` constructor is `const fn`,
the closure body is non-capturing (`$parent` is a path), and closure-to-
fn-pointer coercion happens at compile time. So `pub const X: Conn<…> = lift_l!(PARENT_CONST);`
is a valid const init — verified by `crate::compose_l!` already doing
the same trick (`src/conn.rs:1282`).

**Type signature** (informal, doc-only since macro hygiene erases it):
- For `parent : Conn<A, B, L>` (or `T: ConnL<A, B>`),
  `lift_l!(parent) : Conn<Extended<A>, Extended<B>, L>`.
- For `parent : Conn<A, B, R>` (or `T: ConnR<A, B>`),
  `lift_r!(parent) : Conn<Extended<A>, Extended<B>, R>`.

### T2 — `lift_k!` marker-declaration macro

**Location:** same file. Mirrors `compose_k!`'s declaration shape
(`src/conn.rs:989`).

**Surface:**
```rust
lift_k!(LIFTED_NAME : A => B = PARENT);
```

**Expansion:**
```rust
pub struct LIFTED_NAME;
impl ConnL<Extended<A>, Extended<B>> for LIFTED_NAME {
    fn conn_l(&self) -> Conn<Extended<A>, Extended<B>, L> {
        lift_l!(PARENT)
    }
}
impl ConnR<Extended<A>, Extended<B>> for LIFTED_NAME {
    fn conn_r(&self) -> Conn<Extended<A>, Extended<B>, R> {
        lift_r!(PARENT)
    }
}
```

The marker form is what `compose_k!` chains accept on either side of
`=>`, so `lift_k!`-emitted markers can sit inside larger `compose_k!`
declarations. Unconditional in-tree example (lifting the iso
`Q000I016 : FixedI16<U0> ↔ i16` from `fixed::i16` so it can splice
into a chain whose adjacent endpoint is `Extended<i16>`):

```rust
// In src/extended.rs's test module (or any downstream crate):
lift_k!(EXTQ000I016 : FixedI16<U0> => i16 = Q000I016);
// EXTQ000I016 now impls ConnL/ConnR/ConnK on
//   Extended<FixedI16<U0>> ↔ Extended<i16>.
```

The motivating downstream chain (linklab's `F64_SECS_EPOCH_VIA_UTC`,
which lifts the iso `EUTCHDUR` to splice with `F064HDUR`) lives
behind the `hifi` feature and is documented in
`doc/plans/plan-2026-05-07-06.md` — it does not appear in this
crate's rustdoc.

**Pre-condition:** `PARENT` must impl `ConnK<A, B>` (i.e. be either
a triple marker or both a `ConnL` and `ConnR` value). Document this
in the macro rustdoc; mis-uses surface as standard "method not in
scope" errors at expansion.

## Deferred

- A const-fn lift `Conn<A, B, K> -> Conn<Extended<A>, Extended<B>, K>`: blocked on fn-pointer-as-const-generic (unstable). Macro form covers every use.
- Lifts through other functors (`Option`, `Result`, etc.): `Extended` is currently the crate's only saturation-bearing functor; revisit when a second concrete one shows up. (`ExtendedFloat` deliberately *cannot* be lifted-through generically — its NaN handling on the `Extend` arm is type-specific to floats.)
- Shipping any composed const in `connections::hifi` (`F064EUTC` / `F064ETAI_VIA_UTC` etc.): out of scope. Per user direction, `linklab-orbital::time` declares its own.
- A per-side lift (`lift_left!` lifts only A's side, `lift_right!` lifts only B's side): only useful when one endpoint is *already* `Extended<…>`. The current both-sides lift handles the gmst_rad case via swap; one-sided lifts can be added if a real call site demands them.
