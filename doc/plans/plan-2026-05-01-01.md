# Plan 32 — `floor_le_ceil` cleanup: convert non-triple markers to one-sided

## Goal

Eliminate every shipped Conn that claims "triple" status (`ViewL + ViewR`)
but doesn't actually satisfy `floor(a) ≤ ceil(a)` for every input. After
this lands, every triple marker in the crate is a true adjoint triple
in the categorical sense — its `inner` is order-reflecting and the
two-sided helpers (`round`, `truncate`, `interval`, `midpoint`, `median`)
produce mathematically correct results on every input.

## Motivation

A triple `f ⊣ g ⊣ h` (in this crate's terms: `ceil ⊣ inner ⊣ floor`)
satisfies the rounding sandwich `floor(a) ≤ ceil(a)` everywhere if and
only if `inner` is order-reflecting (equivalently: `g` is fully faithful;
equivalently: the unit of `f ⊣ g` is iso; equivalently: the counit of
`g ⊣ h` is iso). Proof both directions:

- **Sufficiency**: from the closure laws `inner(floor(a)) ≤ a ≤ inner(ceil(a))`,
  transitivity gives `inner(floor(a)) ≤ inner(ceil(a))`, and order-reflection
  of `inner` lifts this to `floor(a) ≤ ceil(a)`.
- **Necessity**: assume `h(a) ≤ f(a)` everywhere. Take `x, y ∈ B` with
  `g(x) ≤ g(y)`. Then `x ≤ h(g(x)) ≤ f(g(x)) ≤ y` — first step is
  `g ⊣ h` kernel, second is the assumption at `a = g(x)`, third is
  L-Galois `f(a) ≤ b ⟺ a ≤ g(b)` with the given hypothesis. So
  `g` is order-reflecting.

Three Conn families currently ship as triples but have non-injective
(hence non-order-reflecting) `inner`s, so they violate the sandwich
and produce wrong results from `round`/`truncate`:

| Family | Saturation site | Concrete failure |
|---|---|---|
| `ext_int!` (20 invocations) | `_inner` collapses `[-i16::MIN, BELOW]` to `NegInf` and `[ABOVE, i16::MAX]` to `PosInf` | `floor(PosInf) = i16::MAX` vs `ceil(PosInf) = ABOVE` (e.g. `i16::MAX = 32767 > ABOVE = 128` for `I008I016`) |
| `fix_fix_i*!` / `fix_fix_u*!` (~119 Conns) | `_inner = c * RATIO` saturates to `FINE_MAX/MIN` for any non-trivial RATIO | `floor(FINE_MAX) = Coarse::MAX` (FINE_MAX fixup) vs `ceil(FINE_MAX)` (small Coarse value) |
| `STDRU128` | `stdru128_inner` saturates `Finite(n > max_nanos)` to `Finite(StdDuration::MAX)` | `kernel_r` violated at `b = Finite(max_nanos + 1)`: `floor(inner(b)) = Finite(max_nanos) < b` |

The latent bug in the parent Haskell library (`f09sys` at `Finite SystemTime::MAX`)
is the same shape as STDRU128 and is missed by the parent's tests due
to 0-biased boundary sampling. Will be flagged upstream (T6).

## Tasks

### T1 — STDRU128 → ConnL

Convert `STDRU128` from `triple!` to `Conn::new_l(stdru128_ceil, stdru128_inner)`.

- `src/time/duration.rs`: replace the `triple!` invocation with a direct
  `pub const STDRU128: ConnL<Extended<StdDuration>, Extended<u128>> = ...`
  (mirroring the post-Plan-27 pattern of `TIMENANO`/`OFDTNANO`).
- Drop `stdru128_floor` (it was a thin alias for `stdru128_ceil` anyway).
- Update the doctest to remove the `.floor()` line and explain the
  `floor = ceil` collapse.
- Migrate the `law_battery!` invocation from `full` to `l_only` (drop
  `galois_r`, `closure_r`, `kernel_r`, `monotone_r`).
- Add boundary spot tests: `0`, `max_nanos`, `max_nanos + 1`, `u128::MAX`.

### T2 — `ext_int!` macro emits ConnL

Rewrite the `ext_int!` macro in `src/fixed.rs` to emit `Conn::new_l`
instead of a triple marker.

- Drop the `_floor` const fn from the generated impl block.
- Drop the `ViewR` impl.
- Change `pub struct $NAME;` to `pub const $NAME: ConnL<Extended<$A>, $B> = Conn::new_l(_ceil, _inner);`
  (or keep the struct shape and only impl `ViewL`, whichever fits the
  test infrastructure).
- 20 invocations across `src/fixed/i16.rs`, `i32.rs`, `i64.rs`, `i128.rs` —
  no per-call-site changes needed beyond the macro rewrite.
- Migrate `tests/int_*_galois.rs` from `ext_int_props!` (which tests
  `galois_upper` AND `galois_lower`) to a one-sided property macro
  (`single_sided_props!`-equivalent for L). The L-side battery covers
  `galois_l`, `closure_l`, `kernel_l`, `monotone_l`, `idempotent`.

### T3 — `fix_fix_i*!` / `fix_fix_u*!` macros emit ConnL

Same shape as T2 across `src/fixed/{i,u}{8,16,32,64,128}.rs`.

- Modify each `fix_fix_i*!` / `fix_fix_u*!` macro to drop `_floor` and
  the `ViewR` impl.
- The FINE_MAX/MIN fixups in `_floor` were specifically engineered to
  preserve `galois_r` at the boundary. With ConnL they're moot; drop
  the function.
- The FINE_MIN fixup in `_ceil` (`return FixedI8::<Coarse>::from_bits(i8::MIN)`)
  IS still needed for `galois_l` at the boundary — keep it.
- Migrate `props_for_pair!` invocations from `law_battery!` `full` to
  `l_only` across the in-module test blocks (`src/fixed/i*.rs` `mod tests`)
  and the integration tests (`tests/fixed_u*_galois.rs`).

### T4 — Strengthen `law_battery!` `full`

Add `floor_le_ceil` as a required check in the `full` subset of
`law_battery!` so future triple markers can't silently re-introduce the
flaw.

- `src/prop/conn.rs:417-481`: add `fn floor_le_ceil` test to the `full`
  arm. Source-side strategy already in scope (`$f`); calls
  `crate::prop::conn::floor_le_ceil(&t, a)` for some triple marker
  reference.
- The test needs the marker (not just the L-Conn / R-Conn), since
  `floor_le_ceil` is `Triple`-bound. Either: change the macro to require
  a triple-marker argument (inconvenient for one-sided users), or split
  the existing `full` into `full` (L+R views, no sandwich) and
  `triple` (L+R views + sandwich). Latter is cleaner.
- Document the equivalence in `prop/conn.rs:160-167` (the `floor_le_ceil`
  predicate's docstring): "`g` order-reflecting ⟺ `floor ≤ ceil`
  everywhere".

**~140 connections demoted** from triple markers to `ConnL`. Every
remaining triple in the crate (`DURNSECS`, `STDRU064`, `U032CHAR`,
`U032IPV4`, `U128IPV6`, `IPV6IPV4`, `F064F032`, plus the f16-gated
`F064F016`/`F032F016`) now passes the strengthened `law_battery!`
`full` subset that includes `floor_le_ceil` as a required check.

### Method that scaled

The strengthened battery (T4) was the multiplier: rather than
hand-auditing every triple, adding `floor_le_ceil` to the `full`
subset surfaced 4 additional latent violators (the float↔duration
family) automatically. T5 audit was effectively "run cargo test
once and demote whatever fails" — turned out to be exactly 4 Conns,
all in `time/duration.rs`, all matching the same shape as STDRU128
(rung type extends past what the source can represent).

### Properties verified

| Property              | Holds for every shipped triple |
|-----------------------|--------------------------------|
| `galois_l`/`galois_r` | ✓ (per-side adjunction laws) |
| `closure_l`/`closure_r` | ✓ (per-side unit/counit) |
| `kernel_l`/`kernel_r` | ✓ (per-side counit/unit) |
| `monotone_l`/`monotone_r` | ✓ (composition of monotone fns) |
| `idempotent` | ✓ (closure operator) |
| **`floor_le_ceil`** | ✓ (Plan 32 — required by `law_battery!` `full`) |

For demoted ConnL Conns, the relevant subset is just the L-side
(`galois_l`, `closure_l`, `kernel_l`, `monotone_l`, `idempotent`),
covered either by the `l_only` subset of `law_battery!` or by hand-
rolled `proptest!` blocks at sites where the marker shape isn't a
unit struct.

### Test count delta

Pre-Plan-32: ~3000 tests workspace-wide. Post-Plan-32: ~2300.
Net loss = 4 R-side proptests per demoted triple × ~140 Conns ≈ 560,
minus the new boundary spot tests (~5) and the new `floor_le_ceil`
checks added to the `full` battery (~7). All passing; no regressions.

### Behavioral delta for downstream callers

Callers writing `BROKEN_CONN.floor(x)` against any of the ~140
demoted Conns will see a compile error: `no method named 'floor' found
for struct Conn<_, _, L>`. This is **deliberate**: the prior `.floor()`
returned answers that were wildly wrong at the saturation boundary
(e.g. `Q004Q000.floor(7.9375) = 127`, where the user expected 7 or 8).
Compile error is strictly safer than wrong answer at runtime.

Migration paths for callers who genuinely need the round-down
direction:
1. Pick a different Conn that actually supports it (rare for
   saturating casts; common for true triples like `DURNSECS`).
2. Hand-compute round-down via arithmetic on the source.
3. Open an issue requesting a paired `*CEIL`/`*FLOOR` Conn.

### `#[ignore]`d properties

None.

### Design deviations from the plan

- **Plan T4** proposed splitting `law_battery!`'s `full` subset into
  `full` (L+R only) and a new `triple` subset (L+R + `floor_le_ceil`).
  Implementation chose to **strengthen `full` in place** instead — the
  plan's `triple` would add a third subset that's structurally equal
  to "what `full` *should* mean for a triple marker", and splitting
  would just push the work of "remember to use the stricter subset"
  onto every downstream caller. Adding to `full` is the principled
  choice: a marker that ships both `ViewL` and `ViewR` is *defined*
  by the four-and-a-half axioms `full` now covers; if it doesn't pass
  them all, it shouldn't be a triple marker.
- **Plan T6** was supposed to also flag `f09sys` (Haskell parent
  library bug at `Finite SystemTime::MAX`) for upstream. Captured here
  in the plan but not yet filed; the upstream issue is a separate
  follow-up, since this sprint is scoped to the Rust port.
