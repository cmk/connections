# Plan 50 — CI usage reduction + docs.rs parity

Reduce GitLab CI job-minutes without weakening the merge gate. The
starting point is the 2026-05-08 CI review of the last 30 GitLab
pipelines: roughly 274 job-minutes total, with `test` (~46%),
`clippy` (~18%), `doc` (~13%), and `fmt` (~11%) dominating. The
pipeline is already optimized in several places (no `target/` cache,
prebuilt `cargo-nextest` / `cargo-deny`, pull-only registry cache for
most jobs, and scheduled/tag-only Kani). This sprint is the next trim:
remove duplicated pipeline classes, cancel stale work, and align docs
jobs with the current docs.rs feature set.

## Scope

1. Avoid the initial branch pipeline before an MR exists.
2. Add interruptible / auto-cancel behavior for superseded work.
3. Slim the post-merge `main` pipeline.
4. Skip Rust build/test jobs for docs-only and review-only changes.
5. Revisit the standalone `fmt` job.
6. Fix the docs.rs / CI rustdoc feature mismatch.

## Non-goals

- Do not weaken the MR merge gate for source-affecting changes.
- Do not change Kani cadence; it stays tag / schedule / manual.
- Do not reintroduce `target/` caching.
- Do not change the crate MSRV.

## T1 — Push/MR Workflow: Avoid Duplicate First Pipelines

Problem: the current workflow pushes the branch first, then opens the
MR. Because no MR exists at the moment of first push,
`CI_OPEN_MERGE_REQUESTS` cannot suppress the branch pipeline. In the
sampled pipelines, first-push branch pipelines cost roughly 25
job-minutes.

Implementation:

- Update `CLAUDE.md` sprint step 9 to use `git push -o ci.skip -u
  origin sprint/<name>` for the first branch publication, followed by
  `glab mr create`.
- Document why this is only for the initial branch publication:
  GitLab's `ci.skip` push option skips branch pipelines but does not
  skip merge-request pipelines.
- Leave subsequent review-round pushes alone. Once the MR exists,
  the existing `workflow: rules` should route pushes to MR pipelines
  and suppress branch duplicates.
- Update any nearby prose that implies the first push pipeline is
  expected to run before the MR is created.

Verification:

- `rg -n "git push|ci.skip|CI_OPEN_MERGE_REQUESTS|merge_request" CLAUDE.md doc/workflow.md .gitlab-ci.yml`.
- Next real MR should show no initial branch pipeline for the first
  publication push, and should still show the MR pipeline.

## T2 — Auto-Cancel Superseded Work

Problem: when a branch receives another push while CI is running,
older work keeps consuming job-minutes. A failed format check should
also cancel already-started downstream work instead of letting it burn
to completion.

Implementation:

- Add `workflow:auto_cancel:on_new_commit: interruptible`.
- Add `workflow:auto_cancel:on_job_failure: all` if GitLab accepts it
  in this project.
- Set `default: interruptible: true` for ordinary check/test/audit
  jobs.
- Override `interruptible: false` only where cancellation would be
  semantically wrong. The likely candidate is `pages`; if a newer
  `main` pipeline supersedes it, canceling the older deploy is still
  acceptable, but this should be decided explicitly in the patch.
- Keep `claude-review` interruptible: a stale advisory review on an
  older diff is actively undesirable.

Verification:

- `glab ci lint`.
- Push two quick updates to a test MR branch and confirm the older
  running pipeline/jobs are canceled.
- Create or use a failing `fmt` branch and confirm downstream jobs do
  not keep burning time after the first failure.

## T3 — Slim Post-Merge `main` Pipelines

Problem: this repo uses MR-gated linear history. After the MR pipeline
has passed, the `main` pipeline repeats `fmt`, `clippy`, `doc`, and
`test`, then adds `deny`, `gitleaks`, and `pages`. In the sampled
data, main-pipeline repeats were the largest potential savings.

Conservative implementation:

- Keep `deny`, `gitleaks`, and `pages` on `main`.
- Remove the separate `doc` job from `main`; `pages` should run the
  same strict rustdoc command and therefore serve as the post-merge
  rustdoc check.
- Consider making `fmt`, `clippy`, and `test` MR-only, not `main`,
  after validating that protected-branch rules require successful MR
  pipelines before merge.

Risk decision:

- The lowest-risk version only removes duplicate `doc` on `main`
  because `pages` already rebuilds docs.
- The high-savings version also removes `fmt`, `clippy`, and `test`
  from `main`; it assumes no direct pushes to `main` bypass MR CI.
  If direct protected-branch pushes remain allowed, do not take the
  high-savings version.

Verification:

- `glab ci lint`.
- Inspect branch protection / merge checks in GitLab before removing
  `fmt`, `clippy`, or `test` from `main`.
- After merge, confirm `main` still runs `deny`, `gitleaks`, and
  `pages`, and that the Pages deploy fails on rustdoc warnings.

## T4 — Skip Rust Jobs for Docs-Only / Review-Only Changes

Problem: review-doc, plan-doc, and prose-only changes currently run
the Rust build/test surface. That is useful before broad release
work, but wasteful for routine review-doc mirrors and plan edits.

Implementation:

- Add shared source-impact path rules for Rust jobs:
  `Cargo.toml`, `Cargo.lock`, `README.md`, `rust-toolchain.toml`,
  `.gitlab-ci.yml`, `.config/nextest.toml`, `deny.toml`,
  `src/**/*`, `tests/**/*`, `.githooks/**/*`, and scripts that are
  invoked by CI or hooks.
- Apply those rules to `clippy` (including the folded format check),
  `test`, and `doc` so they run for source-impacting changes and skip
  for `doc/plans/**` / `doc/reviews/**` only.
- Keep `gitleaks` and `deny` on their existing `main` / schedule
  cadence.
- Ensure `.gitlab-ci.yml` edits always run the relevant jobs; CI
  changes should not self-skip.

Verification:

- `glab ci lint`.
- Open or simulate a docs-only MR and confirm Rust jobs are omitted.
- Open or simulate a source-changing MR and confirm folded
  format+clippy, `test`, and `doc` are present.

## T5 — Revisit the Standalone `fmt` Job

Problem: the standalone `fmt` job costs about one minute per pipeline,
mostly image/component setup. Local hooks already enforce formatting,
and `clippy` has to compile the crate anyway. However, a standalone
`fmt` job gives faster, clearer failure output.

Implementation options:

1. **Keep standalone `fmt` but make it more skippable.** Combine with
   T4 and leave behavior otherwise unchanged. Lowest risk; least
   savings.
2. **Fold `fmt` into `clippy`.** Run `cargo fmt --all -- --check`
   as the first `clippy` script line and delete the separate `fmt`
   job. Saves the job startup cost but makes formatting failures
   appear under `clippy`.
3. **Make `fmt` manual for MR pipelines.** Not recommended unless
   GitLab usage becomes tight; it weakens an inexpensive, precise
   gate.

Preferred implementation for this sprint: option 2. Folding `fmt`
into `clippy` preserves a blocking format gate while removing one
runner startup from every source-impacting pipeline. Also update the
stale MR template line that says the CI `fmt` job is warn-only; it is
blocking today.

Verification:

- `cargo fmt --all -- --check`.
- `glab ci lint`.
- Confirm an intentional formatting drift still fails CI somewhere
  obvious.

## T6 — docs.rs / CI Rustdoc Feature Parity

Problem: `Cargo.toml` docs.rs metadata currently builds
`byte,testing,macros`, while `.gitlab-ci.yml` `doc` and `pages` still
build `testing,macros`. That means CI is not checking the exact
stable docs.rs feature surface.

Implementation:

- Update `.gitlab-ci.yml` `doc` and `pages` commands to
  `--features byte,testing,macros --document-private-items`.
- Update comments in both jobs so they no longer claim
  `testing,macros` alone matches docs.rs.
- Update `.githooks/pre-push` to match the CI/doc feature set, or
  explicitly document why pre-push uses a smaller feature set. Prefer
  matching the CI/doc command.
- Recheck any README or workflow prose that names the old docs
  command.

Verification:

- `RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --features byte,testing,macros --document-private-items`.
- `cargo doc --no-deps --features byte,testing,macros --document-private-items`
  for the Pages command shape.
- `cargo test --workspace --doc`.

## Final Verification

Run locally:

```sh
cargo fmt --all -- --check
scripts/check-pii.sh
scripts/check_readme_mirror.sh
cargo test --workspace --quiet
cargo clippy --all-targets --quiet -- -D warnings
RUSTDOCFLAGS="-D warnings" cargo doc --no-deps --features byte,testing,macros --document-private-items
glab ci lint
```

Expected CI shape after the sprint:

- Source-changing MR: folded format+clippy, `doc`, `test`, and
  `claude-review` when secrets are present.
- Docs/review-only MR: no Rust build/test jobs unless `.gitlab-ci.yml`,
  hooks, source, tests, or Cargo metadata changed.
- First publication push before MR creation: skipped branch pipeline.
- Review-round push after MR exists: one MR pipeline, stale older
  pipelines canceled.
- `main`: `deny`, `gitleaks`, and strict `pages` for landed changes;
  folded format+clippy and `test` only for source-impacting direct
  pushes.

## Review Notes

- T3 is the only policy-sensitive task. Branch protection still
  permits Maintainer pushes to `main`, so this sprint uses the
  conservative T3 variant: keep `clippy` / `test` on source-impacting
  `main` pushes and remove only the duplicate strict rustdoc job.
- T5 folds `fmt` into `clippy` for actual job-minute savings. The
  format failure will now appear under the `clippy` job name.
- T1 is mostly documentation/workflow, but it has the clearest
  immediate savings for this repo's current "push then create MR"
  routine.
