# Plan 28 — Port template-rust's Layer 2 `.githooks/pre-commit` to connections

## Goal

Add a git-side pre-commit hook (`.githooks/pre-commit`) to connections that
runs the same fmt / pii / readme-mirror / test / clippy chain as the
existing Claude Code `PreToolUse` hook, but **at git's standard
`pre-commit` hook point** — after staging, before the commit object is
created, in the actual repo working directory. Document the two-layer
approach in CLAUDE.md and the activation step every fresh clone needs.

## Motivation

MR !42 (Plan 24) shipped a clippy `bool_assert_comparison` lint to GitLab
CI and got caught there. Diagnosing locally why the pre-commit hook
silently no-op'd uncovered a structural gap, not a matcher quirk:

> **The Claude Code `PreToolUse` hook runs in Claude Code's session cwd**,
> not in the cwd a `cd` inside the matched command would reach. Every
> Plan 23 / Plan 24 commit used the shape `cd <worktree> && git add … &&
> git commit -m "…"`. The hook fired on the matched compound command, but
> its `cargo fmt`/`cargo clippy` checks ran in the project root (= main
> worktree, at the previously-merged state — clean). The lints in the
> dirty plan-N worktree never got checked. CI caught one; the rest were
> lucky.

Confirmed via Claude Code docs: hook input JSON includes a `cwd` field
recording the cwd at hook-invocation time, and "hooks run in the current
working directory at the time the hook is invoked." Subsequent `cd`s
inside the matched Bash call don't apply.

This is the design the existing **Layer 1** hook makes — fast feedback
during agent iteration, runs *before* the Bash tool's command body. By
construction, it can't see worktree state. **Layer 2** — a git-side
`pre-commit` hook — fires at git's standard hook point, *after* staging,
in the worktree where the commit is being made. Same checks, run in the
right place, regardless of how the commit was invoked (chained Bash,
agent-direct, terminal, IDE).

template-rust already ships this two-layer design; connections didn't,
so it inherited none of the safety net.

## Tasks

### T1 — Add `.githooks/pre-commit`

Port template-rust's `.githooks/pre-commit` to connections, extending the
chain by one step (connections has `scripts/check_readme_mirror.sh`,
template-rust doesn't):

```bash
#!/usr/bin/env bash
set -euo pipefail
repo_root=$(git rev-parse --show-toplevel)
cd "$repo_root"

cargo fmt --all -- --check
scripts/check-pii.sh
scripts/check_readme_mirror.sh
cargo test --workspace --quiet
cargo clippy --all-targets --quiet -- -D warnings
```

`set -euo pipefail` ensures any step's nonzero exit code aborts the chain;
git's pre-commit hook contract aborts the commit on a nonzero hook exit.
Mark the file executable in the commit (`git update-index --chmod=+x`).

### T2 — Update CLAUDE.md §Pre-commit hook

Rewrite the §Pre-commit hook section to describe both layers explicitly,
mirroring template-rust's framing:

- **Layer 1** — Claude Code `PreToolUse`. Fast feedback during agent
  iteration. Limitation: runs in Claude Code's session cwd, so chained
  `cd <worktree> && git commit` calls have it checking the *launch* cwd,
  not the worktree. Useful for direct `git commit` runs from the project
  root.
- **Layer 2** — Git `pre-commit` hook. Runs at git's standard hook point
  in the actual worktree, regardless of invocation shape. Unbypassable
  safety net.
- **Activate Layer 2** on a fresh clone:
  `git config core.hooksPath .githooks`.

Keep the existing five-step chain description; both layers run the same
chain.

## Why MR !43 was rewritten

Initial diagnosis blamed Layer 1's matcher pattern (`Bash(git commit*)`)
for missing compound `cd … && git commit …` invocations, and the first
two commits on this branch added a permissive `matcher: "Bash"` + `if:
"Bash(* git commit *)"` hedge. That fix is structurally unnecessary —
matchers DO match subcommands of compound Bash calls per the Claude Code
docs; template-rust's CLAUDE.md confirms the same. The bug was the cwd
behavior of where the hook command runs, not which commands trigger it.

Reset the branch to `origin/main` and redid the work with the correct
diagnosis. The matcher hedge is dropped; `.githooks/pre-commit` is the
real fix.
