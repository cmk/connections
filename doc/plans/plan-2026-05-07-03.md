# Plan 47 — feature-flagged `byte` module: sortable byte-encoding Conns

## Goal

Add a `byte` cargo-feature-gated module exposing **lawful** Conns
between every primitive numeric host type and a fixed-width
sortable byte representation `[u8; N]`. "Lawful" here means
order-preserving in both directions, so each Conn ships as an
`iso!` (degenerate Galois, `floor = ceil = forward`) and round-trips
exactly. Verify the laws with **Kani** for byte-widths where the
SMT input space stays tractable (1–4 bytes), fall back to
**proptest** for 8 / 16-byte hosts.

This is **not** a `bytemuck` dependency. Bit-pattern `as`-casts and
endian transmutes (`f32::to_bits`, `i32 as u32`, `u32::to_le_bytes`)
are **not** Galois — they break monotonicity at the sign bit and at
the float sign+exponent boundary. The "byte" half of these Conns is
a **totalOrder-encoded** byte array, the same construction the crate
already uses internally for ULP walks (`signed32` /
`signed16`, `src/float.rs:685`, `src/float/f16.rs:28`). Lifting the
existing helpers to first-class Conns is the contribution.

## Context

### Why this isn't bytemuck

[`bytemuck`](https://docs.rs/bytemuck) provides
`cast::<A, B>` for `A: NoUninit, B: AnyBitPattern` of the same size —
a typed reinterpretation of memory. Three forms are relevant:

1. **`f32 ↔ u32` via `to_bits` / `from_bits`** — bijective on bit
   patterns; **not monotone** (`(-0.0).to_bits() = 0x80000000 >
   f32::INFINITY.to_bits() = 0x7F800000`). Fails Galois.
2. **`i32 ↔ u32` via `as` cast** — bijective; **not monotone**
   (`i32::MIN as u32 = 0x80000000`, larger than `i32::MAX as u32 =
   0x7FFFFFFF`). Fails Galois.
3. **`uN ↔ [u8; N]` via `to_be_bytes` / `to_le_bytes`** — bijective.
   `BE` is monotone on **unsigned** hosts only; `LE` is never monotone
   above 1-byte width.

The crate has resisted shipping bit-fiddling Conns precisely because
they aren't lawful. The fix is the well-known "sortable encoding"
trick: pre-condition the bit pattern so byte-lex order matches
numeric order, then `to_be_bytes`. This is what RocksDB's
`BytewiseComparator`, FoundationDB's `Tuple`, and FaunaDB's index
keys all use. The crate already has the float half implemented for
ULP walking (`signed32` at `src/float.rs:685`); this sprint exposes
it.

### Sortable encoding — the four cases

For every primitive `T` of byte-width `N`, define
`forward: T → [u8; N]` such that `to_be_bytes` post-composed with
the appropriate per-family pre-encoding gives an order-preserving
bijection. The four families:

| Host family | Pre-encoding (`T → uN`)                                 | Order-preserving? |
|-------------|---------------------------------------------------------|-------------------|
| `uN`        | identity                                                | yes (trivially)   |
| `iN`        | `(x as uN) ^ (1 << (N*8 - 1))` (flip sign bit)          | yes               |
| `fN` (IEEE) | `let b = x.to_bits(); if b >> (N*8-1) == 0 { b ^ (1 << (N*8-1)) } else { !b }` | yes (matches IEEE 754 totalOrder) |
| `bool`      | `x as u8`                                               | trivial (one-sided `conn_l!` — `inner` is non-injective) |

The reverse direction inverts the pre-encoding. For each pair the
round-trips are bit-exact (no precision loss); both Galois laws and
`floor_le_ceil` hold because `inner` is a true two-sided inverse.

### Existing internal precedent

| Helper                                | Location              | Role                          |
|---------------------------------------|-----------------------|-------------------------------|
| `signed16` / `unsigned16`             | `src/float/f16.rs:28` | f16 totalOrder ↔ i32          |
| `signed32` / `unsigned32`             | `src/float.rs:685`    | f32 totalOrder ↔ i32          |
| `Ipv4Addr::{from,to}_bits`            | `src/addr/ip.rs:38`   | u32 ↔ Ipv4Addr (ordered iso)  |
| `Ipv6Addr::{from,to}_bits`            | `src/addr/ip.rs:74`   | u128 ↔ Ipv6Addr (ordered iso) |

The `addr` module is the closest existing analog: an `iso!`-built
total bijection between a numeric primitive and a wrapper whose
derived `Ord` matches the primitive's. The `byte` module follows
exactly the same shape, just with `[u8; N]` (lex-ordered) on the
right.

| `I128OBYT`| `i128 ↔ [u8; 16]`        | `byte::sixteen`       |

**Total: 14 Conns.**

`OBYT` lives in the right (coarser) slot per the placement rule —
the byte array is the more-general representation; the typed primitive
is the more-specific endpoint. (Same logic as `IPV4` sitting on the
right of `U032IPV4`.)

`BOOLOBYT` uses 4L+0D on both sides, matching the existing `ABCDXYZW`
convention used by `time::*` and `addr::*`.

### Note on `[u8; N]` orderability

`[u8; N]` derives `Ord` lexicographically — exactly what we want.
No newtype required; the array type is its own canonical sortable
representation. (`Ipv4Addr` adds little over `[u8; 4]`; here we
intentionally don't add a wrapper to avoid a second name to learn.)

### T1 — Helper functions (`src/byte.rs` + per-width files)

Three categories, one fn pair per host width:

```rust
// Unsigned: identity bit projection.
const fn u8_to_obyt(x: u8) -> [u8; 1] { [x] }
const fn obyt_to_u8(b: [u8; 1]) -> u8 { b[0] }

// Signed: flip sign bit then BE bytes.
const fn i8_to_obyt(x: i8) -> [u8; 1] {
    [(x as u8) ^ 0x80]
}
const fn obyt_to_i8(b: [u8; 1]) -> i8 {
    (b[0] ^ 0x80) as i8
}
// (Wider widths use `to_be_bytes` / `from_be_bytes` after the XOR.)

// Float: totalOrder pre-encoding then BE bytes.
fn f32_to_obyt(x: f32) -> [u8; 4] {
    let b = x.to_bits();
    let sortable = if (b >> 31) == 0 { b ^ 0x8000_0000 } else { !b };
    sortable.to_be_bytes()
}
fn obyt_to_f32(arr: [u8; 4]) -> f32 {
    let sortable = u32::from_be_bytes(arr);
    let b = if (sortable >> 31) != 0 { sortable ^ 0x8000_0000 } else { !sortable };
    f32::from_bits(b)
}
```

The float helpers reuse the proven `signed*` / `unsigned*`
constructions from `src/float.rs:685` — same algebra, different
output type (`[u8; N]` instead of `i32`). Cross-reference both via
doc-link from the new helpers.

`f16` helpers gated `#[cfg(feature = "f16")]`. Reuse
`crate::float::f16::{signed16, unsigned16}` if their semantics
match; otherwise inline the 16-bit version.

### T4 — `bool` Conn (`byte::one`)

1 Conn via `iso!`: `BOOLOBYT : bool ↔ [u8; 1]`. Forward is
`(b as u8).into()`; back is `b[0] != 0` — total in both directions
(any `u8` decodes to `false` if zero, `true` otherwise; not strictly
bijective on the byte side because `1`–`255` all collapse to `true`).
**That collapse means `BOOLOBYT` is *not* an iso** — it's a one-sided
`conn_l!` with order-preserving `ceil` and lossy `inner`. Ship as
`conn_l!`, drop the `iso_roundtrip_l` and `roundtrip_ceil` props for
this one Conn (they don't apply), keep the rest.

(`char` was originally part of T4; deferred — see Deferred section.)

### T6 — Proptest battery (`src/byte/tests.rs` or per-file `#[cfg(test)]`)

For each Conn, three properties (≥3 because `iso!` makes the full
triple lawful, so all three need to hold):

| Property            | What it asserts                                    |
|---------------------|----------------------------------------------------|
| `iso_roundtrip_l`   | `inner(ceil(a)) == a` for every `a: T`             |
| `roundtrip_ceil`    | `ceil(inner(b)) == b` for every `b: [u8; N]`       |
| `order_preserving`  | `a ≤ b ⟹ ceil(a) ≤ ceil(b)` (lex on bytes)        |

Float strategies use the existing `arb::extended_float_*` from
`src/prop/arb.rs`; integer strategies use `proptest::any::<T>()`
biased by `prop_oneof!` toward `MIN`/`MAX`/`0` per CLAUDE.md.

NaN handling: `f32::NAN.to_bits() != f32::NAN.to_bits()` is **false**
(NaN payloads are preserved through `to_bits`/`from_bits`), so the
round-trip property holds bit-exactly even for NaN. Order-preserving
for NaN holds **iff** we adopt IEEE totalOrder semantics (NaNs sort to
the extremes); proptest assertion uses `f32::total_cmp` for the
ground-truth comparison, not `<`.

## Deferred

### `CHAROBYT` (`char ↔ [u8; 4]`)

Rust's `char` is a **strict subset** of `u32`: any value in
`0..=0x10FFFF` *except* the surrogate range `0xD800..=0xDFFF` (those
2,048 codepoints are reserved for UTF-16 surrogate pairs). That
puncture makes the Conn awkward:

- Forward (`char → [u8; 4]`) is total — `(c as u32).to_be_bytes()`.
- Back (`[u8; 4] → char`) is **partial**: ~2.1M out of 2³² byte
  arrays decode to no valid `char`. `char::from_u32(...).unwrap()`
  panics on those — and an `inner` fn is supposed to be total.

Three landing options the next revision can pick from:

1. **Wrap source as `Extended<char>`** and have `inner` saturate
   invalid bytes to `NegInf` / `PosInf` (pinned to the punctured
   edges). Same trick `time::TIMENANO` uses to total `Time`. Lawful
   triple. Cost: a new wrapper layer for what should be a simple
   serialization Conn.
2. **Drop `CHAROBYT` from the module** — keep the other 14 isos
   uniformly clean. Use case is rare and downstream can convert via
   `char as u32` + `U032OBYT` when needed. Picked for this sprint.
3. **Ship as `conn_l!` + saturating `inner`** — round invalid u32s
   up to the nearest valid scalar (clamp surrogates to `0xE000`,
   clamp `> 0x10FFFF` to `char::MAX`). Lawful one-sided Conn, no
   panic, but it stops being an iso and the saturation behavior is
   non-obvious to users.

This sprint takes option **2**. Revisit if a downstream crate asks.

### Other deferrals

| Item                                          | Reason                                                                              |
|-----------------------------------------------|-------------------------------------------------------------------------------------|
| **Endian variants** (`F064LBYT`, `U032LBYT`)  | LE byte order is **not** monotone above 1-byte width — it would have to ship as a non-Galois iso (raw bijection) or behind a separate `LeBytes` newtype with custom Ord. Out of scope; revisit if a downstream crate asks. |
| **Plain bit-pattern Conns** (`F032U032`)      | Not Galois (sign-bit + float-bit-layout monotonicity violations documented above). Use `signed32` directly if the algebra is needed. |
| **`bytemuck` dependency**                     | No need — std's `to_be_bytes` / `from_be_bytes` cover the contract. Adding `bytemuck` would pull a transitive `unsafe` surface for zero gain. |
| **`u128`/`i128` Kani harnesses**              | 256-bit two-input symex stalls CBMC; covered by proptest only.                      |
| **`f64` Kani harnesses**                      | Same — 128-bit symbolic. `f32` may also need T1 fallback depending on solver.       |
| **Conn lifters** (`Vec<T> ↔ Vec<[u8; N]>`)    | Would need `compose!`-style machinery on slices; orthogonal to this sprint.         |
| **SIMD types** (`__m128`, `float32x4_t`)      | Out of scope; would need ARM/x86 cfg gating and don't share a natural total order.  |

**Verification tally.**

| Surface     | Count | Status                                                     |
|-------------|-------|------------------------------------------------------------|
| Proptests   | 63    | all green; full default + `--features byte` runs both pass |
| Doctests    | 6     | all green under `--features byte --doc byte`               |
| Kani T0     | 38    | all green (1- and 2-byte hosts exhaustive; 4-byte under 64-bit symex) |

**Plan deviations.**

1. **All float-host Galois harnesses (proptest + Kani) use the
   totalOrder analog, not the classical `<=` predicate.** The plan
   listed `galois_l` / `galois_r` for every Conn. For `F032OBYT`
   (and by symmetry `F064OBYT`, `F016OBYT`) the classical predicate
   `(ceil(a) ≤ b) == (a ≤ upper(b))` fails on byte arrays that decode
   to NaN (NaN comparisons return `false` for any `<=`, breaking the
   bool-equality of the two sides). Replaced with `galois_l_total` /
   `galois_r_total` using `total_cmp`, which is the law the byte
   encoding actually implements (and the same totalOrder claim the
   documentation makes). Proptest covers all three float widths;
   Kani covers `F032OBYT` exhaustively at 64-bit two-input symex.

2. **`BOOLOBYT::galois_l` is not Kani-verified.** Kani 0.67 produces
   spurious counterexamples on this single harness — every
   concrete-playback inputs evaluates to `true == true` by hand, yet
   Kani reports a failure. Reduced pure-byte models of the same
   predicate verify cleanly, isolating the issue to `bool` PartialOrd
   dispatch under symbolic execution. The proptest battery covers
   the law; the failure is documented inline at
   `src/kani_proofs/byte_one.rs` and excluded from the harness list
   to keep CI green. Re-enable when upstream Kani fixes the bool/cmp
   interaction or by reduction to a `u8`-pair predicate that
   verifies cleanly today.

3. **README headline-doctest mirror skipped.** The
   `feedback_doctests_and_readme.md` discipline mirrors the headline
   Conn's doctest into `README.md` so `cargo test --doc` catches
   drift. That works for always-on Conns; for `byte`-feature-gated
   Conns it would need either (a) `byte` in the default features
   (rejected — keeps the dep-free crate dep-free for byte=off
   consumers), or (b) a feature-gated doctest in README, which
   `check_readme_mirror.sh` doesn't currently support. The in-source
   doctest at `src/byte/four.rs:18` covers correctness; the README's
   modules-table entry covers discoverability. Re-evaluate if
   `check_readme_mirror.sh` grows feature-gating support.

5. **Float OBYT Conns (`F016OBYT`, `F032OBYT`, `F064OBYT`) deferred
   in round 2 of MR review.** Codex flagged a soundness issue: the
   `iso!` macro emits a `ConnK` value, which formally claims the
   Galois law `(ceil(a) ≤ b) == (a ≤ upper(b))` over the host's
   `PartialOrd`. For an `f32` host, `0.0 <= NaN` evaluates to
   `false` in `PartialOrd`, while `ceil(0.0) <= ceil(NaN)` evaluates
   to `true` in `[u8; 4]` lex order — so the trait claim is false
   on NaN-decoding bytes. Round-1 §Review only acknowledged the
   *test* deviation (using `total_cmp` instead of `<=` in
   harnesses); it did not address the *type-system* claim.
   Considered fixes: (a) wrap source as `ExtendedFloat<f*>` —
   doesn't help, the inner `partial_cmp` still returns `None` for
   `Extend(NaN)` vs `Extend(finite)`; (b) introduce a totalOrder
   newtype around `[u8; N]` — feasible but heavyweight; (c) use a
   type-level `NonNan<f*>` source — feasible and clean; (d) defer
   to a follow-up sprint. Picked **(d)** for round 2 to land the 11
   integer + bool Conns now and let the float strategy be designed
   without rushing. The byte encoding algebra (signed-bit-flip,
   totalOrder pre-encoding) is correct; the issue is purely the
   Conn type's law-claim. Helpers and proptest battery removed
   along with the constants so dead code doesn't linger.
