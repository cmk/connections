### T3 — `.gitlab-ci.yml` `kani:` job

Insert a new `kani:` job before the existing `pages:` job, in the
`audit` stage alongside `deny` / `gitleaks` / `claude-review`.

Properties:

- **Image**: `rust:1.92`. The pinned project toolchain (1.85 in
  `rust-toolchain.toml`) cannot install `kani-verifier 0.67+`
  because its transitive `home 0.5.12+` requires rustc ≥ 1.88. The
  host cargo is only used to install the launcher; `cargo kani
  setup` then downloads and pins Kani's own nightly toolchain
  (currently `nightly-2025-11-21`), under which the harnesses
  actually compile.
- **Cache**: cargo registry as elsewhere, plus `.cargo/bin/` so
  the kani-verifier launcher binaries persist between runs (skips
  the ~30s `cargo install` on warm runs). A `prefix: kani-v0.67`
  scopes the cache to a Kani version line; bump the prefix to force
  a clean install.
- **Trigger**: `$CI_COMMIT_TAG`, scheduled pipelines, and a manual
  `$KANI_RUN` variable (mirrors the `$CLAUDE_REVIEW_MR` pattern).
  No per-MR or per-feature-branch run — `cargo kani setup` is too
  heavy (~1 GB toolchain download on cold runners) for the marginal
  value over the local loop.
- **Advisory**: `allow_failure: true` per plan 39's recommendation.
  A Kani regression on a tagged release informs but does not block
  — the full pre-merge gate stays on `fmt` / `clippy` / `test` /
  `deny` / `gitleaks`.
- **Retry**: `max: 1` on `script_failure` / `runner_system_failure`.
  Lower than `claude-review`'s `max: 2` because Kani failures are
  more often legitimate proof regressions than transient
  infrastructure issues.

### End-to-end scenario

After merge, the next tagged release (or a manual
`glab ci run --variables KANI_RUN=1 --branch <branch>`) will trigger
the `kani` job and run the full proof tree. A regression surfaces
in the job log without blocking the release; no MR pipeline gains
the ~3 min of Kani install + setup + run.
