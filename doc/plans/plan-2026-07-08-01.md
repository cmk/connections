# Plan 01 - Replace ExtendedFloat with N5

## Goal
Replace the current three-variant `ExtendedFloat<T>` enum with a faithful Rust
port of the Haskell `N5` float ordering, so `F016`/`F032`/`F064` carry IEEE
payloads directly and `-inf <= NaN <= +inf` supplies the float-side bounds.

## Dependency Graph
T1 -> T2 -> T3 -> T4, T2 -> T5, T3 -> T6, T4 -> T6, T5 -> T6

## Tasks

T1 - API and compatibility decision.

Problem/motivation: The current `ExtendedFloat<T>` shape adds symbolic
`Bot`/`Top` values outside IEEE `-inf`/`+inf`. That is not the upstream
Haskell `N5` semantics and makes `Extend(inf)` distinct from the advertised
top/bottom behavior. The migration needs one public type decision before code
starts.

Solution/approach: Introduce `float::N5<T>` as a transparent single-field
newtype over the IEEE payload. `F016`, `F032`, and `F064` become aliases for
`N5<f16>`, `N5<f32>`, and `N5<f64>`. Do not keep enum-style constructors.
If a short compatibility alias is useful, `type ExtendedFloat<T> = N5<T>` may
exist temporarily, but no `Bot`/`Top`/`Extend` API remains. Add `N5::new`,
`N5::into_inner`, `From<T>`, and small helper names only where they remove
call-site churn.

Types/API surface: `pub struct N5<T>(pub T)` or an equivalent transparent
newtype; aliases `F016`/`F032`/`F064` remain stable.

T2 - Port the Haskell N5 order exactly.

Problem/motivation: Haskell `Data.Order.N5` does not preserve native float
`PartialOrd`; it defines a custom preorder where NaN is reflexive, NaN is
below positive infinity, and NaN is above negative infinity.

Solution/approach: Implement `PartialEq`, `Eq`, and `PartialOrd` for
`N5<f16>`, `N5<f32>`, and `N5<f64>` by mirroring Haskell's `n5Le`:

```text
nan <= nan      = true
nan <= y        = y == +inf
x   <= nan      = x == -inf
otherwise       = x <= y
```

This keeps the crate on standard Rust law bounds (`Eq + PartialOrd`) without a
crate-local `Preorder` trait.

Types/API surface: `impl_n5_float!(f16/f32/f64)` replacing
`impl_float_ext!`.

T3 - Migrate float Conns without re-porting their algorithms.

Problem/motivation: Every Conn currently pattern-matches
`ExtendedFloat::{Bot, Extend, Top}`. Under N5 there is only the IEEE payload,
so symbolic endpoint arms must become ordinary `-inf`/`+inf` handling.

Solution/approach: Rewrite existing Rust connection bodies to unwrap `N5<T>`,
run the current rounding/walk logic, and wrap results back into `N5<T>`.
Use the Haskell library only as a boundary-law reference, especially
`Data.Connection.Float.fxxext`:

```text
upper(NegInf) = N5(-inf)
upper(PosInf) = N5(+inf)
ceil(N5(NaN)) = PosInf
floor(N5(NaN)) = NegInf
ceil(N5(-inf)) = NegInf
floor(N5(+inf)) = PosInf
```

For finite out-of-range values, preserve the existing Rust improvements:
boundary-saturation guards, round-down integer/fixed embeddings, ULP walks,
duration/epoch plateau solvers, and one-sided/triple distinctions. Do not
blindly re-port Haskell connection bodies.

Types/API surface: float-to-float Conns remain `F064F032`, `F064F016`,
`F032F016`; float-to-`Extended<T>` Conns keep their public const names and
endpoint aliases.

T4 - Rebuild strategies and law coverage around N5.

Problem/motivation: Current proptest generators include synthetic `Bot` and
`Top`; after this migration, the critical values are IEEE `-inf`, `+inf`, and
NaN. Existing tests also encode the current enum-only endpoint behavior.

Solution/approach: Replace `extended_float_*` strategies with `n5_float_*`
strategies weighted toward NaN, infinities, signed zeros, denormals, min/max
finite values, and ordinary random payloads. Update law batteries so float
Conns are checked over the full N5 domain wherever runtime is acceptable. Keep
bounded subsets only where the existing plateau solvers require them, and
document each subset next to the test.

Types/API surface: `src/prop/arb.rs` strategy rename or compatibility wrapper;
tests under `src/core/f0XX.rs`, `src/fixed.rs`, `time`, `hifi`, and integration
tests updated to construct `N5` values.

T5 - Rework float numeric ergonomics.

Problem/motivation: `ExtendedFloat` currently has enum-specific arithmetic,
fold/fold_with, constants, `Try`, and method shims. Some are meaningless for a
single-payload N5 newtype; others should simply delegate to IEEE arithmetic.

Solution/approach: Replace enum-only APIs with payload APIs:
`into_inner`/`map` instead of `fold`, simple numeric operator impls that wrap
IEEE results, and constants that wrap the underlying float constants. Revisit
the `try_trait` feature: `N5<T>` has no failure sentinel, so either remove the
`Try` impl for this type or make it an always-continue payload extraction with
an uninhabited residual, whichever gives clearer downstream ergonomics.

Types/API surface: arithmetic impls for the same float widths; docs and tests
for `try_trait`, catamorphisms, and numeric constants updated or removed.

T6 - Documentation and review calibration cleanup.

Problem/motivation: Multiple docs currently claim Haskell N5 semantics while
the implementation uses synthetic enum endpoints. The migration must leave the
type claim and examples coherent.

Solution/approach: Update `README.md`, `doc/design.md`, `src/float.rs`,
module docs, doctests, and review-calibration references to describe the
newtype semantics precisely. The `N5` type rustdoc must link to the
[modular-lattice N5 example](https://en.wikipedia.org/wiki/Modular_lattice#Examples).
Replace examples using `ExtendedFloat::Extend(x)` with `N5::new(x)` or
`F064::from(x)` depending on the final constructor decision. Explicitly
document that `+inf` and `-inf` are the N5 top/bottom for the float side.

Types/API surface: public docs for `float::N5`, `F016`, `F032`, `F064`.

|----------|--------|-----------|
| `n5_nan_reflexive` | `float` | `N5(NaN) == N5(NaN)` and `N5(NaN) <= N5(NaN)` |

## Deferred
- A compatibility shim for downstream crates may be added only if the migration
  is otherwise too disruptive. The preferred result is a clean `N5` API, not a
  second symbolic endpoint spelling.
- Native `f128` remains out of scope.
- Re-porting Haskell connection bodies is out of scope; Haskell is the law and
  boundary reference only.
