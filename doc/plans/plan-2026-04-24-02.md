# Plan ŌĆö `compose_conn!` declarative composition macro

## Goal

Add a declarative macro `compose_conn!` that composes two
`const Conn`s at compile time into a fresh `const Conn`, without
needing heap allocation, closure capture, or a new `DynConn`
variant. Gives callers a Category-style `>>>` for the compile-
time-known case that covers ~95% of real usage.

## Motivation

`Conn<A, B>` stores three bare `fn` pointers by design (`const`-
constructible, `Copy`, no heap). Generic composition `(.then)` is
impossible at the fn-pointer level because the composed fn would
need to capture both input Conns, which bare fn pointers can't do.

Today, any "chain two Conns" use-case resolves to one of:

1. Hand-compose at the call site: `F06S48.ceil(F12F06.ceil(p))`.
2. Pre-enumerate the composed Conn as a named constant with its
   own triple of bare fns ŌĆö the route taken by upstream's
   `F12Sxx` constants today.

Both are fine but the named-constant path requires re-writing
the Galois arithmetic for each pair, duplicating what the
components already prove. A declarative macro that takes two
existing `const Conn`s and expands to a fresh `const Conn` whose
fn-pointer bodies invoke the inputs by name eliminates the
duplication without changing the representation.

Downstream impact: `agogo`'s `pico_to_samples` can stay a match
dispatch (a handful of rates), but if it ever wants a named
`const FOO: Conn<Pico, Rate> = compose_conn!(F12F06, F06...)`
the macro will make that trivial.

## Design

### Shape

```rust
compose_conn! {
    $(#[$m:meta])*
    $v:vis const $NAME:ident : Conn<$A:ty, $C:ty> = $FIRST:expr, $SECOND:expr $(,)?;
}
```

Expands to a full `const` declaration:

```rust
$(#[$m])*
$v const $NAME: Conn<$A, $C> = {
    fn ceil(x: $A) -> $C  { $SECOND.ceil($FIRST.ceil(x)) }
    fn inner(x: $C) -> $A { $FIRST.inner($SECOND.inner(x)) }
    fn floor(x: $A) -> $C { $SECOND.floor($FIRST.floor(x)) }
    $crate::conn::Conn::new(ceil, inner, floor)
};
```

The intermediate type `B` doesn't appear in any emitted source ŌĆö
it's inferred from `$FIRST.ceil(x): B` and passed to `$SECOND.ceil(B)`.
The user only has to state the two boundary types.

### Why full-const-decl form

The macro needs explicit `$A` and `$C` types to write the local
`fn ceil(x: $A) -> $C` signatures; fn items can't have inferred
signatures. Bundling the whole `const NAME: Conn<A, C> = ...` into
the macro invocation gives the user one place to name the types
and the macro one unambiguous parse. Trying to infer from context
(e.g. from the surrounding `let` or `const` annotation) runs into
Rust's macro-hygiene / type-inference boundary.

### Composition direction

Mirrors `then` / Haskell's `>>>`:

- `ceil` applies `$FIRST.ceil` then `$SECOND.ceil` (covariant).
- `floor` applies `$FIRST.floor` then `$SECOND.floor` (covariant).
- `inner` applies `$SECOND.inner` then `$FIRST.inner`
  (contravariant ŌĆö embedding goes backwards).

Matches `doc/design.md`'s ┬¦Composition description.

## Tasks

### T1 ŌĆö Macro definition

New file `src/compose.rs` (or inline at the top of `src/conn.rs`)
with:

```rust
#[macro_export]
macro_rules! compose_conn {
    (
        $(#[$m:meta])*
        $v:vis const $NAME:ident : Conn<$A:ty, $C:ty> = $FIRST:expr, $SECOND:expr $(,)?;
    ) => {
        $(#[$m])*
        $v const $NAME: $crate::conn::Conn<$A, $C> = {
            fn ceil(x: $A) -> $C  { $SECOND.ceil($FIRST.ceil(x)) }
            fn inner(x: $C) -> $A { $FIRST.inner($SECOND.inner(x)) }
            fn floor(x: $A) -> $C { $SECOND.floor($FIRST.floor(x)) }
            $crate::conn::Conn::new(ceil, inner, floor)
        };
    };
}
```

3. **Adjoint laws**: a proptest exercising the composed Conn's
   Galois law, reusing the existing `property::*` shared
   strategies.

   F12F06) == F12F06` at representative inputs, and the mirror
   for right-identity. Tests that the macro doesn't interact
   weirdly with `Conn::identity()`.

## Deferred

- **Rewrite upstream `F12Sxx` constants as `compose_conn!` expansions.**
  Possible follow-up refactor: the six `F12S44..F12S192` are
  today written directly with hand-computed NUM/DEN. They could
  instead be `compose_conn!(F12F06, F06Sxx)` where `F06Sxx` is a
  new intermediate Conn<Micro, Sxx>. Would eliminate ~60 lines
  of duplicated arithmetic but requires introducing the
  intermediate Conns first. Out of scope for this MR.
- **Runtime composition (`DynConn`).** Remains the correct
  solution for runtime-parameterised cases (Sxx chosen at
  runtime). Deferred until a use-case emerges; `agogo` just
  removed its one such case (`PicoSampleConn`) via match dispatch.
