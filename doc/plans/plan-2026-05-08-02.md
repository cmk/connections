# Plan — Narrow `lift_l!` / `lift_r!` to `:path` (MR !74 follow-up)

## Context

The `claude-review` CI bot on MR !74 flagged a real footgun in
`lift_l!` and `lift_r!`. Both macros take `$parent:expr` and reference
`$parent` **twice** in the body — once inside the `ceil`/`floor`
closure and once inside the `inner` closure. Each closure is non-
capturing and coerces to `fn(_) -> _`, so for any path-resolvable
function-call expression like `lift_l!(build_conn())`, the macro
silently calls `build_conn()` *on every invocation* of the resulting
Conn's `.ceil()` / `.upper()` — once per closure, two evaluations per
round-trip.

For pure constructors that's wasteful but observably correct. For
anything side-effecting (or anything that returns a different `Conn`
each call), the lifted Conn would be malformed in a way the trait-
claim audit can't catch — the closures get out of sync.

The Rust closure-coercion check rules out the *let-bound local*
case automatically, but the *function-call-on-a-global* case slips
through `:expr`. `lift_k!` already uses `:path`, which catches the
function-call case at parse time. Aligning `lift_l!`/`lift_r!` to
`:path` removes the inconsistency and the footgun in one move.

The bot also flagged a misleading comment on
`lift_k_compose_chain_runtime_smoke` — it described the constraint
as "must be referenceable as paths" but `EXTQ000I016.conn_l()` is a
method-call expression, not a path. The actual constraint is "must
not capture environment", which is weaker and accepts method calls
on global paths.

## Goal

Narrow `lift_l!` and `lift_r!` from `$parent:expr` to `$parent:path`,
update the one in-tree expression-form call site, and rewrite the
runtime-smoke comment to describe the constraint correctly.

## Tasks

### T1 — Narrow `lift_l!` / `lift_r!` to `:path`

- `src/extended.rs`: change both macro patterns from `$parent:expr`
  to `$parent:path`.
- Update the rustdoc on each macro to spell out the
  double-evaluation rationale (cite the bot's finding) and the
  workaround (pre-bind to a `const` first).
- Update the module-level paragraph (line ~31) to drop "if `parent`
  is a stable path" wording — the constraint is now structural.

### T2 — Refactor `LIFTED_ID_I64`

The only in-tree expr-form call site:

```rust
// before
const LIFTED_ID_I64: Conn<Extended<i64>, Extended<i64>, L> =
    lift_l!(Conn::<i64, i64, L>::identity());
```

becomes a pre-bound chain:

```rust
const ID_I64: Conn<i64, i64, L> = Conn::identity();
const LIFTED_ID_I64: Conn<Extended<i64>, Extended<i64>, L> = lift_l!(ID_I64);
```

`Conn::identity()` is `const fn`, so `ID_I64` is a const path that
`:path` accepts. No semantic change.

### T3 — Clarify runtime-smoke comment

Replace the misleading "Both compose_l! operands must be referenceable
as paths" with a paragraph that:

- States `compose_l!` is `:expr`-fragmented (looser than
  `lift_l!`/`lift_r!`).
- Names the actual constraint: each operand must compile to a
  non-capturing closure body (so the macro-synthesized closures
  coerce to `fn(_) -> _`).
- Explains why `EXTQ000I016.conn_l()` (a method call) qualifies:
  `EXTQ000I016` is a unit-struct path with static dispatch — the
  closure body re-evaluates `.conn_l()` on each invocation but
  captures nothing.
- Notes the let-bound counter-example for completeness.

## Critical files

  smoke comment.

## Deferred

- Auditing `compose_l!` / `compose_r!` for the same double-evaluation
  hazard. These macros use `:expr` and have the same structure
  (operands referenced inside two non-capturing closures). The bot
  didn't flag them — they predate this MR — and a structural fix
  would touch every compose call site in the crate. Treat as a
  separate (much larger) sprint if pursued.
- A compiler-side enforcement that `:path` resolves to a global item
  (would also reject let-bound paths). Not currently expressible in
  `macro_rules!`; a procedural macro could do it but is overkill for
  this footprint.

## Review (post-implementation)

- All three bot findings addressed in one commit.
- Test count unchanged (40 lift tests, 6 lift doctests, 1494 total
  lib tests). The narrowing is purely API-tightening; behaviour on
  every existing call site is identical.
- No `#[ignore]`d tests added.
