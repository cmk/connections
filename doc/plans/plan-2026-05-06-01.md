### T2 ŌĆö Replace `pub fn interval` body

In `src/conn.rs:471`, replace the existing `Option<Ordering>` body
with the bracket function:

```rust
pub fn interval<T, A, B>(t: &T, x: A) -> Interval<A>
where T: ConnK<A, B>, A: Copy + PartialOrd, B: Copy {
    let lo = t.conn_r().lower1(|b| b, x);
    let hi = t.conn_l().upper1(|b| b, x);
```

### T4 ŌĆö Rewrite `round` to consume `Interval<A>`

Today's `round` (`src/conn.rs:660-664`) ties **toward zero**
(`Equal | None ŌåÆ truncate(t, x)`). Preserve that:

```rust
pub fn round<T, A, B>(t: &T, x: A) -> B
where T: ConnK<A, B>,
      A: Copy + PartialOrd + Sub<Output = A> + From<u8>,
      B: Copy
{
    match interval(t, x) {
        Interval::Bounded { lo, hi } => {
            match (x - lo).partial_cmp(&(hi - x)) {
                Some(Ordering::Greater) => t.conn_l().ceil(x),
                Some(Ordering::Less)    => t.conn_r().floor(x),
                _                       => truncate(t, x),
            }
        }
        Interval::Empty => truncate(t, x),
    }
}
```

Bound delta: drops `Add + Div` (was needed by `midpoint`); keeps
`Sub + From<u8> + PartialOrd`. Strict relaxation ŌĆö every Conn that
satisfied the old bounds also satisfies the new.

## Review

### Property reformulations

Two predicates planned in T6 / T7 were reformulated mid-sprint
because the originals were either *false against the spec* or
*tautological*. Both reformulations are sound and ship in the
sprint as named in this Review.

- **`bracket_idempotent` ŌåÆ `bracket_endpoints_self_bracket`.**
  The original "bracket of `(lo, hi)` re-brackets to the parent
  bracket" failed against five batteries (`F064F032`, `addr::ip`
  (`U032IPV4` / `U128IPV6`), `char::U032CHAR`, `time::STDRU064`,
  `time::DURNSECS`). The failure is structural, not a bug: bracket
  endpoints `lo = lower(floor(x))` and `hi = upper(ceil(x))` are
  themselves grid-aligned closure points, so their own brackets
  are *singleton* intervals (`Bounded { lo, hi: lo }` and
  `Bounded { lo: hi, hi }`), not the parent `Bounded { lo, hi }`.
  The reformulated property asserts exactly that ŌĆö each endpoint
  re-brackets to its own singleton ŌĆö and it captures the correct
  idempotence-flavored invariant the original tried to express.
  Wired into `@batch full` and `@batch numeric_only`.

- **`imap_monotonic_preserved` ŌåÆ `imap_identity_preserves` +
  `imap_saturating_add_preserves`.** The original property's
  intent ŌĆö "any monotone `f` preserves the bracket through `imap`"
  ŌĆö needed a way to construct an arbitrary monotone `f` from
  `i32`-arbitrary input, which proptest can't do generically. The
  initial substitution shipped only `imap_identity_preserves`
  (`f = id`) which is trivially monotone; the reviewer correctly
  flagged that as too weak. Round 1 of feedback adds
  `imap_saturating_add_preserves`, which exercises a strictly
  monotone non-decreasing `f(a) = a.saturating_add(k)` over
  `i64 ├Ś i64 ├Ś i64`. Saturation gives `f` a well-defined extreme
  behavior (no overflow), so the property runs against the full
  `i64` range without filter-rejection. Together the two cover
  the planned monotone-preservation invariant: identity for the
  fixpoint case, saturating-add for the strictly-shifting case.

### Battery wiring: `numeric_only` subset

The plan's T6 listed `round_picks_endpoint` as a property that
must be wired into `@batch full`. During implementation, I
discovered that wiring it directly would tighten `full`'s blanket
bounds on `A` to `Sub<Output = A> + From<u8>`, which excludes the
`addr::ip` (`Extended<u32>` / `Extended<u128>`), `char::U032CHAR`
(`Extended<char>`), and `time::DURNSECS` (`Extended<i64>` /
`time::Duration`) batteries ŌĆö none of those types impl
`Sub<Output = Self>` and `From<u8>` together.

Round 1 of feedback resolves this by adding a new battery subset
`numeric_only` ŌĆö a strict superset of `full` that ships the three
arithmetic-bound contract properties (`round_picks_endpoint`,
`truncate_picks_endpoint`, `truncate_toward_zero`) on top of
`full`'s 14 tests. `F064F032` (the only `full`-mode numeric
battery in this crate) opts in via `subset: numeric_only`; the
non-numeric `full` batteries (`addr::ip::*`, `char::U032CHAR`,
`time::*`) stay on `full` and remain green.

### Strengthened predicates: `filter_*_via_*`

`filter_l_dual_to_ceil` and `filter_r_dual_to_floor` shipped in
the initial sprint as definitional restatements of the predicate
bodies (`c.filter_l(a, b) == (c.ceil(a) <= b)` is literally the
body of `filter_l`). Round 1 of feedback renames them to
`filter_l_via_upper` and `filter_r_via_lower`, asserting the
*Galois-adjoint characterizations* ŌĆö `filter_l(a, b) Ō¤║ a Ōēż upper(b)`
and `filter_r(a, b) Ō¤║ lower(b) Ōēż a` ŌĆö which are non-trivial: they
hold only because `ceil ŌŖŻ upper` (resp. `lower ŌŖŻ floor`) is a
true Galois adjunction. A regression that broke L/R Galois
without breaking `filter_l`'s body would now show up here.

### Recommendations

- Future work: if a second numeric `full`-mode triple is added
  (e.g. `F064F016` once `f16` stabilizes, or any `Extended<iN>`
  triple bridging via decimal SI), opt it into `subset:
  numeric_only` so the `round` / `truncate` contract picks it up
  for free.
- `filter_l_via_upper` / `filter_r_via_lower` overlap semantically
  with `galois_l` / `galois_r` (both encode the Galois adjunction).
  They differ only in surface form ŌĆö the former checks the law
  through `filter_l` / `filter_r`, the latter through the raw
  inequality. Keeping both costs little and gives independent
  failure surfaces; consolidate only if proptest cost becomes a
  measured concern.
