# Plan 29 (refined) — Strict 2-fn `Conn` with marker-type triples

## Context

The current Rust port collapses Haskell's `Cast (k :: Side) a b`
distinction into a single `Conn<A, B>` struct holding `(ceil, inner,
floor)` as three always-present `fn`-pointer fields. One-sided Conns
are encoded as triples where `floor == ceil`. The collapsed
representation paints over the difference between a Galois connection
(structurally a 2-fn pair `f ⊣ g`) and an adjoint triple
(structurally a richer object: two adjunctions sharing the middle
function). It also forces hand-audits when one-sided Conns get
their wrong-side property tests stripped (Plan 27 had to do this for
six time-domain Conns), because the type system can't distinguish
"L-Galois" from "R-Galois" from "full triple" at compile time.

This plan replaces that representation with the structural-truth
encoding:

- **`Conn<A, B, K: Kind = L>` holds exactly two fn-pointer fields.**
  No three-fn struct anywhere in the crate. The kind tag `K ∈ {L, R}`
  determines which adjoint role each field plays under the Galois
  law. `K` defaults to `L` — convention pick: most one-sided Conns
  in the crate today are `new_l`-shaped, and "left-Galois" is the
  Haskell parent's canonical form. R-kind Conns ascribe explicitly
  via `Conn<A, B, R>` or the `ConnR<A, B>` alias.
- **Adjoint triples are zero-sized marker types** that implement two
  trait projections (`ViewL<A,B>` and `ViewR<A,B>`) whose
  associated consts are the L-view and R-view `Conn`s. The "third
  function" (the leftover adjoint that distinguishes a triple from
  a one-sided Conn) lives as a free function in module scope,
  referenced from the marker's trait-impl bodies — never stored as
  data.
- **Two-sided operations** (`round`, `truncate`, `interval`,
  `midpoint`, `median`) bind on `T: ViewL<A,B> + ViewR<A,B>`,
  so they are callable only on triple markers, not on one-sided
  `Conn` consts. The previous "collapses to a point on one-sided"
  behavior is removed; one-sided Conns simply don't admit
  two-sided ops.
- **Composition of one-sided Conns** stays macro-driven, kind-
  preserving (`L ∘ L → L`, `R ∘ R → R`, mixed = compile error).
  **Composition of triples** uses a declaration-form macro that
  generates a fresh marker unit-struct with composed `ViewL` /
  `ViewR` impls.

The library is pre-1.0 and not yet on crates.io, so the public-API
rename (28 full-triple `Conn` consts becoming `Conn` unit-struct
markers) is in scope.

- `src/conn.rs` — `Conn<A, B, K>` struct, `Kind` sealed trait, `L` /

## Tasks

### T1 — Two-fn `Conn<A, B, K>` with sealed `Kind = {L, R}`

```rust
mod kind {
    mod sealed { pub trait Sealed {} }
    pub trait Kind: sealed::Sealed + Copy + 'static {}

    #[derive(Copy, Clone, Debug)] pub struct L;
    #[derive(Copy, Clone, Debug)] pub struct R;

    impl sealed::Sealed for L {} impl Kind for L {}
    impl sealed::Sealed for R {} impl Kind for R {}
}

pub use kind::{Kind, L, R};
pub type ConnL<A, B> = Conn<A, B, L>;
pub type ConnR<A, B> = Conn<A, B, R>;

pub struct Conn<A, B, K: Kind = L> {
    f: fn(A) -> B,
    g: fn(B) -> A,
    _k: PhantomData<fn() -> K>,
}

impl<A, B, K: Kind> Copy for Conn<A, B, K> {}
impl<A, B, K: Kind> Clone for Conn<A, B, K> { fn clone(&self) -> Self { *self } }
```

The two fields play different roles under each kind:

| Kind | `f`           | `g`           | Galois law                                  |
|------|---------------|---------------|---------------------------------------------|
| `L`  | lower adjoint (`ceil`)  | upper adjoint (`inner`) | `f(a) ≤ b ⟺ a ≤ g(b)` |
| `R`  | upper adjoint (`floor`) | lower adjoint (`inner`) | `g(b) ≤ a ⟺ b ≤ f(a)` |

### T2 — Kind-specific constructors

```rust
impl<A, B> Conn<A, B, L> {
    pub const fn new_l(ceil: fn(A) -> B, inner: fn(B) -> A) -> Self {
        Conn { f: ceil, g: inner, _k: PhantomData }
    }
}
impl<A, B> Conn<A, B, R> {
    pub const fn new_r(inner: fn(B) -> A, floor: fn(A) -> B) -> Self {
        Conn { f: floor, g: inner, _k: PhantomData }
    }
}
impl<T_> Conn<T_, T_, L> { pub const fn identity() -> Self { Conn::new_l(id, id) } }
```

The legacy `Conn::new(ceil, inner, floor)` constructor is **removed** —
full triples no longer correspond to a `Conn` value, they correspond
to a marker type with two `Conn` consts on it (T4). `Conn::new_iso`
is also removed; isos with both views needed become marker types,
isos used at one kind only become a single `Conn<_,_,L>` const.

### T3 — Inherent ops by kind

L-side ops live on `Conn<_, _, L>`; R-side ops live on `Conn<_, _, R>`.

```rust
impl<A: Copy, B: Copy> Conn<A, B, L> {
    pub const fn ceiling(self, a: A) -> B { (self.f)(a) }
    pub const fn upper(self, b: B) -> A { (self.g)(b) }
    // plus ceiling1 / ceiling2 / upper1 / upper2 (closure lifters)
}
impl<A: Copy, B: Copy> Conn<A, B, R> {
    pub const fn floor(self, a: A) -> B { (self.f)(a) }
    pub const fn lower(self, b: B) -> A { (self.g)(b) }
    // plus floor1 / floor2 / lower1 / lower2
}
```

`ceiling(&l_conn, x)` becomes `l_conn.ceiling(x)`. The previous free-fn
forms get either deleted or kept as thin wrappers — likely deleted in
favor of method syntax (cleaner; the rustdoc for `Conn` just lists
methods on each kind).

Calling `floor` on an `L`-Conn is a compile error (the method doesn't
exist on `Conn<_, _, L>`). Same for `ceiling` on `R`. The kind-
discipline benefit is structural, not trait-bounded.

### T4 — `ViewL` / `ViewR` traits + adjoint-triple markers

```rust
pub trait ViewL<A, B> {
    const L: Conn<A, B, L>;
}
pub trait ViewR<A, B> {
    const R: Conn<A, B, R>;
}

/// Convenience super-trait: both views available ⇒ this is an adjoint triple.
pub trait Triple<A, B>: ViewL<A, B> + ViewR<A, B> {}
impl<T, A, B> Triple<A, B> for T where T: ViewL<A, B> + ViewR<A, B> {}
```

A shipped triple is a public unit struct + two impls; the three free
fns live in module scope:

```rust
// In src/float/f32.rs:
const fn ceil_f64_to_f32(x: F064) -> F032 { ... }
const fn inner_f32_to_f64(y: F032) -> F064 { ... }
const fn floor_f64_to_f32(x: F064) -> F032 { ... }

pub struct F064F032;
impl ViewL<F064, F032> for F064F032 {
    const L: Conn<F064, F032, L> = Conn::new_l(ceil_f64_to_f32, inner_f32_to_f64);
}
impl ViewR<F064, F032> for F064F032 {
    const R: Conn<F064, F032, R> = Conn::new_r(inner_f32_to_f64, floor_f64_to_f32);
}
```

The unit struct holds nothing. The "leftover g" — the third fn —
exists only as a module-scope `const fn`, referenced inside the
`R` const expression. **No struct anywhere in the crate has 3
fn fields.**

A `triple_decl!` macro condenses the boilerplate:

```rust
triple_decl! {
    F064F032 : F064 ↔ F032 {
        ceil:  ceil_f64_to_f32,
        inner: inner_f32_to_f64,
        floor: floor_f64_to_f32,
    }
}
```

expanding to the unit struct + `ViewL` + `ViewR` impls.

### T5 — Two-sided ops bind on `Triple`

```rust
pub fn round<T, A, B>(_t: &T, x: A) -> B
where T: Triple<A, B>, A: ..., B: ... {
    // Bracket using T::L.ceiling(x) and T::R.floor(x), dispatch.
}

pub fn truncate<T, A, B>(_t: &T, x: A) -> B where T: Triple<A, B>, ... { ... }
pub fn interval<T, A, B>(_t: &T, x: A) -> Option<Ordering> where T: Triple<A, B>, ... { ... }
pub fn midpoint<T, A, B>(_t: &T, x: A) -> A where T: Triple<A, B>, ... { ... }
pub fn median<T, A>(_t: &T, w: A, x: A, y: A, z: A) -> A
where T: Triple<(A, A), A>, ... { ... }

// Plus the lifter variants round1 / round2 / truncate1 / truncate2.
```

The `_t: &T` argument is a reference to a zero-sized struct — purely a
turbofish carrier; no runtime cost. Calling `round(&l_conn, x)` where
`l_conn: Conn<A, B, L>` is a compile error: `Conn<A, B, L>` does not
implement `Triple<A, B>`.

The previous "collapses to a point on one-sided" behavior is removed.
One-sided Conns simply don't have a meaningful `round` operation — the
math says they shouldn't.

### T6 — Slot-swap (`swap_l` / `swap_r`)

Per the `Conn 'L a b ≡ Conn 'R b a` equivalence (no Ord flip):

```rust
impl<A, B> Conn<A, B, L> {
    pub const fn swap_l(self) -> Conn<B, A, R> {
        // Same fn pointers, swapped slot interpretation, swapped types.
        Conn::<B, A, R>::new_r(self.f, self.g)
        //                         ^ inner ^ floor (in R interpretation)
    }
}
impl<A, B> Conn<A, B, R> {
    pub const fn swap_r(self) -> Conn<B, A, L> {
        Conn::<B, A, L>::new_l(self.g, self.f)
    }
}
```

These are zero-cost type-level relabels. Triple markers don't get
slot-swap directly; downstream callers project to a side first
(`F064F032::L.swap_l()`) if they need the swapped form.

### T7 — Property-fn signatures by kind

```rust
// Kind-monomorphic — no implication trait needed:
pub fn conn_galois_l<A, B>(c: &Conn<A, B, L>, a: A, b: B) -> bool {
    (c.ceiling(a) <= b) == (a <= c.upper(b))
}
pub fn conn_closure_l<A, B>(c: &Conn<A, B, L>, a: A) -> bool { a <= c.upper(c.ceiling(a)) }
pub fn conn_kernel_l<A, B>(c: &Conn<A, B, L>, b: B) -> bool { c.ceiling(c.upper(b)) <= b }
pub fn conn_monotone_l<A, B>(c: &Conn<A, B, L>, a1: A, a2: A) -> bool { ... }
pub fn conn_idempotent_l<A, B>(c: &Conn<A, B, L>, a: A) -> bool { ... }

pub fn conn_galois_r<A, B>(c: &Conn<A, B, R>, a: A, b: B) -> bool { ... }
// ...closure_r, kernel_r, monotone_r, idempotent_r similarly.

// Triple-bound predicates: floor ≤ ceil, round-trips that span both views:
pub fn triple_floor_le_ceil<T, A, B>(_t: &T, a: A) -> bool
where T: Triple<A, B>, A: Copy, B: Copy + PartialOrd {
    T::R.floor(a) <= T::L.ceiling(a)
}

pub fn triple_roundtrip_ceil<T, A, B>(_t: &T, b: B) -> bool
where T: Triple<A, B>, A: Copy, B: Copy + Eq {
    T::L.ceiling(T::L.upper(b)) == b
}

// triple_roundtrip_floor, triple_ulp_bound, triple_round_picks_endpoint,
// triple_truncate_*, triple_midpoint_between, triple_interval_at_midpoint, etc.
```

A test block for a triple exercises both batteries via the marker:

    // No galois_r call possible — TIMENANO has type Conn<_, _, L>, not Conn<_, _, R>.

The Plan-27 audit becomes structural: a one-sided const can't be
fed to the wrong-side predicate because the function signature
rejects it.

### T8 — `compose_l!` / `compose_r!` for one-sided, `compose!` for triples

```rust
// Same-kind one-sided compose. Macro expansion uses non-capturing
// closures referencing the parent paths — preserves fn-pointer
// coercion and `Conn`'s `Copy + const` shape.
compose_l!(C1, C2, ...)  // (Conn<_,_,L>, Conn<_,_,L>, ...) → Conn<_,_,L>
compose_r!(C1, C2, ...)  // (Conn<_,_,R>, Conn<_,_,R>, ...) → Conn<_,_,R>

// Triple compose: declaration-form macro generates a fresh marker
// unit-struct with composed view consts.
compose! {
    F064F016 = F064F032 . F032F016
}
// expands to:
pub struct F064F016;
impl ViewL<F064, F016> for F064F016 {
    const L: Conn<F064, F016, L> = compose_l!(F064F032::L, F032F016::L);
}
impl ViewR<F064, F016> for F064F016 {
    const R: Conn<F064, F016, R> = compose_r!(F064F032::R, F032F016::R);
}
```

Cross-kind / mixed L-R composition is a type error — the `compose_l!` /
`compose_r!` macros only accept paths to same-kind Conns. (Slot-swap
via `swap_l` / `swap_r` is the explicit conversion if needed.)

Open question: the `compose_l!` macro's expansion needs the parent
paths to resolve to module-scope `const`s for the non-capturing-closure
trick to coerce to `fn` pointers. Associated consts on generic types
(`T1::L` where `T1: ViewL<...>`) might not satisfy this — Rust's
const evaluator may or may not accept them as paths inside a closure
body that's coerced to `fn`. **A spike at the start of T8 confirms
this works; if not, the fallback is a `const fn` that takes the two
Conns by value and returns the composed Conn — which loses the macro's
variadic chain ergonomics but keeps composition working.** The spike
should take ≤ 1 hour.

### T9 — Migrate the 28 full-triple consts to marker types

Per `doc/plans/plan-2026-04-28-07.md` precedent, the 28 callsites
across `fixed/`, `float/`, `time/`, `addr/` get rewritten:

```rust
// before:
pub const F064F032: Conn<F064, F032> = Conn::new(ceil, inner, floor);
// after:
pub struct F064F032;
impl ViewL<F064, F032> for F064F032 { const L:  ConnL<F064, F032> = ...; }
impl ViewR<F064, F032> for F064F032 { const R: ConnR<F064, F032> = ...; }
```

The naming convention stays — `F064F032` is still the public name —
but it's now a unit struct rather than a `Conn` const. Downstream users
get `F064F032::L` / `F064F032::R` for projection, `round(&F064F032, x)`
for two-sided ops, `compose_l!(F064F032::L, ...)` for L-side
composition.

The 33 `new_l` consts and 15 `new_r` consts stay as `Conn` value
consts (no marker indirection — they're not triples, so they don't
need both views). Because `K = L` is the default, the 33 `new_l`
consts keep their existing `pub const TIMENANO: Conn<Time, Ext<i64>>`
annotation unchanged. The 15 `new_r` consts need explicit
ascription: `pub const FOO: Conn<A, B, R> = ...` (or the alias
`ConnR<A, B>`).

The 16 `new_iso` consts each get audited:
- If both views are needed downstream → migrate to marker (per T9 above).
- If only one kind is used → keep as a single `Conn<_,_,L>` (or `R`) const.

Audit scope: grep each iso name across the crate for any caller that
exercises the other kind (which won't compile after T1). For each one
that fails, decide: marker promotion or use-site rewrite to the
existing kind.

## Deferred

- **`Reverse<T>` wrapper + `rev_left` / `rev_right`** (Haskell math
  equivalence #2: `Conn 'L a b ≡ Conn 'L (Down b) (Down a)`). Drafted
  in the original Plan 29 (T4) but lifted out: the wrapper itself is
  a meaningful design problem (lifting `PartialEq`/`Hash`/`Display`/
  numeric ops onto `Reverse<T>`, deciding whether shipped Conns get
  reversed endpoints). Follow-up sprint.
- **Composition spanning shape boundaries** (e.g. composing a triple
  marker with a one-sided Conn to produce a fresh one-sided Conn that
  loses triple-ness). Doable via additional macro forms but not in
  initial scope; the same effect is reached via
  `compose_l!(triple::L, one_sided)` today.
- **Mixed-kind compose** (`L ∘ R → ?`). Forbidden by design in this
  plan; a future plan could add it via explicit slot-swap insertion.
- **Phasing out `new_iso` semantics into a dedicated `Iso<A, B>`
  marker trait.** Possible cleanup: an iso that satisfies
  `ViewL<A,B> + ViewR<A,B>` where `L.f ≡ R.g_swap` and
  similarly. Worth doing only if downstream surfaces want
  iso-specific dispatch; not in scope here.

## Alternates considered

The two designs spelled out below were seriously considered and
rejected in favor of the strict 2-fn Conn approach. Documented here
for re-evaluation if the chosen design surfaces costs we missed
during implementation.

### Alternate A — Three-kind `L` / `R` / `T` with `ImpliesL` / `ImpliesR` / `Meet`

Single `Conn<A, B, K>` struct holding three fn-pointer fields
`(ceil, inner, floor)` with `K = T` as the default. Three flat kinds:

```rust
pub struct L; pub struct R; pub struct T;
pub trait Kind: Sealed {}     // sealed to {L, R, T}
pub trait ImpliesL: Kind {}   impl ImpliesL for L {}    impl ImpliesL for T {}
pub trait ImpliesR: Kind {}   impl ImpliesR for R {}    impl ImpliesR for T {}

pub trait Meet<Other: Kind>: Kind { type Output: Kind; }
// 7 impls: T∧T=T, T∧L=L, L∧T=L, L∧L=L, T∧R=R, R∧T=R, R∧R=R; no L∧R.

pub fn ceiling<A, B, K: ImpliesL>(c: &Conn<A,B,K>, a: A) -> B { (c.ceil)(a) }
pub fn floor  <A, B, K: ImpliesR>(c: &Conn<A,B,K>, a: A) -> B { (c.floor)(a) }
pub fn round  <A, B, K: ImpliesL + ImpliesR>(c: &Conn<A,B,K>, x: A) -> B { ... }
//   ↑ in practice this means K = T, since L/R don't satisfy both.

pub fn compose<A, B, C, K1, K2>(
    inner: Conn<A, B, K1>, outer: Conn<B, C, K2>,
) -> Conn<A, C, <K1 as Meet<K2>>::Output>
where K1: Meet<K2>, K2: Kind { ... }
```

**Wins:** Most ergonomic at the call site — `floor(&triple_conn, x)`
works directly because `T: ImpliesR`. No flip ceremony, no view
projection. `Meet` gives Haskell-style automatic kind specialization
on compose. Smallest delta from today's code — the existing 28
`Conn::new` consts compile unchanged via the `K = T` default; the
33 `new_l` / 15 `new_r` consts add a kind annotation
(`ConnL<…>` / `ConnR<…>`).

**Loses:** Keeps a third kind `T` that doesn't exist in Haskell
(Haskell's full triples are kind-polymorphic via `forall k. Cast k a b`
in the `Cast` pattern synonym; `T` is a Rust pragmatism since Rust
can't quantify a const over a kind). The 3-fn struct still holds an
"adjoint that's the same as another adjoint" for one-sided Conns —
a representational artifact rather than structural truth. The Plan
27-style audit benefit is real but trait-bounded rather than
structural.

**Why rejected:** the strict 2-fn approach tells the math truth and
keeps the kind universe minimal. Ergonomic wins are real but smaller
than the structural-faithfulness wins of the chosen design once
weighed against the pre-1.0 free hand on API shape.

### Alternate B — Four-kind `L<S>` / `R<S>` (parameterized shape)

Single `Conn<A, B, K>` struct with three fn-pointer fields. Kinds
parameterized by shape (`One` for one-sided, `Tri` for triple-shape):

```rust
pub struct One; pub struct Tri;
pub struct L<S = One>(PhantomData<S>);
pub struct R<S = One>(PhantomData<S>);

pub trait IsL {}                 impl<S> IsL for L<S> {}
pub trait IsR {}                 impl<S> IsR for R<S> {}
pub trait IsTriple {}            impl IsTriple for L<Tri> {} impl IsTriple for R<Tri> {}
pub trait Flip: Kind { type To: Kind; }
impl Flip for L<Tri> { type To = R<Tri>; } impl Flip for R<Tri> { type To = L<Tri>; }

// API:
pub fn ceiling<A, B, K: IsL>(c: &Conn<A,B,K>, a: A) -> B { ... }
pub fn floor  <A, B, K: IsR>(c: &Conn<A,B,K>, a: A) -> B { ... }
pub fn round  <A, B, K: IsTriple>(c: &Conn<A,B,K>, x: A) -> B { ... }
impl<A, B, K: Flip> Conn<A, B, K> { pub const fn flip(self) -> Conn<A,B,K::To> { ... } }
```

**Wins:** Single struct type, four kinds capture the shape lattice
faithfully (one-sided vs triple), composition kind algebra falls out
mechanically (`L<One> ∘ L<One> = L<One>`, `L<Tri> ∘ L<One> = L<One>`,
etc.). Triples can flip between L and R configurations explicitly.
Closed kind universe.

**Loses:** Forces explicit `.flip_r()` ceremony at every R-side op on a
triple currently in L config, even though the operation is zero-cost.
Use sites read `floor(&FOO.flip_r(), x)` instead of `floor(&FOO, x)`.
Friction scales with usage — every downstream caller pays it. The
distinction between `L<One>` and `L<Tri>` after composition is also a
phantom: once the third fn is erased by composition, you can't recover
it, so tracking it separately doesn't buy the user-facing behavior
anything.

**Why rejected:** the flip ceremony pays for type discipline that the
strict 2-fn approach gets for free (one-sided Conns aren't a `Triple`,
so two-sided ops don't compile on them — same outcome, no syntactic
tax).
