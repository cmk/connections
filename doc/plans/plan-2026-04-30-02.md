## Plan 31 — Codegen-macro consolidation

## Goal

Reduce ~1000 lines of duplicated codegen across the crate by lifting
four recurring patterns into a single shared form. Concretely:

1. Add `iso!` declaration macro for the 14 degenerate-Galois bijection
   sites; migrate every site.
2. Add `prop::conn::law_battery!` macro for the 9-property test
   battery; migrate the ~10 hand-rolled / per-host-type copies.
3. Migrate the remaining manual `pub struct NAME; impl NAME { _ceil,
   _inner, _floor } impl ViewL impl ViewR` sites to use the existing
   `triple!` macro (single-instance use today: `char.rs::U032CHAR`).
4. Lift the seven internal saturating-cast macro families from
   `pub(crate)` to a documented public surface (`crate::macros::*`),
   gated on a new `macros` cargo feature so the default surface stays
   narrow.

Net: removes >800 LoC of duplicated boilerplate, no behavioral change,
all existing law batteries still pass.

## Context

Plan 30 (MR !47) just landed `src/char.rs` and feature-gated
`src/unix.rs`. Both expose the iso pattern in their final form: a
`pub struct NAME` + ~25 lines of mechanical `impl NAME` / `impl
ViewL` / `impl ViewR` boilerplate. That re-surfaced the question of
whether the shape deserves a macro. A walk through the rest of the
crate found four reusable patterns; this sprint folds them.

The `triple!` declaration macro from Plan 29B already covers shape
(3) when the helpers are written as free `fn`s — `char.rs::U032CHAR`
is the working example. Every other site in the crate manually
inlines the pattern, partly out of convention (helpers namespaced
under the marker) and partly because `triple!` post-dates the older
sites. This sprint normalises on `triple!`.

## Tasks

### T1 — `iso!` declaration macro

Add to `src/conn.rs` next to `triple!`:

```rust
/// Ship a degenerate-Galois iso (lossless bijection) from a single
/// `(forward, back)` function pair. Equivalent to `triple!` with
/// `ceil = floor = forward`, but reads as the iso it is.
#[macro_export]
macro_rules! iso {
    (
        $(#[$meta:meta])*
        $vis:vis $name:ident : $A:ty <=> $B:ty {
            forward: $fwd:expr,
            back:    $bk:expr $(,)?
        }
    ) => {
        $crate::triple! {
            $(#[$meta])*
            $vis $name : $A => $B {
                ceil:  $fwd,
                inner: $bk,
                floor: $fwd,
            }
        }
    };
}
```

The arrow `<=>` (vs `=>` in `triple!`) is a syntactic cue that this
is a bijection, not an asymmetric adjoint.

**Migration sites** (14):

|------|---------|----------------------|
| `src/unix.rs` | `UPIDI032`, `UUIDU032`, `UGIDU032` | ~60 |
| `src/addr/ip.rs` | `U032IPV4`, `U128IPV6` | ~40 |
| `src/fixed/i8.rs` | `Q000I008` | ~20 |
| `src/fixed/i16.rs` | `Q000I016` | ~20 |
| `src/fixed/i32.rs` | `Q000I032` | ~20 |
| `src/fixed/i64.rs` | `Q000I064` | ~20 |
| `src/fixed/i128.rs` | `Q000I128` | ~20 |

Each migration: replace ~25-line `pub struct NAME; impl NAME {…}
impl ViewL …; impl ViewR …;` with an `iso!` block referencing free

`$expr` arms — but free `fn`s keep the const-fn property).

Total saved: ~300 lines.

### T2 — `prop::conn::law_battery!` shared property macro

Add to `src/prop/conn.rs`:

```rust
/// Generate the standard 9-property law battery (`galois_l/r`,
/// `closure_l/r`, `kernel_l/r`, `monotone_l/r`, `idempotent`) inside
/// a fresh `mod $mod_name` block.
///
/// Strategies are passed as `proptest::strategy::Strategy` expressions
/// (typically calls into `crate::prop::arb::*` or local `arb_*` fns).
/// The `Conn` is named via its triple-marker type; `ViewL::L` and
/// `ViewR::R` are pulled by trait dispatch so the macro doesn't care
/// whether the marker is one-sided or full-triple — non-existent
/// arms simply won't compile, and you `# [cfg(any())]`-out the bad
/// arm at the call site (rare; one-sided sites typically only assert
/// the surviving side).
#[macro_export]
macro_rules! law_battery {
    (
        mod $mod_name:ident,
        conn: $conn:ty,
        fine:   $fine_strat:expr,
        coarse: $coarse_strat:expr $(,)?
    ) => { /* … emit the 9 proptests against $conn::L and $conn::R … */ };
}
```

Key API decisions:

- **Strategy expressions, not types.** Each call passes
  `arb_extended_u32()` etc. — same shape as the current per-site
  proptests.
- **Marker-typed, not const-typed.** The macro takes
  `$conn: ty` and dispatches via `<$conn as ViewL<_, _>>::L`. This
  matches what `tests/fixed_u8_galois.rs` already does (the
  fully-qualified form is necessary in integration-test crates
  because the inherent `.ceil()` etc. methods aren't in scope
  without the trait import).
- **Subset selection.** Some Conns are one-sided (e.g.
  `U008N008` has no R-Galois). Provide a `subset:` flag:

  ```rust
  law_battery! {
      mod u008n008,
      conn: U008N008,
      fine:   any::<u8>(),
      coarse: arb_nz_u8(),
      subset: l_only,    // emits galois_l/closure_l/kernel_l/monotone_l/idempotent only
  }
  ```

  Variants: `full` (default, 9 tests), `l_only` (5), `r_only` (5),
  `iso_only` (round-trip only, for `Uid`/`Gid`-style non-`Ord` cases).

**Migration sites:**

| File | Current shape | Notes |
|------|---------------|-------|
| `src/fixed/i{8,16,32,64,128}.rs` | local `props_for_pair!` macro | 5 copies × ~80 LoC |
| `tests/fixed_u{8,16,32,64,128}_galois.rs` | local `props_for_pair!` macro | 5 copies × ~80 LoC |
| `src/fixed/i64.rs`, `src/fixed/i128.rs` | wrap with `proptest_config!(cases: 64)` | macro must accept an optional config arm |
| `src/unix.rs::tests` | hand-rolled `upidi032_*` x10 | one full battery + two iso_only |
| `src/time/duration.rs::tests` | hand-rolled `DURNSECS` x9 | one full battery |
| `src/time/clock.rs::tests` | hand-rolled `TIMENANO`/`TIMESECS` | two l_only batteries |
| `src/addr/ip.rs::tests` | hand-rolled `IPV6IPV4`/`U032IPV4`/`U128IPV6`/etc. | mixed full + iso_only |
| `src/char.rs::tests` | hand-rolled `U032CHAR` x9 | one full battery |
| `src/float/f32.rs::tests` | hand-rolled `F064F032` x9 | one full battery (cases: 256) |

Total saved: ~500 lines.

**Out of scope:** the per-site **spot tests** stay where they are.
Only the proptest battery is consolidated.

### T3 — Migrate remaining manual triple sites to `triple!`

Sites (currently inlining `pub struct + impl + ViewL + ViewR`):

- `src/time/duration.rs` — `DURNSECS`, `F064DURN`, `F032DURN`,
  `STDRU064`, `STDRU128`, `F064STDR`, `F032STDR` (7)
- `src/float/f32.rs` — `F064F032` (1)
- `src/addr/ip.rs` — `IPV6IPV4` (1)
- `src/float/f16.rs` — `F032F016`, `F064F016` (2, gated)

For each: lift the inherent `_ceil` / `_inner` / `_floor` methods to
free `fn`s in the same module, then collapse the marker + two
impls into a `triple! { pub NAME : A => B { ceil: …, inner: …,
floor: … } }` block.

**Decision point during T3:** if hoisting the helpers loses meaningful
namespacing (the helpers all share short names like `ceil`/`floor`
that would clash inside one module), extend `triple!` to accept an
inline-method form:

```rust
triple! { pub DURNSECS : Duration => Extended<i64> {
    fn ceil(d: Duration) -> Extended<i64> { … }
    fn inner(b: Extended<i64>) -> Duration { … }
    fn floor(d: Duration) -> Extended<i64> { … }
}}
```

The macro emits the unit-struct + `impl NAME { const fn _ceil … }`
block + the two trait impls. Downside: macro grows; benefit: no
namespace collisions, helpers stay co-located with their marker.

Recommendation: try the free-fn form first per site; if any module
ends up with three or more colliding helper names, add the inline
form as a second arm of `triple!`. Document the choice in the
sprint Review.

Total saved: ~250 lines (11 sites × ~20 lines each).

### T4 — Public macro surface (`uint_uint!` family)

Today, seven macros in `src/fixed.rs` are `pub(crate)` because their
generated Conns weren't intended as a downstream API surface. They
are, however, a useful primitive set for any downstream crate
defining its own bounded-integer lattice.

Lift them under a new feature gate and module:

```rust
// src/lib.rs
#[cfg(feature = "macros")]
pub mod macros {
    //! Re-exported codegen macros for downstream lattices.
    //!
    //! Stable subject to the same SemVer policy as the rest of the
    //! crate's public API. Naming and signatures match the
    //! `pub(crate)` forms used internally; future expansion of the
    //! crate's own macro toolkit will land here.

    pub use crate::fixed::{
        ext_int, int_int_narrow, int_uint, int_uint_narrow,
        nz_int_ext, nz_uint_ext, uint_int_sat, uint_uint,
        uint_uint_narrow,
    };
    pub use crate::{compose, compose_l, compose_r, iso, triple};
    // law_battery! re-exported under `prop::conn` already; not here.
}
```

Cargo.toml:

```toml
[features]
default = []
macros = []
```

This is opt-in — the default feature set is unchanged. Documented
in `README.md` as "advanced: building your own lattices on top of
this crate."

**Module-level docs** for `macros`: list each macro, its
constraints (which the `pub(crate)` doc comments already describe
well — they read as user-facing prose), and link to the worked
examples in `src/fixed/{i,u}*.rs`.

Total added: ~30 lines (the re-export module + Cargo feature).

## Deferred

- **Public, parameterized `prop::conn::iso_battery!`** that
  auto-emits the four iso-specific spot checks (forward/back round
  trips). Currently inline in T2; promote to a standalone macro if
  T1 finds three or more sites that want only the iso laws and not
  the full battery. Decision deferred to post-merge; not load-bearing
  for the sprint.
- **`triple!` inline-method form** — only added in T3 if naming
  collisions force it; otherwise punt.
- **Macro-level integration tests for the seven `pub(crate)`
  saturating-cast macros once they go public.** A round-trip
  integration test against a downstream `u24`/`i24` newtype would
  validate the `pub` signatures hold up; deferring because we'd
  have to ship the newtype as test fixture and that's its own
  small design exercise.
- **Doc-test deduplication.** Most `iso!`-migrated markers carry
  one or two doc-tests of the form `assert_eq!(NAME.ceil(x), y)`.
  These stay where they are — they're cheap, user-facing, and
  unrelated to the boilerplate-collapse goal.
