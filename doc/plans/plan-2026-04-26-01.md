# Plan 04 — `compose!` macro

## Goal

Add a variadic `compose!` macro that expands a chain of compile-time-known
`Conn` consts into a single fresh `Conn<Src, Dst>`. No runtime dispatch,
no closures-in-fields, no shape change to `Conn`.

## Context

The crate has no first-class composition primitive. Composition appears
only as hand-nested calls (e.g. `composition_pico_to_uni_matches_ladder_climb`
in `src/conn/fixed.rs:548`), and a runtime `Conn::then` was deferred in
plan-2026-04-14-01 §Deferred and again in plan-2026-04-24-01 §Deferred
because `Conn` stores bare `fn` pointers, which cannot capture the two
input `Conn`s a runtime composition would need.

In practice, the parents you want to compose are always module-scope
`const`s (the `F??F??` ladder, the `S???S???` rate pairs, the `F12S??`
cross-tier conns). For that case, a code-gen macro is strictly cheaper
than runtime composition: it emits nested calls directly, the closures
capture nothing (they only reference module-scope `const`s), and the
resulting `Conn` is `Copy` and `const`-constructible — same shape as a
hand-written one.

The runtime `Conn::then` stays deferred. This sprint also adds a
section to `doc/design.md` (replacing the current pseudo-implementation
at lines 91–112, which doesn't compile against the `fn`-pointer struct)
that captures the storage-shape rationale and the path forward if a
future consumer needs runtime composition.

### T2 — `compose!` macro in `src/conn.rs`

Add the `macro_rules! compose` block at module scope, `#[macro_export]`-ed.
Variadic over `N ≥ 2` parents; the macro takes only parents, and the
caller supplies the type via the binding annotation.

Public arm:

```rust
#[macro_export]
macro_rules! compose {
    ($first:path $(, $rest:path)+ $(,)?) => {
        $crate::conn::Conn::new(
            |a| $crate::compose!(@nest_ceil  a; $first $(, $rest)+),
            |z| $crate::compose!(@nest_inner z; $first $(, $rest)+),
            |a| $crate::compose!(@nest_floor a; $first $(, $rest)+),
        )
    };
    // @nest_ceil / @nest_floor — covariant (left-to-right wrap)
    (@nest_ceil $x:expr; $last:path) => { $last.ceil($x) };
    (@nest_ceil $x:expr; $first:path $(, $rest:path)+) => {
        $crate::compose!(@nest_ceil $first.ceil($x); $($rest),+)
    };
    (@nest_floor $x:expr; $last:path) => { $last.floor($x) };
    (@nest_floor $x:expr; $first:path $(, $rest:path)+) => {
        $crate::compose!(@nest_floor $first.floor($x); $($rest),+)
    };
    // @nest_inner — contravariant (right-to-left wrap)
    (@nest_inner $z:expr; $last:path) => { $last.inner($z) };
    (@nest_inner $z:expr; $first:path $(, $rest:path)+) => {
        $first.inner($crate::compose!(@nest_inner $z; $($rest),+))
    };
}
```

Re-export at the crate root via the `#[macro_export]` attribute (no
explicit `pub use` needed; `#[macro_export]` attaches the macro to the
crate root automatically).

**Open question — const context.** Non-capturing closures coerce to
`fn(A) -> B` at non-const argument sites. They *should* also coerce in
const context (the coercion is a no-op at the IR level). The build
gate: `const F: Conn<Pico, Uni> = compose!(F12F09, F09F06, F06F03, F03F00);`
must compile. If it doesn't, fall back to a sectioned-syntax variant
that uses named `fn` items (see Deferred §). Decide at impl time.

### T4 — `doc/design.md` update

Replace the broken pseudo-implementation at lines 91–112 with the
storage-shape ELI5 + macro reference. New section structure:

- **Composition: code-gen vs. runtime.** Lead paragraph naming
  `compose!` as the supported primitive.
- **Why not a runtime `Conn::then`.** Explain: `Conn`'s fields are
  `fn(A) -> B`, which is just a code address and has no slot to stash
  captures. A runtime `then` would need closures, which require either
  generic type parameters on `Conn` (with `fn`-pointer defaults) or a
  separate `Box<dyn Fn>` variant. Both are real complexity, deferred
  until a consumer needs it.
- **`compose!` example.** A two-line snippet showing
  `const F12F00_BIS: Conn<Pico, Uni> = compose!(F12F09, F09F06, F06F03, F03F00);`.
- **Combinator table update at line 120.** Replace the `>>>` / `.then()`
  row with a `compose!(F, G, …)` row marked **implemented**. Leave
  `.then()` listed as **deferred** for future reference. `product` /
  `coproduct` rows untouched.

## Deferred

- **Runtime `Conn::then`.** Recorded in the new `doc/design.md`
  section. Re-open when a concrete consumer needs runtime composition.
  The macro covers all known compile-time use cases (ladder, rate, pico).
- **`product` / `coproduct`.** Same rationale as `then`.
- **Sectioned-syntax fallback.** If non-capturing closures fail to
  coerce in const context (T2 build gate), fall back to:
  `compose!(F12F00_BIS : Pico => Uni ; F12F09, F09F06, F06F03, F03F00);`
  using named `fn` items inside a `const { ... }` block. Same
  semantics, more verbose call site. Decide at impl time based on
  what the compiler accepts.
