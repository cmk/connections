# Plan 35 — `conn_l!` / `conn_r!` macros for one-sided Conns

## Context

The crate ships three declaration-form macros for building Conn
artifacts: `triple!`, `iso!`, and `compose!` (`compose_l!` /
`compose_r!`). They cover the *full-triple*, *iso*, and *composition*
shapes. They do **not** cover the most common shape we ship — a bare
one-sided `pub const` of type `ConnL<A, B>` or `ConnR<A, B>` built
through `Conn::new_l` / `Conn::new_r`.

Today these constants are written by hand:

```rust
pub const TIMENANO: ConnL<Extended<Time>, i64> = {
    fn ceil(t: Extended<Time>) -> i64  { … }
    fn inner(ns: i64) -> Extended<Time> { … }
    Conn::new_l(ceil, inner)
};
```

Two reasons to add `conn_l!` / `conn_r!`:

1. **Footgun.** `new_l` takes `(ceil, inner)`; `new_r` takes
   `(inner, floor)` to mirror Haskell's `CastR` (called out at
   `src/conn.rs:200-205`). Positional args mean a `new_l → new_r` flip
   silently swaps the slots. Named-field syntax (`ceil:` / `inner:` /
   `floor:`) makes that class of bug impossible.
2. **Symmetry.** Five macros (`conn_l!`, `conn_r!`, `iso!`, `triple!`,
   `compose!`) cover the five Conn shapes — readers learn one
   syntactic pattern. `iso!` is already a thin shim over `triple!`;
   `conn_l!` / `conn_r!` bottom out at `Conn::new_l` / `Conn::new_r`
   the same way.

Boilerplate savings vs raw `Conn::new_l(ceil, inner)` is modest
(~1 line per site). The primary motivation is the footgun fix, not
boilerplate reduction.

## Goal

Ship `conn_l!` and `conn_r!` declaration-form macros in `src/conn.rs`,
convert the ~10 hand-written one-sided `pub const` sites in `time/`
and `lattice.rs`, and update the README macro reference.

## Tasks

### T1 — Add `conn_l!` and `conn_r!` to `src/conn.rs`

**File:** `src/conn.rs`, alongside `triple!` / `iso!` (around line 643).

Both `#[macro_export]`. Both emit a `pub const`, not a marker struct
(this is the one distinction from `triple!` and the reason a separate
macro is needed).

Syntax (mirrors `triple!`):

```rust
conn_l! {
    pub TIMENANO : Extended<Time> => i64 {
        ceil:  time_to_ns,
        inner: ns_to_time,
    }
}
// expands to:
pub const TIMENANO: $crate::conn::ConnL<Extended<Time>, i64> =
    $crate::conn::Conn::new_l(time_to_ns, ns_to_time);
```

```rust
conn_r! {
    pub UFOO : u8 => i8 {
        inner: u_to_i,
        floor: i_to_u,
    }
}
// expands to:
pub const UFOO: $crate::conn::ConnR<u8, i8> =
    $crate::conn::Conn::new_r(u_to_i, i_to_u);
```

Notes:

- Field-name order in source mirrors the Galois adjunction order
  (`ceil ⊣ inner` for L, `inner ⊣ floor` for R). The macro reorders
  positionally for the constructor; users never see the swap.
- `$(#[$meta:meta])*` and `$vis:vis` accepted, same as `triple!`.
- `$ceil:expr`, `$inner:expr`, `$floor:expr` accept either a `fn` path
  or an inline expression that coerces to `fn`.
- Trailing comma optional, matching `triple!`.

Doctest at each macro definition. Mirror the `triple!` / `iso!`
doctest style verbatim (the existing ones are `ignore` blocks at
`src/conn.rs:613-620` and `src/conn.rs:654-660`); upgrade to
executable doctests if cheap, otherwise keep `ignore` for parity.

### T2 — Macro-expansion equality unit tests

**File:** `src/conn.rs` `mod tests`.

Two unit tests confirming the macros expand to the same `(f, g)` slot
contents as a hand-written `Conn::new_l` / `Conn::new_r` call over the
same fns. Structural fact, not a numeric invariant — unit tests, not
proptest.

```rust
#[test]
fn conn_l_macro_expands_to_new_l() { … }

#[test]
fn conn_r_macro_expands_to_new_r() { … }
```

The tests construct a hand-written const + a macro-built const over
the same identity-shaped functions, then assert agreement on both the
ceil/floor slot and the inner slot for representative inputs.

Per-family generator macros (`uint_uint!`, `ext_int!`,
`int_int_narrow!`, `uint_int_sat!`, `nz_*_ext!`, …) **stay as-is**
— they're already factored value-form macros for their patterns;
rewriting them on top of `conn_l!` / `conn_r!` would add macro
indirection without changing emitted code.

Each converted site keeps its previous public type signature so
downstream `use` sites are unchanged.

  macro *names* match the kind.

- T1 / T2 / T3 / T4 all delivered. Macros sit alongside `triple!` /
  `iso!` in `src/conn.rs`, with `#[macro_export]` and `ignore`-style

  Module-scope `const`s pulled out alongside (`NS_PER_HOUR`, …,
  `MIN_JD`, `MAX_JD`).
