# Plan 08 — Port the rest of `Data.Connection.Cast` to the Rust port

(Sprint A — `Cast` accessors and lifters.)


## Context

This crate has shipped the foundation (`Conn<A, B>` triple, `compose!`
macro, `identity`, `Ple`, the `Extended` and `ExtendedFloat` enums,
lattice traits) plus 100+ concrete connections (decimal & binary
fixed-point ladders, signed/unsigned integer ladders, sample-rate
pairs, `f64↔f32`). What is missing from the upstream Haskell
`Data.Connection.Cast` module is the **operations on connections**
layer: the L/R-side accessors and
lifters, the two-sided helpers (`round`, `truncate`, `midpoint`, `interval`,
`median`), the lifting combinators (`bounded`, `ordered`, `select`, `divide`),
and the `Down`/filter helpers. The user explicitly called out the two-sided
helpers as priority.

## The L vs R bias question (answer up front)

**Don't add a `CastR` constructor. Don't add a `Side` phantom marker.** Two
reasons:

1. **`design.md` §86–134 already settled this**: the Rust `Conn<A, B>` carries
   the full triple (`ceil ⊣ inner ⊣ floor`) in three function-pointer fields,
   so it subsumes both Haskell `CastL` (length-2, `floor = ceil`) and `CastR`
   (length-2 in the opposite direction, also `floor = ceil`) and the
   polymorphic `Cast k` (length-3) without needing a type-level tag. Crucially,
   the two-sided helpers (`round`/`truncate`/`midpoint`/`interval`/`median`)
   that Haskell expresses as `forall k. Cast k a b` need no rank-2 polymorphism
   in Rust — every `Conn` already exposes both `ceil` and `floor`.

2. **Haskell instance asymmetry is overwhelming**: 122 `Cast 'L`-only concrete
   connections, **0** `Cast 'R`-only across `Class.hs`, `Int.hs`, `Word.hs`,
   `Float.hs`, `Fixed.hs`, `Time.hs`, `Ratio.hs`. `'R`-tagged values in Haskell
   only appear as the right side of `swapL`/`swapR` plumbing or via the
   polymorphic-`k` path. There is nothing to port that needs `CastR` as a
   distinct shape.

What we *do* need is to expose **both naming conventions** as free functions
on the unified `Conn`: `ceiling`/`upper`/`maximize` (L-side names) **and**
`floor`/`lower`/`minimize` (R-side names) all act on the same struct. They
read the same `inner` field but under different documentation contracts (L
treats `inner` as the upper adjoint of the lower-half pair; R treats it as the
lower adjoint of the upper-half pair).

### T2 — Add `src/conn/cast.rs` with the 14 free functions

All as `pub fn` taking `&Conn<A, B>` first. Bodies are one-liners delegating
to the existing `c.ceil` / `c.inner` / `c.floor` field methods on
`src/conn/mod.rs`. Closures are taken by value as `impl Fn(...) -> ...` —
no `fn`-pointer coercion needed because the lifters consume their `f`
argument inside the body and never store it.

L-side (Cast.hs:217-312):

Note: `upper`/`lower` both return `c.inner(b)` — the same field. They differ
in *documented contract*: `upper` is the upper adjoint of `(ceil, inner)`;
`lower` is the lower adjoint of `(inner, floor)`. Each one's rustdoc cites
the relevant Galois law.

`swapL`/`swapR` are deliberately omitted — they're identity in our
representation. The module-level rustdoc explains why (cross-link to
`design.md` §"Core type").

### T3 — Add `cast_*` law predicates to `src/property/laws.rs`

Mirror the existing `conn_*` predicate naming.
New predicates:

| Predicate | Asserts |
|-----------|---------|
| `cast_upper_unit_law(c, a)` | `a.ple(&upper1(c, \|x\| x, a))` (closure / unit law) |
| `cast_lower_counit_law(c, a)` | `lower1(c, \|x\| x, a).ple(&a)` (kernel / counit law) |
| `cast_ceiling_unit_law(c, b)` | `ceiling1(c, \|x\| x, b).ple_then_ple(...)` — see Cast.hs:298 (the comonad counit law) |
| `cast_floor_counit_law(c, b)` | mirror |
| `cast_maximize_eq_ceiling(c, a, b)` | `maximize(c, a, b) == ceiling(c, (a, b))` |
| `cast_minimize_eq_floor(c, a, b)` | `minimize(c, a, b) == floor(c, (a, b))` |
| `cast_upper2_diag(c, a)` | `upper2(c, \|p, q\| p, a, a) == upper1(c, \|x\| x, a)` |

Predicates take `&Conn<A, B>` plus generators of `A` / `B` and return `bool`
(no `prop_assert!`), matching the existing `laws::conn_*` shape so the
test-side macros stay simple.

### T7 — Substantially flesh out `README.md`

The current `README.md` is a 17-line stub that reads like a port of the

1. **Tagline + one-paragraph pitch.** Galois connections as Rust values:
   safe, lawful casting between numeric types, rounding helpers that
   never silently lose precision, and a composable lattice algebra. Keep
   the math link to nLab / Wikipedia for readers who want it; do not lead
   with the math.

2. **Why this exists in Rust.** Three concrete Rust-native motivations
   that don't depend on knowing the Haskell library:
   - The `as` operator and `From`/`Into` are silent on rounding — `as

     or "nearest". This crate gives you `c.ceil(x)`, `c.floor(x)`, and

   - Composing two casts with `as` does not preserve the round-trip

## Subsequent sprints (outline only)

### Sprint B — Two-sided helpers

Adds `interval`, `midpoint`, `round`/`round1`/`round2`, `truncate`/
`truncate1`/`truncate2`, `median` to `src/conn/cast.rs`. Introduces one new
trait, `Pcompare: Ple` in `src/lattice.rs`, with `pcompare(&self, &Self) ->
Option<Ordering>`. Blanket impls for the totally-ordered primitives (returns
`Some(self.cmp(other))`); float impls return `Some(Equal)` for NaN-vs-NaN
and `None` for one-sided-NaN, otherwise delegate to `partial_cmp`. Same
shape as the existing N5 `Ple` impls at `lattice.rs:350-422`.

Arithmetic bounds for `interval`/`midpoint`/`round`/`truncate` (`Copy + Add
+ Sub + Div + Pcompare + From<u8>`) live inline at each helper rather than
behind a sealed `Numeric` trait — the trait would only be a tidiness
wrapper, and inline bounds keep the abstraction count low. `From<u8>`
covers the literals `0` and `2` and is implemented by every primitive
numeric type used in the lib.

`round` mirrors Haskell verbatim (Cast.hs:453-458): `Some(Greater) → ceil`,
`Some(Less) → floor`, otherwise (`Some(Equal)` or `None` for NaN-bearing
operands) → `truncate`. `truncate` rounds toward zero. Body of `midpoint` uses
the same precision-preserving `lower1(id, x) / 2 + upper1(id, x) / 2` shape as
Haskell (Cast.hs:441-443) — divide before adding, to avoid overflow on
integer connections.

### Sprint C — Combinators (`bounded`, `ordered`)

Hits the closure-vs-fn-pointer issue (`design.md` §156–190). Solution: ship
`bounded!(T)` and `ordered!(T)` as macros that expand to a fresh
module-scope-suitable `Conn<(), T>` / `Conn<(T, T), T>` const, the same way
`compose!` produces non-capturing closures. Concrete consts:
`BOUNDED_I32`, `BOUNDED_I64`, `BOUNDED_F64`, `ORDERED_I32`, `ORDERED_F64`,
etc., per the small set of types the lib already exercises. Defines (or
adopts) a `Bounded` trait. `select` and `divide` deferred — they construct
`Conn` from two existing `Conn` values, which needs the same struct-shape
generalization that `Conn::then` needs, and that's `design.md`-flagged as
a separate sprint.

### Sprint D — `Down<T>` newtype + `filterL`/`filterR`

Adds `pub struct Down<T>(pub T)` with reversed `PartialOrd`/`Ple` impls
(non-conflicting with `T`'s own; same pattern as `std::cmp::Reverse`). Adds
`filterL(&c, a, b) -> bool` returning `c.ceil(a).ple(&b)` (Cast.hs:594) and
`filterR(&c, a, b) -> bool` returning `b.ple(&c.floor(a))` (Cast.hs:611).
Defers `up_l`/`up_r`/`down_l`/`down_r` to the same later sprint that
generalizes `Conn`'s field types — they construct a new `Conn` from an
existing one and capture it, hitting the closure-vs-fn-pointer issue.

## Deferred (across the whole port)

- **`mapped`** — needs HKT (Haskell `Functor f => Cast k a b -> Cast k (f a) (f b)`). Per `design.md` line 206: omit; specialize per container if needed downstream.
- **`strong` / `choice`** — same closure-capture issue as `Conn::then`. Per `design.md` line 200–202: deferred until `Conn` field types are generalized. Sprint C does NOT take them.
- **`select` / `divide`** — same as above. Deferred from Sprint C.
- **`up_l` / `up_r` / `down_l` / `down_r`** — same as above. Deferred from Sprint D.
- **`extend` / `extended`** — `Cast.hs:619-633` are tiny helpers over `Extended<T>`. The Rust `Extended` enum at `src/extended.rs` already provides constructor-level access. Only port if a downstream needs it.
- **`Preorder` trait full surface** — `design.md` §268–274 deliberately rejects
  this in favor of `PartialOrd` + the localized `Ple`. Sprint B adds the
  minimum extension (`Pcompare`) on top.
