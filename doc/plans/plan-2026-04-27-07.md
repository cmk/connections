# Plan 19 — `conn::cast` cleanup: drop `maximize`/`minimize`, π-driven doctests, edge-case prose

## Goal

Three small cleanups on `src/conn/cast.rs` and its dependents:

1. Remove `maximize` / `minimize` (unused outside the module — only
   `median` calls them, and it can call `c.ceil` / `c.floor` directly).
2. Replace every trivial `Conn::identity::<i32>()` doctest with one
   that exercises [`F064F032`](../../src/conn/float/f64.rs) on
   `Extend(std::f64::consts::PI)`, so readers see what a real
   nontrivial connection looks like.
3. State once at the module level what the chain combinators
   (`interval` / `midpoint` / `round` / `truncate`) do on a length-2
   connection (where `ceil == floor`) and on NaN-bearing inputs;
   trim the per-function NaN paragraphs to point back to it. In
   particular, delete the stray "Callers that need different NaN
   handling should branch upstream" tail at `cast.rs:397`.

## Dependency Graph

```
T1 (remove maximize/minimize)
T3 (Add/Sub/Div/From<u8> for ExtendedFloat<T>)  ──► T2 (π doctests)
                                                       │
                                                       ▼
```

T1 and T3 are independent. T2 depends on T3 (the arithmetic

## Tasks

### T1 — Remove `maximize` / `minimize`

**Motivation.** These are curried forms of `ceiling` / `floor` over
a `Conn<(A, B), C>`. They were ported faithfully from Haskell
`Data.Connection.Cast`, but the Haskell motivation was the phantom
`Side` data kind: `maximize` lived on `Cast 'L`, `minimize` on
`Cast 'R`. Our `Conn` carries the full triple, so both naming
conventions are simultaneously valid via `c.ceil((a, b))` /
`c.floor((a, b))` — the curried wrappers add no expressivity.
Their only in-tree caller is `median`.

**Edits.**

- `src/conn/cast.rs`:
  - Delete `pub fn maximize` and `pub fn minimize`.
  - Update `median` to call `c.ceil((p, q))` / `c.floor((p, q))`
    directly. Update its rustdoc to drop the `maximize`/`minimize`
    synonyms — describe the join/meet using `ceil`/`floor`.
  - Drop the `maximize`/`minimize` mentions in the module-level
    doc.
  - Delete `maximize_eq_ceiling_pair`, `minimize_eq_floor_pair`
    (deterministic) and `maximize_eq_ceiling_random` /
    `minimize_eq_floor_random` (proptest). Keep `ORDERED_PAIR`
    (still used by `median`'s tests).
- `src/lib.rs`: drop both names from the `pub use` list and from
  the `## Cast operations` table.
- `src/property/laws.rs`: delete `cast_maximize_eq_ceiling` and
  `cast_minimize_eq_floor`.
- `doc/design.md`: drop the two API-mirror table rows.

### T3 — `Add` / `Sub` / `Div` / `From<u8>` for `ExtendedFloat<T>`

**Motivation.** `interval` / `midpoint` / `round` / `truncate` need
arithmetic on the source `A`. `ExtendedFloat<f64>` doesn't impl
`Add` / `Sub` / `Div` / `From<u8>` today, so T2's π-doctests for
those helpers can't compile. The minimum viable surface is the four
operators those four helpers mention; nothing else.

**Edits in `src/conn/float.rs`** (extend the existing
`impl_float_ext!` macro or add a parallel `impl_float_arith!` macro,
whichever reads cleaner — decide at edit time):

| Op | `Bot`, `Bot` | `Top`, `Top` | `Bot` ↔ `Top` | `Bot`/`Top`, `Extend(_)` | `Extend(a)`, `Extend(b)` |
|----|-------------|-------------|---------------|---------------------------|---------------------------|
| `Add` | `Bot` | `Top` | `Extend(NaN)` | absorbing endpoint | `Extend(a + b)` |
| `Sub` | `Extend(NaN)` | `Extend(NaN)` | endpoint of LHS | endpoint of LHS for Bot/Top operand; sign-flipped Extend otherwise | `Extend(a - b)` |
| `Div` | `Extend(NaN)` | `Extend(NaN)` | endpoint by sign | endpoint of LHS / `Extend(0)` for Bot/Top divisor | `Extend(a / b)` |
| `From<u8>(n)` | — | — | — | — | `Extend(T::from(n))` (well-typed for f32/f64; for f16/bf16, use `from_f64(n.into())`) |

**Caveat block (new doc above the impls):**

> The numeric impls below are the minimal surface needed to thread
> `ExtendedFloat<T>` through the chain combinators in
> `crate::conn::cast` (`interval`, `midpoint`, `round`,
> `truncate`). They are **not** intended as a general-purpose
> numeric algebra. `Mul`, `Neg`, `Rem`, `PartialOrd` arithmetic on
> Bot/Top, etc. are deliberately omitted — add them only when a
> downstream caller documents the need.

**Property tests (`#[cfg(test)] mod tests`):**

| Property | Invariant |
|----------|-----------|
| `add_extends_passes_through(a, b)` | `Extend(a) + Extend(b) == Extend(a + b)` for non-NaN finite operands |
| `sub_zero_is_id(a)` | `Extend(a) - Extend(0u8 ⇒ Extend(0)) == Extend(a)` |
| `from_u8_round_trips(n)` | `ExtendedFloat::<f64>::from(n) == Extend(n as f64)` |

(Driven against `arb_f64()` for finite-input properties; deterministic
for the `from_u8` round-trip.)

### T2 — π-driven doctests on `F064F032`

**Motivation.** Every doctest in `cast.rs` today is on a
`Conn<i32, i32> = Conn::identity()`. That makes `upper`, `ceiling`,
`midpoint`, `round`, etc. each reduce to `f(x) == x` — informative
about the function signature, useless about what a connection
actually does. Switching to `F064F032` over `Extend(PI)` shows the
real bracket structure.

**Doctest pattern (simple accessors / lifters):**

```rust
use connections::conn::float::f64::F064F032;
use connections::conn::float::ExtendedFloat::Extend;
use connections::conn::cast::upper;

let pi = Extend(std::f64::consts::PI);
let pi_down = F064F032.floor(pi);
assert!(upper(&F064F032, pi_down) <= pi);
```

**Doctest pattern (arithmetic helpers — needs T3):**

```rust
use connections::conn::float::f64::F064F032;
use connections::conn::float::ExtendedFloat::Extend;
use connections::conn::cast::midpoint;

let pi = Extend(std::f64::consts::PI);
let lo = F064F032.inner(F064F032.floor(pi));
let hi = F064F032.inner(F064F032.ceil(pi));
let mid = midpoint(&F064F032, pi);
assert!(lo <= mid && mid <= hi);
```

`median`'s doctest stays on the hand-built `ORDERED_PAIR` fixture —
F064F032 is not a `Conn<(A, A), A>`.

### T4 — Module-level "Behavior on edge inputs" + trim per-fn NaN paragraphs

**Motivation.** Every chain combinator currently re-explains its
length-2 / NaN behavior in its own register. `round` even ends with
"Callers that need different NaN handling should branch upstream"
(line 397) — anxious, vague, and out of step with the rest of the
file. State the behavior once and refer back.

**Module doc (append to existing `## Two-sided helpers` section):**

```text
## Behavior on edge inputs

The two-sided helpers (`interval`, `midpoint`, `round`, `truncate`,
and their `1`/`2` lifters) bracket the input by calling **both**
`c.ceil` and `c.floor`. Two cases collapse the bracket:

1. **Length-2 (one-sided) connection** — when `c.ceil == c.floor`
   (the connection is a single adjoint pair, not a triple), the
   bracket is a point: `lo == hi`. `interval` returns
   `Some(Equal)`; `midpoint`, `round`, and `truncate` all return
   that single value. The helpers are well-defined on length-2
   conns; they're just not interesting.

2. **NaN-bearing source** — for source types that admit NaN (e.g.
   `ExtendedFloat<f64>`), `partial_cmp` between bracket endpoints
   returns `None`. `interval` propagates the `None`; `round` and
   `truncate` fall through to the `truncate` arm, which dispatches
   on `x >= 0` (false for NaN, so the `c.ceil(x)` branch fires).

Per-function docs reference this section rather than restating it.
```

**Per-function trims:**

- `round`: replace the existing `**NaN behavior.**` paragraph with
  a one-liner pointing at the module section. Delete the trailing
  "Callers that need different NaN handling should branch
  upstream" sentence.
- `interval`, `midpoint`, `truncate`: append a one-liner pointing
  at the module section after their `**Overflow:**` /
  `**Precondition:**` paragraphs (where applicable).

### End-to-end scenario

```rust
use connections::conn::cast::{interval, midpoint, round};
use connections::conn::float::ExtendedFloat::Extend;
use connections::conn::float::f64::F064F032;

let pi = Extend(std::f64::consts::PI);
let mid = midpoint(&F064F032, pi);
let _interval = interval(&F064F032, pi); // Some(Less | Greater); never Equal at π
let r = round(&F064F032, pi);            // whichever f32 endpoint is closer
assert!(F064F032.floor(pi) <= r && r <= F064F032.ceil(pi));
```
