# Plan 26 — Make `cargo fmt` pre-commit step blocking

## Goal

Flip the `cargo fmt --all -- --check` step in
`.claude/settings.json`'s pre-commit hook from **non-blocking warning**
to **blocking**, matching the other quality gates (check-pii, tests,
clippy). Update CLAUDE.md §Pre-commit hook to describe the new
behavior. Fold an existing doc drift fix in passing — the section
lists 4 steps but the hook actually runs 5 (it includes
`scripts/check_readme_mirror.sh`).

## Motivation

Plan 25 / MR !39 hit a real CI fmt failure that local tooling let
through: import ordering across the per-host-type files plus the
rustfmt indent-instability bug on the new `nz_*_ext!` macros (T3).
Both slipped past `/sprint-review` because the local pre-commit hook
only logged a warning rather than aborting. The CI fmt-check job is
the safety net that caught it; that round-trip is wasteful when the
local hook could have failed-fast.

After this lands, no Plan 25-shaped fmt diff can reach CI: any commit
that doesn't pass `cargo fmt --all -- --check` is rejected at
`git commit` time.

## Dependency Graph

```
T2 (drop stale F064B016 readme_mirror entry) ──► T1 (flip hook) ──► commit
```

T2 first — without it, the new blocking hook would refuse the very
commit that lands the new hook (`scripts/check_readme_mirror.sh`
exits 1 on a manifest entry pointing at the long-removed path
`src/conn/float/f64.rs`).

## Tasks

### T2 — Drop the stale `F064B016` `readme_mirrors.txt` entry

Pre-existing rot, surfaced by the regression test for T1.
`scripts/readme_mirrors.txt` lists:

```
F064B016|src/conn/float/f64.rs|use connections::conn::float::f64::F064B016;
```

But:
- The path `src/conn/float/f64.rs` doesn't exist (the crate root
  module is `crate::float`, not `crate::conn::float`).
- The Conn const `F064B016` doesn't exist anywhere in `src/` or
  `README.md` — the half-bridge Conns were removed in some prior
  refactor (CLAUDE.md notes "Half-bridge Conn relocation into
  `conn::half::{f16,bf16}` per the placement rules" as deferred).

The manifest entry is fully stale on both sides. Since the existing
hook had `cargo fmt`'s `|| echo` ahead of `check_readme_mirror.sh`,
the chain still proceeded for fmt-clean commits — but the script
itself was already failing silently (its stderr went to the hook
log, but its exit code was 1).

Fix: delete the line from `scripts/readme_mirrors.txt`. The file
becomes header-comments-only, which `check_readme_mirror.sh` handles
correctly (the loop body never executes, `drift=0`, exit 0).

### T1 — Flip the hook from warn to block

`.claude/settings.json`: change the first segment of the hook command
from

```
{ cargo fmt --all -- --check || echo '[warn] cargo fmt found formatting differences (non-blocking; run: cargo fmt --all)'; }
```

to plain

```
cargo fmt --all -- --check
```

The trailing `&&` chain is unchanged, so `check-pii.sh` /
`check_readme_mirror.sh` / `cargo test` / `cargo clippy` continue to
run only when fmt is clean. (They do today, too — the `|| echo` made
the step "always succeed", so the chain proceeded regardless. The
new shape just rejects unformatted commits up front, which is the
desired behavior.)

`CLAUDE.md` §Pre-commit hook: rewrite the bullet list to reflect
both changes:

- Mark step 1 (`cargo fmt`) as **blocking** rather than "non-blocking,
  logs a warning."
- Insert step 3 (`scripts/check_readme_mirror.sh`) which exists in the
  hook but was missing from the list. Renumber 3 → 4 (test) and
  4 → 5 (clippy).

## Verification

### Properties (must pass)

This plan is a workflow / config change with no Conn-shape impact.
No new property tests.

### Spot checks

- `.claude/settings.json` parses as valid JSON
  (`python3 -c 'import json; json.load(open(".claude/settings.json"))'`).
- The hook command, when extracted, runs `cargo fmt --all -- --check`
  as the first segment with no `||` fallback — confirmed by inspecting
  the JSON.
- `cargo fmt --all -- --check` is currently clean on this branch (sanity:
  the new hook would not block this branch's own commit).
- An intentional whitespace mistake in a scratch `.rs` file gets
  rejected by the hook (manual one-shot — not committed).

### Build gates

- `cargo build` clean.
- `cargo test --workspace` clean.
- `cargo clippy --all-targets -- -D warnings` clean.
- `cargo fmt --all -- --check` clean.

End-to-end: a fresh checkout where someone runs
`git commit` after introducing fmt drift gets aborted with the
standard rustfmt diff, instead of silently committing past it.

## Deferred

- A "fmt-on-save" editor hook story for downstream contributors —
  outside the repo's scope; this lands the safety net, not the
  workflow upgrade.
- Any change to `cargo fmt` rules / `rustfmt.toml` itself. The bug
  this addresses was procedural (warn vs block), not stylistic.
