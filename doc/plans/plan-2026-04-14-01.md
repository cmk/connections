# Plan 01 — Foundation types

## Goal

Ship `Conn<A, B>`, `FX<T>`, and `property.rs` — the three modules everything else depends on.

## Dependency Graph

```
T1 (Conn) ──→ T3 (property.rs)
T2 (FX)   ───┘
```

T1 and T2 are independent. T3 depends on both.

## Tasks

### T1 — `conn.rs`

**Problem:** The core connection type doesn't exist yet. Every other module depends on it.

**Solution:** Implement `Conn<A, B>` as a struct of three `fn` pointers (`ceil`, `inner`, `floor`) per the design doc. Include:

- `Conn::new(ceil, inner, floor)` — full triple constructor
- `Conn::new_left(ceil, inner)` — one-sided constructor (sets `floor = ceil`)
- `Conn::id()` — identity connection
- `Conn::then()` — sequential composition (covariant on ceil/floor, contravariant on inner)
- `product()` — lift into product order `(A, C) → (B, D)`
- `coproduct()` — lift into `Result<A, C> → Result<B, D>`
- `Conn::bounded()` — bounded type → unit
- `Conn::ordered()` — `(A, A) → A` via min/max

**Types:**

```rust
pub struct Conn<A, B> {
    ceil:  fn(A) -> B,
    inner: fn(B) -> A,
    floor: fn(A) -> B,
}
```

### T2 — `float_ext.rs`

**Problem:** IEEE 754 floats have broken reflexivity (`NaN ≤ NaN` is false) and no top/bottom elements. Connections involving floats need a type that repairs both.

**Solution:** Implement `FX<T>` — a float extension enum with synthetic bounds:

```rust
pub enum FX<T> {
    Bot,
    Finite(T),
    Top,
}
```

Derive/implement: `Debug`, `Clone`, `Copy`.

`PartialEq` impl: `Finite(a) == Finite(b)` delegates to `T::partial_eq` **except** when both are NaN, where it returns `true`.

`PartialOrd` impl:
- `Bot ≤ x` and `x ≤ Top` for all `x`
- `Finite(a).partial_cmp(Finite(b))` delegates to `T::partial_cmp` **except** when both are NaN, where it returns `Some(Equal)`

NaN remains incomparable with all finite values and ±∞ — only self-comparison is patched.

### T3 — `property.rs`

**Problem:** Every connection must satisfy six algebraic laws. Without reusable test helpers, each connection module will duplicate the same assertion logic.

**Solution:** Generic proptest helper functions that take a `Conn<A, B>` and sample values, then assert each property:

| Helper | Assertion |
|--------|-----------|
| `adjoint(conn, a, b)` | `ceil(a) ≤ b ⟺ a ≤ inner(b)` |
| `closure(conn, a)` | `a ≤ inner(ceil(a))` |
| `kernel(conn, b)` | `ceil(inner(b)) ≤ b` |
| `monotone_ceil(conn, a1, a2)` | `a1 ≤ a2 ⟹ ceil(a1) ≤ ceil(a2)` |
| `monotone_inner(conn, b1, b2)` | `b1 ≤ b2 ⟹ inner(b1) ≤ inner(b2)` |
| `idempotent(conn, a)` | `inner(ceil(inner(ceil(a)))) = inner(ceil(a))` |

Each function returns a `bool` or calls `prop_assert!` directly. Strategies for `FX<f32>` and `FX<f64>` that bias toward NaN, ±∞, ±0, denormals, and min/max finite values.

## Review

### Design deviations

- **`FX<T>` replaced by `Ple` trait.** The design doc called for a `FloatExt<T>`
  (renamed `FX<T>`) enum with synthetic `Bot`/`Top` elements. In practice,
  `f64_f32` operates on bare floats (matching the Haskell `Cast k Double Float`),
  so we implemented an N5 partial-order trait (`Ple`) instead. `FX<T>` is still
  the right approach for float↔integer connections where range extension is needed;
  it was not needed for this sprint.

- **`property.rs` not a separate module.** The Haskell `Data.Connection.Property`
  module exports generic property helpers. Our Rust versions are inlined in each
  module's `#[cfg(test)]` block because they are currently monomorphic (typed to
  specific `Conn<f64, f32>`, `Conn<i32, i32>`, etc.). Extract to a shared module
  when we have enough connections to justify generic helpers.

- **Generic composition (`then`, `product`, `coproduct`) deferred.** `Conn` uses
  `fn` pointers, which cannot capture state. Generic composition requires closures
  (`Box<dyn Fn>` or a generic `F: Fn` representation). Deferred to a future sprint
  that introduces `ConnBox` or similar. Concrete connections are constructed directly.

- **`conn_compose_adjoint` property dropped.** Cannot test composition without
  `then`. Replaced by `prop_identity_f64_adjoint` (identity connection on `f64`,
  exercising N5 ordering).

### Code review findings (post-implementation)

1. **NaN guard was dead code** — `ceil_f64_f32` and `floor_f64_f32` checked
   `is_nan()` after an exactness check that already handled NaN via N5
   reflexivity. Moved NaN guard to the top of both functions.
2. **`conn.rs` had no tests** — added spot checks and property tests.
3. **`order.rs` had only spot checks** — added proptest properties for
   reflexivity, transitivity, antisymmetry, and bot/top.
4. **Edge-case strategies underweighted** — changed from 49:1 to 4:1
   (~20% edge cases).
