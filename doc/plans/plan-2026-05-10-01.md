## Context

MR !91 (Phase 1: 24 Float‚ÜíExtended<intN> Conns) and MR !92 (Phase 2:
122 Float‚ÜíQ-format Conns) shipped successfully. The merged review
file `doc/reviews/review-00092.md` enumerates four follow-ups:

1. Mirror the boundary-saturation fix into `__float_ext_int_ceil_body!`

2. Bump law-battery cases from 256 ‚Üí 1024 to match plan-02's stated
   target (the integration tests currently inherit the proptest
   default of 256).
3. Add an unbounded f64 generator variant that targets the saturation
   plateaus (`u64::MAX as f64 = 2^64`, `i64::MAX as f64 = 2^63`,

Per user direction, items (1)‚Äì(3) ship as MR A (small, tightly

T6's choice of direct over composed; `f16 as f64` is exact so the
existing macros work unchanged with `$float = f16`).

## MR A ‚Äî Tasks (items 1‚Äì3)

### T1 ‚Äî Apply boundary-saturation fix to `__float_ext_int_ceil_body!`

**Why**: `src/float.rs:1429-1470` has the same `if $v > hi` strict
guard the pre-fix `__float_fix_ceil_body!` had. Affected L-only
Conns: `F032U064`, `F032I064`, `F064U064`, `F064I064` ‚Äî exactly the
ones where `<int>::MAX as f64` rounds up.

**What**: Mirror the Phase 2 fix (`src/fixed.rs:594-612`):

```rust
} else {
    let scaled_ceil_int = $v.ceil() as $int;
    if scaled_ceil_int == <$int>::MAX
        && <$int as $crate::float::BitsToF64Rd>::to_f64_rd(<$int>::MAX) < ($v as f64).ceil()
    {
        $crate::extended::Extended::PosInf
    } else {
        $crate::extended::Extended::Finite(scaled_ceil_int)
    }
}
```

The `BitsToF64Rd` impls already cover all relevant int widths
(`src/float.rs`, sealed-trait dispatch added during Phase 2). No new
infrastructure needed.

**Out of scope**: i128/u128 targets via `Extended<intN>` aren't part
of Phase 1 and aren't shipped ‚Äî no fix needed there.

### T3 ‚Äî Plateau-biased f64 / f32 generators

**Why**: The merged review's Item 3. Bounded generators miss the
saturation plateau, so the law battery never exercised the fix.

**What**: Add a new boundary-set to `arb_f64_bounded` (and the
mirror in `tests/common/mod.rs`) covering:

```rust
Just(2.0_f64.powi(63)),    // i64::MAX as f64
Just(-2.0_f64.powi(63)),   // i64::MIN as f64
Just(2.0_f64.powi(64)),    // u64::MAX as f64
Just(2.0_f64.powi(127)),   // i128::MAX as f64
Just(-2.0_f64.powi(127)),  // i128::MIN as f64
Just(2.0_f64.powi(128)),   // u128::MAX as f64
// plus `next_up()` and `next_down()` of each, to land *just past*
// the plateau (which is what would have shipped the original bug).
```

For f32, add the same shape but with `f32::next_up`/`next_down`
applied to the plateau values that fit in f32 (only u32-side
plateaus fit; the rest saturate to ¬±‚àû). Saturation to ¬±‚àû is also
worth covering via `Just(f32::INFINITY)` (already present).

**Where**: `src/prop/arb.rs` (extend `arb_f64_bounded` /
`arb_f32_bounded`). Mirror in `tests/common/mod.rs`. Keep the
existing |v|‚â§1e9 / |v|‚â§10 uniform branch ‚Äî the plateau values are
additional `Just(_)` arms in the boundary `prop_oneof!`.

**Naming**: extend the existing strategy in place rather than adding
`*_unbounded` variants. The bounded random branch stays bounded; the
boundary set just gets richer.
