# Plan 04 — Manual re-review trigger + claude-review hardening

## Goal

Make `claude-review` re-runnable on demand (no new commit required) and
more resilient to transient Anthropic 5xx blips.

## Context

Sprint A shipped `claude-review` as a CI job that fires on
`merge_request_event` pipelines only. Live run on MR !2 surfaced two
pain points worth addressing now:

1. **No way to re-run without a commit.** The natural use case is "that
   review was stale / truncated / overloaded out — run it again."
   Pushing an empty commit works but pollutes history and also fires
   the full audit stage unnecessarily.

2. **Anthropic 529 Overloaded blips terminate the job** when SDK
   default `max_retries=2` doesn't clear the load window. The job is
   `allow_failure: true` so nothing blocks, but the review doesn't post
   either.

Both are small, contained script/config fixes.

## Tasks

### T1 — Manual trigger path in `claude_review.py`

Add a fallback input path. When the job runs in `merge_request_event`
mode, GitLab sets `CI_MERGE_REQUEST_IID`, `CI_MERGE_REQUEST_DIFF_BASE_SHA`,
and `CI_COMMIT_SHA`. When the job is triggered manually (via `glab ci run`
or "Run pipeline" in the web UI), those MR-scoped vars are absent.

The script learns to:

1. Read the MR iid from `CI_MERGE_REQUEST_IID` first; fall back to
   `CLAUDE_REVIEW_MR` (the variable passed to manual triggers).
2. If `CI_MERGE_REQUEST_DIFF_BASE_SHA` is missing, fetch it and the
   head SHA from
   `GET /projects/:id/merge_requests/:iid` (fields `diff_refs.base_sha`
   and `sha`).
3. Ensure both SHAs are present locally via
   `git fetch origin $base_sha $head_sha` before running `git diff`.

Everything else (prompt shape, finding parser, inline-anchor POST) is
unchanged. The current `git diff base...head` call just receives the
right SHAs regardless of trigger source.

### T2 — CI rules + retry hardening

Two changes in `.gitlab-ci.yml` for the `claude-review` job:

1. **Second rule** so the job fires on manual triggers:

   ```yaml
   rules:
     - if: $CI_PIPELINE_SOURCE == "merge_request_event" && $ANTHROPIC_API_KEY && $GITLAB_TOKEN
     - if: $CLAUDE_REVIEW_MR && $ANTHROPIC_API_KEY && $GITLAB_TOKEN
   ```

   Either branch is sufficient; the script handles both envelopes.

2. **Job-level `retry: 2`** with `when: runner_system_failure` restricted
   to transient classes. Actually GitLab's `retry:when:` supports
   `script_failure`, `api_failure`, `runner_system_failure`,
   `job_execution_timeout`, etc. We want to rerun on `script_failure`
   for the 529-blip case — the whole job re-fires minutes later when
   Anthropic's load window has shifted.

And in `scripts/claude_review.py`:

3. **Bump `anthropic.Anthropic(max_retries=5)`** so the SDK itself
   rides a longer backoff inside one invocation before surfacing the
   error to CI.

### T3 — Document manual re-review in CLAUDE.md

Short block under the existing "Automated MR reviewer" section:

- Invocation via `glab ci run`:
  ```
  glab ci run --variables "CLAUDE_REVIEW_MR=42" \
    --branch "$(glab mr view 42 -F json | jq -r .source_branch)"
  ```
- Invocation via web UI: Project → Build → Pipelines → Run pipeline →
  branch = MR's source branch → variable `CLAUDE_REVIEW_MR` = iid.
