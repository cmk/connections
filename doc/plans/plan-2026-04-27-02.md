# Plan 14 — `conn::time` v2 (Phase A): DURNFD09 + OFDT*

## Context

Three classes of follow-up were deferred from `conn::time` v1 (MR
!17 / !19):

1. **Float-seconds bridges** — IEEE floats ↔ `Duration`.
2. **`OffsetDateTime` / `UtcOffset`** — timezone-aware time.
3. **`Weekday` / `Month` enum tags** — calendar tag bytes.

Two adjacent sprints have unlocked the design:

- **MR !18** dropped `Ple` for `Eq + PartialOrd` and renamed the
  decimal rungs to `FD00..FD12`.
- **MR !21** (sprint/float-half — merged just before this sprint)
  added `half::f16` / `half::bf16` support, type aliases `F016` /
  `F032` / `F064` / `B016`, the cross-precision float conns
  (`F032F016`, `F032B016`, `F064F016`, `F064B016`), and the
  `def_walk_helpers!` macro.

This MR is **Phase A** of the v2 follow-up. Phase B (the four
float-Duration bridges using `def_walk_helpers!`) will land as a
follow-up MR.

## Deferred (out of this MR)

- **F064DURN / F032DURN / F016DURN / B016DURN** — the four
  float-Duration bridges. `def_walk_helpers!` is now available, but
  the bridges need a `shift_duration` helper, four
  `widen_f???_to_duration` helpers, and careful NaN policy. Phase B
  sprint picks them up.
- **WDAYU008 / MNTHU008** — `time::Weekday` and `time::Month` don't
  derive `PartialOrd` in v0.3.45, and the orphan rule blocks a
  local impl. Future sprint should introduce newtype wrappers
  (`WeekdayIso(Weekday)`, `MonthInt(Month)`) with `PartialOrd` by
  `number_from_monday()` / `u8::from`. Their arb strategies ship
  now to be ready.
- **UTCOSECS** — `time::UtcOffset`'s `Ord` orders by a packed
  `as_u32()` byte representation (a hash-key implementation
  detail), not by `whole_seconds`. The Galois machinery requires
  the source-side `PartialOrd` to align with the rung's natural
  order. Future sprint should introduce `UtcOffsetSecs(UtcOffset)`
  newtype with `PartialOrd` by `whole_seconds`.

## Implementation notes

### `DURNFD09`

`Duration`'s native sub-second resolution is the nanosecond, so
`Finite ⇄ Finite` is exact. `Duration`'s range (≈ ±2.9 × 10²⁰ s)
exceeds `FD09`'s (`±i64` nanos ≈ ±292 years), so out-of-FD09
Durations saturate the rung to `Extended::NegInf` / `PosInf`.
Saturation arms at `Duration::MIN`/`MAX` follow DURNSECS's
pattern: `ceil`/`floor` are Galois-forced to `±Inf`.

### `OFDTNANO` / `OFDTSECS`

Both saturate at the **UTC range** (not the wider OffsetDateTime
instant range). `OffsetDateTime::from_unix_timestamp_nanos` only
constructs UTC OFDTs in `[Date::MIN UTC, Date::MAX UTC]`; instants
past that range (reachable via Date::MAX with negative offset) fall
outside the round-trippable subset and the rung saturates.

For `Finite(odt with instant > UTC_MAX)`:
- `ceil` → `UTC_MAX + 1` (smallest rung whose `inner` returns `PosInf`).
- `floor` → `UTC_MAX` (largest rung whose `inner` is still Finite, hence ≤ that instant).

Symmetric for instants below `UTC_MIN`.

OffsetDateTime's `Ord` impl in `time` 0.3 is **instant-based** (re-bases the right operand to the left's offset before comparing), so
two same-instant-different-offset OFDTs collapse into one
equivalence class under the law machinery.

### `arb_duration` widening

The previous uniform slot was
`(-10⁹..=10⁹, -999_999_999..=999_999_999)`. Widened to
`(any::<i64>(), -999_999_999..=999_999_999)` — `Duration::new`
normalises sign mismatches without panicking. Boundary slots
extended to include `Duration::seconds(±i64::MAX)`,
`±Duration::nanoseconds(1)`, `Duration::MAX − 1ns`,
`Duration::MIN + 1ns`.

## Verification

Each new conn drives the standard battery from `property::laws`:

- `conn_galois_l/r`, `conn_closure_l/r`, `conn_kernel_l/r`,
  `conn_monotone_l/r`, `conn_idempotent`,
  `conn_roundtrip_ceil/floor` (over bounded rung strategies).

`conn_floor_le_ceil` is intentionally not driven (same precedent as
DATEJDAY / U008I016: saturation arms force `floor > ceil`).
`conn_ulp_bound` likewise omitted for OFDTSECS at Extended-source
extremes (`ceil = i64::MIN` vs `floor = utc_min_s − 1` is far more
than 1 ULP apart).

`*_preorder` proptest mods drive
`lattice_{reflexive,transitive,antisymmetric,bot,top}` against the
natural extremes — for OFDT this is `ofdt_{min,max}_instant()` (the
**instant** extremes, since OffsetDateTime's PartialOrd is
instant-based), not the narrower UTC range.

End-to-end: 1757 lib tests + 27 doctests pass. `cargo clippy
--all-targets --no-deps -- -D warnings` clean. `cargo doc --no-deps`
no new warnings.
