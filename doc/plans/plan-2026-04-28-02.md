# Plan 24 — `addr/` submodule + `LATTBOOL` canonical helper

## Goal

Land Galois-connection coverage for the `std::net` address types
(`Ipv4Addr`, `Ipv6Addr`, `IpAddr`, `SocketAddr`, `SocketAddrV4`,
`SocketAddrV6`) under a new top-level `addr/` submodule, plus the
canonical `bndbin`-style helper `LATTBOOL` in `lattice.rs` for any
bounded lattice.

The original plan had wider scope (`char`, `Ordering`, `bool`); see
**Deferred** for what was cut and why.

## Dependency Graph

```
T0 (lattice::LATTBOOL helper + Ordering Join/Meet impls)
T1 (addr/ip.rs — int↔IP isos + v4-mapped bridge)
T2 (addr/ip.rs — IpAddr sum projections)
T3 (addr/socket.rs — SocketAddr sum projections)
T4 (prop/arb.rs — strategies for ip/sock/ordering)
T5 (lib.rs — pub mod addr)
T6 (CLAUDE.md prefix table — IPV4/IPV6/IPAD/SOV4/SOV6/SOVX/LATT/BOOL)
T7 (README mention)
```

## Tasks

### T0 — `lattice::LATTBOOL` + `Ordering` Join/Meet

In `src/lattice.rs`:

- `pub fn LATTBOOL<A>() -> Conn<A, bool>` where `A: Join + Meet +
  Copy + 'static`. Generic full-triple Conn (`Conn::new(ceil, inner,
  floor)`) implementing the Haskell `bndbin` pattern: `ceil(a) =
  (a != A::bot())`, `floor(a) = (a == A::top())`, `inner(false) =
  bot`, `inner(true) = top`. Both Galois laws hold.
- `impl Join for Ordering` (`bot = Less`) and `impl Meet for Ordering`
  (`top = Greater`) — the smallest non-trivial concrete `Join + Meet`
  type, doubling as the in-crate test surface for `LATTBOOL`.
- Naming exception: `LATTBOOL` uses the 8-char Conn-name shape on a
  generic function. `#[allow(non_snake_case)]` keeps the const-style
  identifier despite the snake_case lint.

### T1 — `addr/ip.rs`: int ↔ IP isos + v4-mapped bridge

Three Conns, all full triples:

- `U032IPV4: Conn<u32, Ipv4Addr>` — total bijection via `Conn::new_iso(forward, back)` over `Ipv4Addr::{from_bits, to_bits}`.
- `U128IPV6: Conn<u128, Ipv6Addr>` — total bijection, same shape.
- `IPV6IPV4: Conn<Ipv6Addr, Extended<Ipv4Addr>>` — v4-mapped projection. `Conn::new(ceil, inner, floor)` with asymmetric ceil/floor at the boundaries of the v4-mapped block (Ipv6 below `::ffff:0:0` → ceil to `Finite(0.0.0.0)` / floor to `NegInf`; above `::ffff:ffff:ffff` → ceil to `PosInf` / floor to `Finite(255.255.255.255)`). `inner` is order-reflecting because the v4-mapped block is a strict subset of Ipv6Addr — the "subsec room" pattern, analogous to how `DURNSECS` survives at `Duration::MIN`.

### T2 — `addr/ip.rs`: `IpAddr` sum projections

Two **one-sided** Conns:

- `IPVXIPV4: Conn<IpAddr, Extended<Ipv4Addr>>` via `Conn::new_left(ceil, inner)`. V4 inputs project to `Finite(addr)` (with `ceil(V4(0.0.0.0)) = NegInf` per Galois L); V6 inputs saturate up to `PosInf`. `inner(NegInf) = V4(0.0.0.0)`, `inner(PosInf) = V6(MAX bits)`.
- `IPVXIPV6: Conn<IpAddr, Extended<Ipv6Addr>>` via `Conn::new_right(inner, floor)`. V6 inputs project to `Finite(addr)` (with `floor(V6(MAX)) = PosInf` per Galois R); V4 inputs saturate down to `NegInf`.

These are one-sided because the source `IpAddr`'s extremes coincide with `inner`'s synthetic-end values, making `inner` non-order-reflecting and breaking `floor_le_ceil` if shipped as a full triple. With `new_left`/`new_right`, `floor = ceil` structurally so the law holds by construction. Each holds one Galois law (per the constructor's contract) and `floor_le_ceil`. See **Review** for the math.

### T3 — `addr/socket.rs`: `SocketAddr` sum projections

Same shape as T2:

- `SOVXSOV4: Conn<SocketAddr, Extended<SocketAddrV4>>` via `Conn::new_left`.
- `SOVXSOV6: Conn<SocketAddr, Extended<SocketAddrV6>>` via `Conn::new_right`.

## Deferred

The original plan had three additional families that were dropped or
deferred during the sprint:

- **`CHARU032: Conn<Extended<char>, u32>`** — the `char ↔ u32`
  embedding with surrogate-gap and above-`0x10FFFF` saturation. The
  surrogate gap forces *different* `inner` functions for the
  ceil-adjoint and floor-adjoint pairings (round-down vs round-up at
  `0xD800..=0xDFFF`), which means it can't be a single full triple.
  Splitting into two one-sided Conns (`CCHRU032` ceil + `FCHRU032`
  floor) was attempted but ended up requiring `Extended` on both
  sides, violating the "don't extend both sides" anti-pattern. Drop
  for this sprint; revisit as a one-sided `Conn<char, u32>` if
  someone wants the embedding direction.

- **`ORDRI008: Conn<Ordering, i8>`** — the natural-bijection
  embedding. Was specced as Plan B (one-sided `new_right` with
  `floor(Less) = -1, floor(Equal) = 0, floor(Greater) = 1`,
  saturating `inner` for out-of-range i8). The orientation flip
  discussion ("more specific on RHS") complicated this and the
  resulting Conn never reached a green test state by the time we
  decided to drop it. The canonical "i8 → Ordering" projection in
  bndtri form is structurally the same as `LATTBOOL::<i8>()` would
  be IF `i8` impl'd `Join + Meet` — but adding `Join + Meet` impls
  for all the integer types is its own scope. Drop for this sprint.

- **`BOOLU008: Conn<bool, u8>`** — same situation as ORDRI008. Was
  specced as Plan B (one-sided `new_right`). Got deleted as
  collateral damage during a rollback and never restored. Same
  observation: equivalent to `LATTBOOL::<u8>()` if `u8` had `Join +
  Meet`.

- **CLAUDE.md prefix table update + README mention** — the new 4L+0D
  prefixes (`IPV4`, `IPV6`, `IPAD`, `SOV4`, `SOV6`, `SOVX`, `LATT`,
  `BOOL`) deserve a row in the §Conn-name format table. Hold for a
  docs sweep alongside the Plan 27 audit.

- **`bndtri` 3-chain analogue of `bndbin`** — `LATTBOOL` covers
  bounded → bool. A symmetric `LATTORDR: Conn<A, Ordering>` for
  bounded → 3-chain would be the natural follow-up. Out of scope.

- **Orphan `prop::arb` strategies (`arb_extended_u32`, `arb_char`,
  `arb_extended_char`)** — `arb_extended_u32` was added during the
  CHAR Conn implementation and `arb_char` / `arb_extended_char` were
  added for the `CCHRU032` / `FCHRU032` proptest battery. All three
  are public with no current in-crate consumer; kept rather than
  removed so a future revisit of the CHAR family doesn't have to
  re-derive them from scratch. Note: `pub fn` strategies are not
  flagged as dead code by clippy, so they don't trip the lint.

## Review

### Major design deviations from the original plan

1. **Module layout: `addr/` instead of four top-level submodules.** The original plan put `char`, `net`, `ordering`, `boolean` each as their own top-level submodule. CLAUDE.md §Module organization explicitly limits top-level submodules to `fixed/`, `time/`, `float/`, `conn/`. Per user direction, the IP family folded into a new `addr/` submodule instead (matching `time/`'s shape). The other planned families were dropped (see Deferred).

2. **Orientation flip: numeric on LHS, domain type on RHS.** Plan-as-written had `IPV4U032 = Conn<Ipv4Addr, u32>` etc. — domain type on LHS. User directive: more-specific (use-wise) types on the RHS, since composition naturally reads left-to-right from generic numerics into specific domains. Conn names flipped accordingly: `U032IPV4`, `U128IPV6`, `IPV6IPV4`, `IPVXIPV4`, `IPVXIPV6`, `SOVXSOV4`, `SOVXSOV6`.

3. **Sum projections shipped as one-sided Conns, not full triples.** This was the biggest course correction of the sprint. The four Conns `IPVXIPV4`, `IPVXIPV6`, `SOVXSOV4`, `SOVXSOV6` were initially specced as full triples with "DURNSECS-style asymmetric arms" at the source MIN/MAX. **That framing was wrong.** DURNSECS doesn't have asymmetric arms — its `inner` is order-reflecting at the bottom thanks to sub-second precision (`inner(NegInf) = Duration::MIN` with subsec −999_999_999, distinct from `inner(Finite(i64::MIN)) = Duration::seconds(i64::MIN)` with subsec 0). My initial Conns *did* violate `floor_le_ceil` at the source MIN/MAX because `inner(NegInf rung) = source MIN = inner(Finite(target MIN))` — non-injective. Fix: convert to `new_left` for the V4-target Conns (Galois L only, V6 saturates up) and `new_right` for the V6-target Conns (Galois R only, V4 saturates down). `floor = ceil` structurally. Lawful.

4. **Mid-sprint addition: `LATTBOOL` helper.** Originally a follow-up idea ("canonical bndbin-style Conn for any bounded lattice"). User asked to land it in this sprint as a generic helper in `lattice.rs`. Pulled `Join + Meet` impls for `Ordering` along with it as the smallest non-trivial test surface.

5. **CHAR family dropped entirely.** Original plan had `CHARU032`. Two reasons it didn't ship: (a) the surrogate-gap forces a two-Conn split (no single `inner` is lawful on both adjoint pairings), and (b) the natural shape of either half required extending both sides, violating the "don't extend both sides" rule. The two-Conn split (`CCHRU032`/`FCHRU032`) was implemented, then dropped at user direction.

6. **`ORDRI008`/`BOOLU008` dropped after partial implementation.** Both were specced as Plan B (one-sided `new_right` with natural bijections). The orientation flip discussion left them in an inconsistent state and there wasn't time to re-derive cleanly. They're equivalent to `LATTBOOL::<i8>()` / `LATTBOOL::<u8>()` if those types had `Join + Meet` impls — folded into the deferred list with that observation as the migration path.

### Lawfulness lessons learned

- **`floor_le_ceil` is not free for full triples.** A `Conn<A, B>` constructed via `Conn::new(ceil, inner, floor)` can satisfy both Galois L and Galois R yet still violate `floor(a) ≤ ceil(a)` if `inner` is non-order-reflecting at a source extreme — i.e., when the rung's synthetic `NegInf` collapses to the same source value as some `Finite(_)` rung value. The standard battery (`galois_l/r`, `closure_l/r`, `kernel_l/r`, `monotone_l/r`, `idempotent`) was missing `conn_floor_le_ceil`, and the violation hid until I tried to defend it with a fake citation. Plan 27 (already stubbed on main) audits the rest of the crate for the same pattern; suspects: `TIMENANO`, `TIMESECS`, `DATEJDAY`, `OFDTNANO`, `OFDTSECS`.

- **One-sided Conns are the safe escape hatch.** When a full triple isn't lawful (because of inner non-order-reflection), `Conn::new_left` / `Conn::new_right` set `floor = ceil` as a fn pointer, making `floor_le_ceil` trivially hold. The cost is that only one Galois law is asserted (the contract documented on the constructor). For Conns where the two adjoints would naturally agree on the bijective range and only differ at synthetic saturation, this is the right shape.

- **Don't fake citations.** During the sprint I attributed a math claim to nLab's adjoint-triple article without checking. The article doesn't say what I claimed. Got called out, retracted, fixed. Lesson: cite only what's been read; first-principles arguments don't need a citation but should be stated as such.

### Test surface delta

Going in, the test battery was 1230 lib tests (Plan 23 baseline). Going
out, it's 1291 — net +61 tests across the new `addr/` Conns and the
`LATTBOOL` Ordering exercise. Every full-triple Conn shipped includes
`floor_le_ceil` in its battery; every one-sided Conn includes the
appropriate side's Galois law plus `floor_le_ceil` (which holds
trivially).

### What I'd do differently

- **State the lawfulness analysis up-front.** I started writing Conns and discovered the lawfulness issues only after tests failed. A pre-flight derivation per Conn (sketch `inner`, check order-reflection at the extremes, decide full-triple vs one-sided) would have saved several rounds of rework.

- **Don't reach for a shape because it "looks like DURNSECS".** DURNSECS is a specific working pattern that depends on subsec precision making `inner` order-reflecting. Pattern-matching on shape without verifying the analogy is how the "DURNSECS-style asymmetric arms" framing slipped in.
