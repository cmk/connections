# Plan 06 — Proptest audit and shared strategy harness

## Goal

Close the mathematically-required property gaps in the `fixed` ladder
and `sample` tiers (closure + idempotent), centralize the duplicated
`arb_f32` / `arb_f64` generators into `src/property.rs`, and lock in
the shrink-hardening patterns already discovered on the
`float_conn_props!` macro so the same patterns apply everywhere they
matter.

## Context

Sprint 3 of the three-sprint tidy. Final cleanup of the property-test
layer. Sprint B's audit identified three issues:

1. **Missing properties**: the 21 `props_for_pair!` ladder conns in
   `conn/fixed.rs` test `adjoint` / `monotone_ceil` / `monotone_floor` /
   `floor_le_ceil` / `inner_then_ceil_is_id` / `inner_then_floor_is_id`
   but not `closure` or `idempotent`. The 19 `props_for_conn!` +
   `props_for_pico_conn!` sample conns have the same pattern.
2. **Duplicated strategies**: `arb_f32` / `arb_f64` are defined
   identically in 4 places — `src/conn.rs`, `src/conn/float.rs`,
   `src/lattice.rs`, `src/conn/fixed.rs` (as `arb_f64_full`).
3. **Shrink pathology**: `any::<f64>()` × `any::<i64>()` was ruinous
   on the `float_conn_props!` macro in `conn/fixed.rs`. The fix
   already landed upstream (`cases: 64, max_shrink_iters: 512` +
   bounded generators for `arb_extended_micro` / `arb_extended_pico`)
   and passed through main via MR !3. This sprint bakes the same
   discipline into `property.rs`'s shared strategies so new conns
   inherit it automatically.

Scope note: `doc/design.md` lists 4 float↔extended pairs, but today
only `F64F06` and `F64F12` are exported (F32 variants intentionally
omitted due to the narrowing-plateau issue documented in `lib.rs`).
Sprint C matches today's reality, not the stale audit.

See memory `proptest_shrink_tuning.md` for the load-bearing rationale.

## Tasks

### T1 — Populate `src/property.rs` with shared generators and property helpers

Current state: empty stub.

Populate:

```rust
//! Shared proptest strategies and property-law helpers.
//!
//! Strategies centralize the bounded domains and boundary-weighted
//! `prop_oneof!` patterns that `conn/*` modules used to reinvent
//! separately. Property helpers encode the six Galois laws from
//! doc/design.md §"Key properties" once so each tier's per-conn
//! macro can call them by name.

use proptest::prelude::*;

/// f64 strategy with 4:1 uniform-to-boundary bias.
///
/// Boundary slot includes NaN, ±∞, ±0, MIN_POSITIVE, MAX, MIN. This
/// is the default generator for float connections that do not
/// saturate into a bounded Rung — for those, use [`arb_f64_bounded`]
/// instead, which avoids the mantissa-bit shrink pathology.
pub fn arb_f64() -> impl Strategy<Value = f64> { … }
pub fn arb_f32() -> impl Strategy<Value = f32> { … }

/// f64 strategy restricted to |x| ≤ 1e9 plus explicit boundaries.
///
/// Use for connections that feed into a bounded integer Rung
/// (Micro / Pico). Unbounded `any::<f64>()` shrinks bit-by-bit
/// through the mantissa on saturation failures, exploring the
/// uninteresting i64::MAX×PREC corner of the input space for
/// minutes. Bounding the range kills that path without dropping
/// boundary coverage.
pub fn arb_f64_bounded() -> impl Strategy<Value = f64> { … }
pub fn arb_f32_bounded() -> impl Strategy<Value = f32> { … }
```

No property helpers in this sprint — the per-tier macros already
express the laws cleanly. Adding a `prop_adjoint!` / `prop_closure!`
layer is design scope creep for a stretch sprint.

### T2 — Rewire call sites; delete duplicated definitions

Four files currently define one or both of `arb_f32` / `arb_f64`:

- `src/conn.rs:182` — `arb_f64()` (tests only, i32/f64 identity).
- `src/conn/float.rs:271` — `arb_f64()`, `arb_f32()` (same shape).
- `src/lattice.rs:129` — `arb_f32()`, `arb_f64()` (for Ple preorder proptests).
- `src/conn/fixed.rs:516` — `arb_f64_full()` (same shape, different
  name).

Each becomes `use crate::property::{arb_f32, arb_f64}` or
`use crate::property::arb_f64_bounded as arb_f64_full` (renaming
`arb_f64_full` to match the shared name; it was an alias for the
same strategy).

The existing `arb_extended_float_f64`, `arb_extended_micro`,
`arb_extended_pico` in `conn/fixed.rs` are tier-specific (they wrap
`ExtendedFloat` / `Extended`) and stay local — they shouldn't move
to `property.rs` because they depend on `conn::fixed`'s own types.

### T3 — Ladder: add `prop_closure_l/r` and `prop_idempotent` to `props_for_pair!`

`src/conn/fixed.rs:382` `props_for_pair!` currently tests 7
properties but neither closure nor idempotent. Per `doc/design.md`
§"Key properties" these are part of the six-property contract; a
Galois connection that satisfies adjoint+monotone but fails closure
or idempotent is ill-formed.

Add to the macro body:

```rust
// Closure: a ≤ inner(ceil(a))     and    inner(floor(a)) ≤ a
#[test]
fn prop_closure_l(x in bounded_fine($prec)) {
    let a = $Fine(x);
    prop_assert!(a <= $conn.inner($conn.ceil(a)));
}
#[test]
fn prop_closure_r(x in bounded_fine($prec)) {
    let a = $Fine(x);
    prop_assert!($conn.inner($conn.floor(a)) <= a);
}

// Idempotent: inner(ceil(inner(ceil(a)))) == inner(ceil(a))
#[test]
fn prop_idempotent(x in bounded_fine($prec)) {
    let a = $Fine(x);
    let once = $conn.inner($conn.ceil(a));
    let twice = $conn.inner($conn.ceil(once));
    prop_assert_eq!(once, twice);
}
```

Macro body expands across 21 pairs. Per-pair test count grows
7 → 10. Aggregate added coverage: **63 new proptest instances**.

### T4 — Float-conn: add `prop_idempotent` to `float_conn_props!`

`src/conn/fixed.rs:665` `float_conn_props!` already has both closure
variants (`prop_closed_l`, `prop_closed_r`) from a prior round. Only
idempotent is missing.

Add to the macro body (inside the `cases: 64` proptest_config block
so it inherits the shrink-hardening):

```rust
// inner(ceil(inner(ceil(a)))) == inner(ceil(a))
#[test]
fn prop_idempotent(a in $arb_src()) {
    let once = $conn.inner($conn.ceil(a));
    let twice = $conn.inner($conn.ceil(once));
    prop_assert!(eq(&once, &twice));  // use the same le()-derived eq helper already in scope
}
```

Expands across 2 conns (F64F06, F64F12). Aggregate: **2 new
proptest instances**.

### T5 — Sample tier: add missing properties to `props_for_conn!` and `props_for_pico_conn!`

`src/conn/sample.rs` has two tier macros (line 398 and line 505). Audit
each, document which of the six laws each currently covers, add
missing ones. Minimum: **closure + idempotent** on both, matching
the design contract.

Include a `proptest_config(…, cases: 128)` override only if the full
suite's runtime grows noticeably (the rate-ratio domain is bounded
so shrink pathology is unlikely). Leave at default otherwise.

### T6 — Add `proptest.toml` at repo root

Small config file:

```toml
cases = 256
max_shrink_iters = 8192
```

Rationale: `cases = 256` is already the default; making it explicit
makes the intent visible and easy to bump. `max_shrink_iters = 8192`
is ~128× lower than the library default (~1M) but ~16× higher than
the tight 512 that the float-conn macro uses locally. Gives a
middle-ground default that still converges on most shrinks; the
float-conn macro's per-block 512 override remains in force where
it's needed.

## Review

### Deviations from the plan

- **Test-count projection off.** Plan predicted "~107 new test
  functions from the macro expansions"; actual is **128** (21 pairs
  × 3 ladder + 2 × 1 float-conn + 15 × 3 rate↔rate + 6 × 3 pico =
  63 + 2 + 45 + 18). Plan undercounted rate↔rate (said 13, actually
  15) and pico conns.
- **Two new tier-local strategies added beyond plan.** The plan said
  "no property helpers in this sprint" and assumed the existing
  `bounded_fine` / `pico_bounded` would work for closure + idempotent.
  They do not — the round-trip through `inner` can overflow on
  i64::MAX-adjacent inputs because `inner` multiplies by PREC (ladder)
  or NUM/DEN (sample), and the existing bounds only guard against
  overflow in `ceil` / `floor` alone. Added:
  - `safe_fine(prec)` in `conn/fixed.rs` — caps Fine at
    `(i64::MAX / PREC) × PREC`.
  - `safe_fine_rate(num)` in `conn/sample.rs` — caps at
    `i64::MAX − num`.
  - `safe_pico_bounded()` inside `props_for_pico_conn!` — caps at
    `i64::MAX − NUM`.
  Each is used only by closure/idempotent; adjoint/monotone keep the
  looser strategies.
- **`arb_f32_bounded` deferred.** Plan listed it alongside
  `arb_f64_bounded`; implemented only `arb_f64_bounded` since no
  current connection saturates f32 into an integer rung. Easy to
  add mirror when an F32F?? conn appears.
- **No property-helper macros.** Plan explicitly deferred them;
  stayed deferred.
