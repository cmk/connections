# Plan 51 — MR review reply workflow guardrails

Encode the MR reply workflow as a guardrail instead of relying on
operator memory. The recent failure mode was not missing primitives:
the repo already had `pull_reviews.py`, `reply_review.py`, review-doc
mirroring, and amend-before-push ordering. The bug was that the docs
and script still made it too easy to bypass those primitives with raw
`glab` calls or shell-quoted Markdown.

## Scope

1. Make `/pull-reviews` and `/reply-reviews` the required path for MR
   review comments.
2. Preserve Markdown reply bodies by default.
3. Verify posted replies before folding the mirror into the fix commit.
4. Document the corrected state transition in the workflow diagram
   prose.
5. Add a mandatory worktree preflight so task edits do not land in the
   primary shared checkout.

## Non-goals

- Do not change GitLab discussion fetching semantics.
- Do not automate pushing after `/reply-reviews`.
- Do not replace `glab`; the scripts remain GitLab-first wrappers over
  the authenticated CLI.

## T1 — Document the Required Review-Round Path

Problem: the top-level workflow described `/pull-reviews` and
`/reply-reviews`, but did not explicitly call raw `glab mr note`,
`glab api`, or shell-quoted Markdown a fallback-only path.

Implementation:

- Update `CLAUDE.md` TDD step 10 to require `/pull-reviews` and
  `/reply-reviews` for MR review comments.
- Update the command summary to state that Markdown, backticks,
  multiline text, and code references should go through stdin or
  `--body-file`, followed by mirror verification before amend.

Verification:

- `rg -n "fallback-only|body-file|verify" CLAUDE.md`.

## T2 — Tighten `/reply-reviews`

Problem: the command workflow had the right high-level order, but it
still showed inline Markdown as the first posting example and did not
make verification a named step.

Implementation:

- State the full checklist order: fix commit local-only, pull latest
  comments, compose replies, post via script, mirror, verify, amend,
  then push.
- Require `scripts/reply_review.py <MR> <note-id> -` or
  `--body-file` for Markdown-sensitive bodies.
- Add explicit verification after posting: rerun `pull_reviews.py`,
  read the appended reply, and confirm the body is uncorrupted before
  amending.

Verification:

- Manually inspect `.claude/commands/reply-reviews.md`.

## T3 — Harden `reply_review.py`

Problem: `scripts/reply_review.py` supported stdin, but inline body
arguments were still accepted for text containing backticks and other
characters that are easy to corrupt in shell quoting.

Implementation:

- Keep stdin support via `-`.
- Add `--body-file PATH` for prepared Markdown.
- Refuse inline body arguments containing Markdown/shell-sensitive
  characters unless `--allow-inline-body` is passed.

Verification:

- `scripts/reply_review.py --help`.
- Parser smoke tests for stdin, `--body-file`, and inline-body
  refusal, without making network calls.
- `python3 -m py_compile scripts/reply_review.py scripts/pull_reviews.py scripts/_glab.py`.

## T4 — Document the State Transition

Problem: `doc/workflow.md` explained the amend-before-push shape, but
not the reply corruption failure mode or the verification step.

Implementation:

- Add a note to the review-round lifecycle legend: replies are posted
  through the script, mirrored, verified, amended into the unpushed fix
  commit, and pushed once.

Verification:

- Manually inspect `doc/workflow.md`.

## T5 — Protect the Shared Checkout

Problem: the workflow asked whether other agents were working, but did
not require a local preflight that identifies the primary checkout
before edits. In a handoff task, that let branch work happen in the
shared root checkout and interfered with other agents.

Implementation:

- Update `CLAUDE.md` to require `git worktree list` and
  `git branch --show-current` before file edits.
- State that the primary checkout is shared coordination space and
  should stay clean on `main`.
- Update `doc/workflow.md` to label `main_clean` as the shared state
  and require task worktrees for edits.

Verification:

- Manually inspect `CLAUDE.md` and `doc/workflow.md`.

## Final Verification

```sh
scripts/reply_review.py --help
python3 -m py_compile scripts/reply_review.py scripts/pull_reviews.py scripts/_glab.py
cargo fmt --all -- --check
cargo test --workspace --quiet
```
